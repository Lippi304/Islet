# Phase 41 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** One new collapsed-only ambient wing (HUD-08), reusing Phase 25's shared gradient/Liquid-Glass shell material and the existing `wingsShape(leftWidth:rightWidth:)` split-wing structure — same structural family as Charging/Device/Focus/Now-Playing, no new chrome, no new material, no new window. There is no new Settings permission surface: `CalendarService`'s existing EventKit authorization (Phase 14/28) is reused verbatim, so the only new Settings element is a single default-ON toggle (D-03), with no explanation popover/deep-link (unlike Phase 38/39, which needed one for Focus/Accessibility). This phase also does not touch the expanded Calendar tab (`calendarFullView`, Phase 28) — the event title is deliberately never shown on the collapsed pill (D-06).

## Countdown Wing Contract (HUD-08)

**Structural constraint (locked, not a style choice):** reuses the existing split-wing `wingsShape(leftWidth:rightWidth:)` helper flanking the physical camera bump — same structural family as Charging/Device/Focus/Now-Playing. Icon lives in the LEFT wing, `mm:ss` text lives in the RIGHT wing, exactly matching D-06's "calendar icon left, event time right" wording from the ROADMAP.

**Layout mechanism — plain `HStack(spacing: 0)` + `Spacer()`, NOT the OSD wing's absolute-offset/derived-width approach.** Phase 39's Volume/Brightness bar needed 16 on-device rounds because a fill bar must land its edge at an *exact* pixel boundary with zero slack. This wing is icon+short-text — the same forgiving shape as Focus's icon+label/dot (Phase 38), which never had a layout issue. Build it the proven simple way: `HStack(spacing: 0) { icon.padding(.leading, 14); Spacer(); text.padding(.trailing, 20) }`, relying on `wingsShape`'s already-fixed `alignment: .leading` (Phase 39 ROUND 11) to size correctly. Do not reach for `.offset(x:)`/`.position()`/`ZStack` for this wing.

**Visibility (behavioral, binds the visual spec):** the wing exists only starting 1 hour before a real calendar event and only while the island is collapsed — same "absence-based" pattern as Charging's wing on unplug and Focus's wing when off (D-06, D-08). There is no "no upcoming event" visual state — the wing simply is not in the view tree between countdowns. There is no hover-reveal (D-08) and no expanded-state variant to design (clicking expands to Home per D-07, identical to Now-Playing's existing click behavior — not a Countdown-specific expanded layout).

**Ambient-tier exclusivity:** per `41-CONTEXT.md` D-01, this wing and Now-Playing's ambient wing (`mediaWingsRow`) never render at the same time — Countdown always wins the one ambient slot while active. There is no dual/combined layout to design in this phase (that's Phase 42).

### Left wing (icon only, no label)

| Property | Value |
|----------|-------|
| Layout | Single centered `Image(systemName: "calendar")`, no `HStack`/label — same icon-only-flank shape as the OSD wings (Phase 39), not the icon+label shape Focus/Charging/Bluetooth use |
| Icon size | `.font(.system(size: 13, weight: .semibold))`, `.symbolRenderingMode(.hierarchical)` — identical size/weight/rendering-mode to every other wing icon in this codebase |
| Icon color | Matches the urgency color below (orange → red) — NOT fixed white, unlike Charging's bolt or Focus's moon. This is a deliberate divergence: the icon is part of the same urgency signal as the time text, so it must recolor with it, not stay neutral |
| Leading padding from notch edge | 14pt — reused verbatim from Focus's shipped left-flank padding (`focusWings(for:)`) |
| Recommended `leftWidth` | 118pt — reused verbatim from Focus's/OSD's own on-device-proven icon-only-flank floor for this project's exact notch geometry. Do not start from `Self.wingsSize.width / 2` (145pt) |

### Right wing (mm:ss countdown text)

| Property | Value |
|----------|-------|
| Layout | Single `Text`, right-aligned via the `HStack`'s trailing `Spacer()` — no icon, no secondary text, matching D-06's "no event title" rule |
| Format (D-04, locked) | `mm:ss` (e.g. `"23:14"`), zero-padded, no hour component (max is 59:59 since the countdown only starts 1 hour before the event) |
| Digit stability | `.monospacedDigit()` modifier — prevents the text's width from jittering as digits tick over every second (standard SwiftUI fix for exactly this case; do not build custom fixed-width padding) |
| Text size/weight | `.system(size: 13, weight: .semibold, design: .rounded)` — matches every other wing's icon size for visual parity between the two flanks |
| Text color | Matches the urgency color (orange → red), same value driving the left-wing icon color — see Urgency coloring below |
| Trailing padding from notch edge | 20pt — reused verbatim from the OSD wings' (Phase 39) most recently tuned trailing padding |
| Recommended `rightWidth` | Default `Self.wingsSize.width / 2` (145pt) is sufficient — unlike OSD's bar, `mm:ss` text is short and self-sizing inside the `HStack`+`Spacer()`; no derived/precision-boundary math is needed here |

### Urgency coloring (D-05, locked)

Modeled explicitly on the iPhone Dynamic Island's own Live Activity countdown treatment:

| State | Color | Applies to |
|-------|-------|------------|
| Countdown active, ≥ 60s remaining | `Color.orange` (literal, fixed — never theme-tinted) | Icon AND `mm:ss` text together, always the same color at the same instant |
| Countdown active, < 60s remaining (final minute) | `Color.red` (literal, fixed — never theme-tinted) | Icon AND `mm:ss` text together |

The color switch is an instant conditional (`remainingSeconds < 60 ? .red : .orange`) evaluated on each per-second tick — no animated/gradient color interpolation between orange and red; that is unrequested complexity for a one-time threshold crossing.

**Reference match:** no new reference image was supplied for this HUD (`41-CONTEXT.md` has no "Specifics" reference beyond the iPhone Dynamic Island orange/red convention named directly in D-05) — this is a mechanical reapplication of the already-approved wing structure (Phase 36/38/39), not a new design pass.

## Live mm:ss Refresh Mechanism (resolves D-04's flagged tension)

`41-CONTEXT.md` flags a tension between Pitfall 7's minute-boundary timer-hygiene convention (the underlying scheduling/re-arm timer must NOT poll every second) and D-04's live `mm:ss` display requirement. Resolution, left to this UI-SPEC per Claude's Discretion:

- **Scheduling/re-arm timer** (when the countdown wing appears, dismisses, and re-arms for a back-to-back event per D-09) stays a separate, minute-boundary-scheduled mechanism inside the new `CalendarCountdownMonitor` — out of this document's scope, governed by `PITFALLS.md` Pitfall 7. This UI-SPEC does not change that contract.
- **The `mm:ss` display refresh** is a `TimelineView(.periodic(from: .now, by: 1))` scoped INSIDE the countdown wing view itself — mirroring `EqualizerBars`' gate-while-visible precedent (`NotchPillView.swift:2757`), but simpler: because this wing has no "present but inactive" state (D-06/D-08 — it is either fully absent or fully active, never idle-but-mounted the way `EqualizerBars` can be paused off a still-visible Now-Playing row), the `TimelineView` does not need an explicit `paused:` flag. It naturally starts ticking when the wing mounts and stops (deallocates) when the resolver removes it from the view tree — the view's own lifecycle IS the gate. This keeps the per-second work confined to exactly the ~60 minutes (at most) the countdown is actually showing, satisfying Pitfall 7's idle-wakeup concern without adding a second always-running `Timer`.

## Verification Notes for Executor

- **New wing view:** `countdownWings(for:)` (or similar name) in `NotchPillView.swift`, mirroring `focusWings(for:)`'s structure (icon left, `Spacer()` to clear the camera bridge, content right) — reuse `wingsShape(leftWidth:rightWidth:)` verbatim, do not invent a new wrapper shape.
- **Do not repeat the OSD wing's absolute-positioning saga** (Phase 39, `39-UI-SPEC.md` ROUNDs 6-16) — this wing is icon+short-text, the same forgiving shape as Focus's icon+label, not a bar that must land at an exact pixel boundary. Plain `HStack(spacing: 0)` + `Spacer()` is correct and sufficient; `wingsShape`'s `alignment: .leading` fix (already shipped) makes this work without further geometry math.
- **Icon and text recolor together, always** — do not implement them as two independently-computed color values; drive both from one shared `urgencyColor: Color` computed property (`remainingSeconds < 60 ? .red : .orange`) so they can never desync, mirroring the "single combined transition" discipline `39-UI-SPEC.md` established for the OSD wing's muted-state icon+bar pair.
- **`TimelineView(.periodic(from:by:))` scoping is a view-layer concern only** — it must NOT be the mechanism that decides when the countdown starts/stops/re-arms (that's `CalendarCountdownMonitor`'s minute-boundary responsibility per Pitfall 7/D-09). The `TimelineView` only computes what `mm:ss` string to render on each tick from the event's start `Date`, for as long as the resolver keeps this wing in the view tree.
- **No Settings permission popover/explanation surface to build** — flag this explicitly since it's a deliberate simplification versus Phase 38/39's pattern, not an oversight: `CalendarService`'s existing EventKit authorization flow already covers this feature, so D-03's toggle is a plain visibility gate with no new deep-link/explanation UI.
- **`Color.red`'s first non-destructive use** — confirm during review that this new usage reads clearly as "urgent countdown," not confusable with the shelf's existing destructive-delete red, given they're both visible only in different, non-overlapping views (collapsed pill vs. expanded Tray) so a literal on-screen collision is not possible, but keep the semantic distinction in mind if this pattern is reused elsewhere later.
- Confirm on-device during this phase's own UAT checkpoint that the 118pt/145pt `leftWidth`/`rightWidth` split (reused from Focus/default) actually fits `mm:ss` text comfortably without truncation at the widest case (`"59:59"`) — flag and adjust in this document if not, same "update the spec in the same commit as the fix" discipline `39-UI-SPEC.md` calls out.
