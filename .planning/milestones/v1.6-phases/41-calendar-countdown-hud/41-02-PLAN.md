<objective>
Build the one genuinely new piece of engineering this phase requires — a deadline-driven (never
polling) EventKit monitor — and wire it into `NotchWindowController` exactly like every sibling
monitor (Focus, Power, Bluetooth) is wired: idempotent `start()`/`stop()`, toggle-gated launch,
toggle-off teardown, `deinit` teardown, and a change-handler that mutates a plain stored property
then calls the shared `renderPresentation()` + `updateVisibility()` tail.

Purpose: this is HUD-08's actual scheduling logic — the countdown must show up starting 1hr
before a real event, tick live, dismiss at the event's start, and immediately re-arm for a
back-to-back event, all without becoming an idle-wakeup regression (PITFALLS.md Pitfall 7).
Output: a live, working `CalendarCountdownMonitor` feeding `NotchWindowController`'s resolver call.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| EventKit -> CalendarCountdownMonitor (via CalendarService) | Calendar event data crosses into this monitor's `recheck()`/`scheduleNext(from:)` — only `EventInput.start: Date` is consumed (via `nextUpcomingEvent`), never `.title`, matching Plan 01's `CalendarCountdownActivity` shape |
| NSNotificationCenter (`.EKEventStoreChanged`) -> CalendarCountdownMonitor | System-wide notification, any app's calendar/reminders edit can trigger a re-check; no attacker-controlled payload (the notification carries no user data, just a "something changed" signal) |
| UserDefaults -> NotchWindowController | `calendarCountdownKey` toggle read via the existing `activityEnabled(_:)` helper, unchanged from every sibling toggle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-41-04 | Denial of Service (minor, local-only) | `CalendarCountdownMonitor.recheck()` triggered by `.EKEventStoreChanged` | accept | Pitfall 4: each re-check is one cheap EventKit query + one pure function call + at most one timer reschedule; not the same class of problem as Pitfall 1's perpetual-timer risk. Debounce is explicitly deferred (Task 1 action) unless Plan 04's on-device Activity Monitor check shows measurable bursts — single-user local app, no remote attacker, ASVS L1 does not require preemptive hardening here |
| T-41-05 | Denial of Service | Naive perpetual/tight-poll timer instead of one-shot deadlines | mitigate | Task 1's `scheduleNext(from:)` arms exactly one `DispatchSourceTimer` with NO `repeating:` argument, cancelled and rescheduled on every re-check — Task 1's acceptance criteria greps for zero `repeating:` occurrences in the new file, closing this class of regression before it can ship |
| T-41-06 | Elevation of Privilege | `CalendarCountdownMonitor`/`handleCalendarCountdownChange` bypassing the resolver's single-arbiter discipline | mitigate | `handleCalendarCountdownChange`'s body is verified (Task 2 acceptance criteria) to contain zero `transientQueue`/`scheduleActivityDismiss` references — the monitor can only ever influence presentation through the plain `calendarCountdownActivity` property that `currentPresentation()` feeds into `resolve(...)`, never by mutating `IslandPresentation`/`transientQueue` directly |
| T-41-SC | Tampering | npm/pip/cargo installs | not applicable | Zero external dependencies — only `Foundation`/`EventKit` (already-linked Apple frameworks) are imported by the new file |
</threat_model>
