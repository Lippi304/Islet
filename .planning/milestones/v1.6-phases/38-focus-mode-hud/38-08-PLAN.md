| Boundary | Description |
|----------|-------------|
| SettingsView (UI, main thread assumed) → NotchWindowController (@MainActor, owns live monitors) | The `requestAuthorization` completion crosses from an OS framework callback (thread not guaranteed) into app state; this plan explicitly dispatches to main before mutating `@State`/calling into the controller. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-09 | Tampering | `UserDefaults` `activity.focus` key read via `activityEnabled(_:)` | accept | App-owned UX preference, not a security boundary (same as every other activity toggle) — a locally-tampered value can at most enable/disable a cosmetic HUD, no privileged action follows from it |
| T-38-10 | Information Disclosure | `INFocusStatusCenter.default.focusStatus.isFocused` polled via `FocusModeMonitor` | accept | Unchanged by this plan — boolean-only Focus state, no named-mode data ever read or exposed (D-01/CONTEXT.md scope) |
| T-38-11 | Denial of Service (thread-safety) | `requestAuthorization` completion calling into `@MainActor` `NotchWindowController` from a non-guaranteed thread | mitigate | Completion wrapped in `DispatchQueue.main.async` before touching `showFocusPermissionExplanation` or calling `focusPermissionGranted()` |
</threat_model>
