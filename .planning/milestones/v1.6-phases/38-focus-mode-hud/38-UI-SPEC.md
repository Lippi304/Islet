# Phase 38 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** Two new surfaces, both extending existing, already-approved visual language rather than inventing anything new:
1. **The Focus collapsed-pill wing** (HUD-05) — reuses Phase 36's Droppy-pill wing language verbatim (icon+label left, status indicator right, `wingsShape()` helper), per `38-CONTEXT.md` D-10/D-11. Renders only while collapsed (D-07) and only while Focus is actually on (D-09 — there is no "Focus Off" visible state).
2. **The Settings permission-request surface** (D-01–D-05) — a new toggle row + persistent status hint + (if the spike lands on the Full Disk Access fallback path, which `38-RESEARCH.md` rates as the likely outcome) a one-time explanation view with a system deep-link button. This is new UI territory for the project — no prior phase has needed a manual, unprompted OS-permission explanation screen (Calendar/Weather/Bluetooth all use real TCC prompts) — so this spec defines it in full even though it's small.

Both render inside Phase 35's Liquid Glass material and Phase 27's sidebar Settings — no new chrome, no new material, no new window.

## Focus Wing Contract (HUD-05)

**Structural constraint (reused from Phase 36, D-01):** the wing renders via the existing `wingsShape(leftWidth:rightWidth:)` helper, flanking the physical camera bump — same split-wing structure Charging/Device/Bluetooth already use. Do not build a new continuous-capsule shape for this HUD.

**Visibility constraint (D-06, D-07, D-09 — behavioral, not visual, but binds the visual spec):**
- The wing exists ONLY while Focus is on AND the island is collapsed. There is no "Focus Off" render at all — the wing simply isn't in the view tree once Focus turns off or the island expands (same precedent as Charging's wing disappearing on unplug — not a dimmed/negative state like Bluetooth's disconnected icon).
- The wing has no `activityDuration`/auto-dismiss animation to design for — it does not fade out on a timer; it appears/disappears exactly when the resolver's presentation changes (standard `matchedGeometryEffect` morph, no bespoke transition).
- Because the wing only ever renders in the "on" state, the right-side status indicator (below) has no "off" variant to design — see note under Right wing.

### Left wing (icon + label)

| Property | Value |
|----------|-------|
| Layout | `HStack(spacing: 4)` — icon then label, matching Phase 36's Bluetooth/Charging left-wing pattern exactly |
| Icon | `Image(systemName: "moon.fill")`, `.font(.system(size: 13, weight: .semibold))`, `.symbolRenderingMode(.hierarchical)` |
| Label text | `"Focus"` — 12px semibold, `.system(size: 12, weight: .semibold, design: .rounded)`, line-height 1.2 |
| Label visibility | Always shown alongside the icon (unlike Bluetooth/Charging's "label only in positive state" rule) — because the wing itself only ever exists in the positive state (D-09), there is no negative/icon-only variant to gate |
| Left padding from notch edge | 12px — reused verbatim from the existing `wings(for:)`/`deviceWings(for:)` left padding (legacy hardware-anchored constant, same spacing exception Phase 36 already documented) |
| Icon/label color (D-11, locked) | `Color.white`, FIXED — never `deviceAccent` or any theme-selectable accent. This differs from Bluetooth's accent-tinted glyph precedent; Focus follows Charging's fixed-color precedent instead, per D-11's explicit rationale (a universal system-level state should read consistently regardless of the user's chosen accent theme) |

### Right wing (status indicator)

| Property | Value |
|----------|-------|
| Indicator | `Circle().fill(Color.green)`, 8×8pt, no stroke — a simple filled dot, not a `BatteryIndicator` (that component is Charging/Device-specific and carries a percentage this HUD has no equivalent for) |
| Color (locked) | Fixed `Color.green` — same "universal active signal, never theme-tinted" rule as Phase 36's Bluetooth-connected ring and Charging's battery fill |
| Variants | **None.** Per the Visibility constraint above, the wing never renders in an "off" state, so there is only ever one indicator appearance (filled green). Do not build a dimmed/hollow "off" variant — it will never be reachable and its absence should not be treated as a missing case during review |
| Right padding from notch edge | 14px — reused verbatim, same legacy hardware-anchored spacing exception documented in `36-UI-SPEC.md` |

**Reference match:** no new reference image was supplied for this HUD (`38-CONTEXT.md` "Specifics" section) — this is a direct, mechanical reapplication of Phase 36's already-approved Droppy-pill wing language, not a new design pass. If a build produces a visual result that reads as "a third HUD wing consistent with the Bluetooth/Charging pair already shipped," it has satisfied this contract.

## Settings Permission Contract (D-01–D-05)

This is the one genuinely new UI surface in this phase. It must exist regardless of which detection path the spike selects (D-02: "one consistent mental model for the user regardless of which technical path won").

### Toggle row

| Property | Value |
|----------|-------|
| Location | Existing Settings sidebar, Activities/Theming section (or its own row — implementation detail, `38-CONTEXT.md` Claude's Discretion) — same section housing the existing Charging/Device/Now-Playing toggles |
| Label | `"Focus Mode HUD"` |
| Control | Standard `Toggle`, off by default (D-01) — same `@AppStorage`-backed pattern as every other activity toggle in `ActivitySettings.swift` |
| Status hint (D-05, locked wording) | A persistent, small secondary-style text directly below/beside the toggle, reflecting REAL permission state — not decorative. Exactly these two strings, no others invented: |

| Permission state | Status hint text |
|---|---|
| Toggle on, permission not yet granted (or denied) | `"Permission needed — tap to grant"` |
| Toggle on, permission granted | `"Active"` |
| Toggle off | *(no hint shown — hint only appears once the toggle is on, since D-02 says authorization is only requested at that moment)* |

Tapping the `"Permission needed — tap to grant"` hint (or the toggle row itself while in that state) re-opens the permission explanation surface below — this satisfies D-04's "no nag, no periodic re-check popup" by making the retry path entirely user-initiated, never automatic.

### Permission explanation surface (shown once, the moment the toggle is switched on — D-02)

Two variants depending on which path the spike selects. Build the one the spike confirms; do not build both speculatively.

**If Full Disk Access path wins (research's likely outcome):**

| Element | Copy |
|---------|------|
| Heading | `"Grant Full Disk Access"` |
| Body | `"Islet needs Full Disk Access to detect when Focus or Do Not Disturb is on. Islet only reads this one system status — it never reads, modifies, or sends any of your files."` |
| Primary button | `"Open System Settings"` — triggers the `x-apple.systempreferences:` deep link (D-03) to Privacy & Security → Full Disk Access, per `38-RESEARCH.md`'s cited URL scheme |
| Secondary/dismiss | `"Not Now"` — dismisses without opening System Settings; toggle stays on (D-04), status hint remains `"Permission needed — tap to grant"` |

**If `INFocusStatusCenter` path wins (low-probability per research, but must be covered so no path is left undesigned):**

| Element | Copy |
|---------|------|
| Heading | `"Allow Focus Status Access"` |
| Body | `"Islet needs permission to detect when Focus or Do Not Disturb is on."` |
| Primary button | `"Continue"` — triggers `INFocusStatusCenter.default.requestAuthorization`, which shows the system's own native TCC-style prompt (no custom copy needed for that system dialog itself — that text is owned by macOS) |
| Secondary/dismiss | `"Not Now"` — same behavior as above |

**If neither path is viable (D-12 descope):** none of the above ships. No toggle, no explanation surface, no half-built permission UI — same clean-abandonment precedent as Phase 37. This UI-SPEC's Settings Permission Contract becomes moot for this phase in that outcome, and REQUIREMENTS.md is updated to drop HUD-05, per `38-CONTEXT.md` D-12.

**Presentation mechanism:** a `.sheet`/`.popover` attached to the Settings toggle row — reuses whatever sheet/popover presentation pattern `SettingsView.swift` already uses elsewhere (planner/executor discretion on the exact SwiftUI presentation modifier; no new presentation pattern needs inventing).

## Verification Notes for Executor

- **Focus wing:** new `focusWings(for:)` in `NotchPillView.swift`, mirroring `wings(for:)`/`deviceWings(for:)` structurally (per `38-RESEARCH.md`'s Code Examples section) — reuse `wingsShape(leftWidth:rightWidth:)`, `Self.wingsLabelWidth`/`Self.wingsSize` constants verbatim, do not invent new sizing constants.
- **Do not build a "Focus Off" render path** — per D-09, there is no visual state for Focus turning off; the resolver simply stops presenting `.focus(...)` and the wing leaves the view tree via the existing morph transition. If a PR review finds an `if focus.isOn { ... } else { ... }` branch inside the wing view itself, that's a sign the resolver-level gating (D-06/D-07/D-09, handled entirely in `IslandResolver`/`NotchWindowController` per `38-RESEARCH.md` §2/§4/§5) leaked into the view layer — flag it.
- **Collapsed-only constraint (D-07) is a resolver concern, not a view concern** — `focusWings(for:)` never needs an "expanded" branch or variant; if Focus is expanded, the resolver's `where !isExpanded` guard (per `38-RESEARCH.md` §2) means `.focus` is never the active presentation in that state, so no expanded-Focus view needs to be built at all.
- **Settings permission surface** is the one genuinely new UI pattern this phase introduces (no prior Settings toggle in this codebase needed a manual OS-permission explanation+deep-link flow — Calendar/Weather/Bluetooth all use real TCC prompts triggered directly by the system). Build only the variant (FDA or `INFocusStatusCenter`) the spike actually confirms; do not scaffold both.
- **Status hint copy is locked verbatim** (D-05) — `"Permission needed — tap to grant"` and `"Active"` are the only two strings; do not paraphrase or add a third "checking..." transitional state unless the spike surfaces a genuine async-check delay that needs one (flag as a gap-closure item if so, don't invent it speculatively).
- Confirm during implementation whether ROADMAP Success Criterion #2's literal "Focus On" text is a hard requirement distinct from the "Focus" label — see the flagged Copywriting Contract row above; resolve before considering the wing visually complete.
