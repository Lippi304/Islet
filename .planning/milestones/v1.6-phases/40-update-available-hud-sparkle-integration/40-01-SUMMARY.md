## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Xcode 26.6 `-scheme` build planner fails embedding Sparkle's xcframework binaryTarget product**
- **Found during:** Task 1 (`xcodebuild build -scheme Islet -configuration Debug`)
- **Issue:** Every `-scheme Islet` build failed with `error: The file "Sparkle" couldn't be opened
  because there is no such file` during an internal (not present anywhere in the `.pbxproj`) "Copy
  Frameworks/Sparkle -> Products/Debug/Sparkle" task. Root-caused via ~20 isolated experiments (all
  ruled out as the cause: `EAGER_LINKING`, `SWIFT_ENABLE_EXPLICIT_MODULES`, `ENABLE_DEBUG_DYLIB`,
  `codeSign` true/false, `ONLY_ACTIVE_ARCH`, global SPM manifest caches, `IsletTests` scheme
  inclusion, destination arch/generic-platform variants). `-project -target Islet` builds (bypassing
  scheme/workspace build planning) never hit it, confirming a toolchain bug specific to Sparkle's
  product type (`.automatic`, wrapping only a prebuilt xcframework `binaryTarget`) rather than a
  project.yml misconfiguration — `MediaRemoteAdapter` (a SOURCE package, `.library(type: .dynamic)`
  explicit) is unaffected. The REAL "Embed Frameworks" phase (proper `Sparkle.framework` bundle,
  correctly signed) already succeeds independently of this bug; only the separate, synthesized,
  bare-name "reflect the product also into `BUILT_PRODUCTS_DIR`" task is broken.
- **Fix:** Added a `prebuildScripts` phase that pre-stages a real (non-symlink) copy of the resolved
  `Sparkle.framework` at the bare-name path (`Frameworks/Sparkle` and `$BUILT_PRODUCTS_DIR/Sparkle`)
  the buggy task expects, so its own dependency-analysis finds the output already up to date and
  skips re-running the broken copy logic — then a `postbuildScripts` phase deletes the bare-name
  copy from inside the app bundle before Xcode's final whole-bundle `CodeSign` step (an un-suffixed
  loose "Sparkle" directory next to the real `Sparkle.framework` otherwise fails bundle-format
  validation there).
- **Files modified:** `project.yml`
- **Verification:** `xcodebuild build -scheme Islet -configuration Debug` exits 0 from a fully clean
  `DerivedData`, reproduced 3 times; `codesign --verify --deep --strict Islet.app` exits 0; final
  `Contents/Frameworks/` contains only `MediaRemoteAdapter.framework` and `Sparkle.framework` (no
  stray bare-name entries).
- **Committed in:** `66e346c` (Task 1 commit)

**2. [Rule 1 - Bug] Sparkle's 3 Info.plist keys never appeared in the built plist via `INFOPLIST_KEY_*`**
- **Found during:** Task 1 (`plutil -p` verification step)
- **Issue:** `INFOPLIST_KEY_SUFeedURL`/`INFOPLIST_KEY_SUPublicEDKey`/`INFOPLIST_KEY_SUEnableAutomaticChecks`
  in `settings.base` never appeared in the built app's `Info.plist`, even with
  `GENERATE_INFOPLIST_FILE: YES` set and other `INFOPLIST_KEY_*` entries (e.g.
  `NSFocusStatusUsageDescription`) synthesizing correctly — Xcode's `INFOPLIST_KEY_*` mechanism only
  recognizes Apple's own well-known Info.plist keys, not third-party ones like Sparkle's. This is
  exactly RESEARCH.md's own predicted Open Question 1 contingency, and the plan's Task 1 action text
  explicitly specifies this fallback.
- **Fix:** Removed the 3 `INFOPLIST_KEY_SU*` lines from `settings.base`; added an `info:`/`properties:`
  block on the `Islet` target instead (XcodeGen's literal-Info.plist-merge feature, generates
  `Islet/Info-Sparkle.plist`, combined with `GENERATE_INFOPLIST_FILE: YES` synthesis for everything
  else).
- **Files modified:** `project.yml`, `Islet/Info-Sparkle.plist` (generated)
- **Verification:** `plutil -p` on the built app's `Info.plist` shows all 3 keys with the exact
  values set in `project.yml`.
- **Committed in:** `66e346c` (Task 1 commit)
