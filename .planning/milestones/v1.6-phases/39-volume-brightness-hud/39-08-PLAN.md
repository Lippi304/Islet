<objective>
Re-attempt native OSD suppression for Volume/Brightness (D-14: `.cghidEventTap` instead of the
already-failed `.cgSessionEventTap`), self-drive the real system volume/brightness/mute value once
suppression works (D-15), and retune `OSDLevelBar`'s fill animation for a snappier single-press
feel (D-16). This is a gap-closure addendum to the already-shipped Phase 39 (39-01 through 39-07,
all complete) â€” it does not touch, renumber, or re-litigate any of that prior work.

Purpose: Phase 39's original spike (39-01) confirmed `.cgSessionEventTap`/`.defaultTap` correctly
decodes and swallows volume/brightness `NX_SYSDEFINED` events but does NOT actually hide the
native OSD on this machine/macOS Tahoe (`suppression-unreliable`). The user re-opened this after
on-device use: the native OSD is still visible ("muss unbedingt weg") and key presses feel
slightly delayed even on a single non-held press. `dannystewart/volumeHUD` (MIT, actively
maintained for Tahoe) proves `.cghidEventTap` + `.defaultTap` + self-driven CoreAudio/
DisplayServices writes works as the real fix for the first problem; the second problem is
addressed as a pure animation-feel retune, since 39-07's own `[OSD-TIMING]` evidence already
conclusively ruled out a backend/pipeline lag (single-digit milliseconds per press).

Output: `OSDInterceptor.swift` rebuilt as a dual-mode HID-level interceptor with a per-type
self-drive kill switch; `VolumeReader.swift`/`BrightnessReader.swift` gain write paths alongside
their existing reads; `OSDLevelBar` gets a materially snappier spring. A mandatory, blocker-severity
on-device checkpoint gates this plan's completion â€” this is genuinely private, undocumented,
system-behavior-dependent territory that cannot be self-certified from code review alone.
</objective>
