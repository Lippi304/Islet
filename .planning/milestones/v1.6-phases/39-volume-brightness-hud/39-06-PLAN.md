<objective>
Add the ONE new Settings UI surface this phase introduces: an opt-in "Replace System Volume/
Brightness OSD" toggle, off by default, with a one-time explanation popover whose primary action
deep-links to System Settings' Accessibility pane — the only piece of this phase's permission UX
that is genuinely new code, since (per `39-RESEARCH.md`'s own explicit finding) Accessibility has
no `requestAuthorization(completion:)`-style API the way Focus Mode's `INFocusStatusCenter` does.

Purpose: closes the loop on D-05/D-06/D-07/D-08 — the toggle exists, is discoverable, explains
itself, and hands the user off to the one place they can actually grant Accessibility, while
`OSDInterceptor`'s own 5s health-check (Plan 39-03) picks up the grant automatically with no
further UI action required.
Output: the toggle row + explanation popover in `SettingsView.swift`'s existing "Activities"
section, mirroring Focus Mode HUD's shipped toggle/popover shell exactly except for the deep-link
body.
</objective>

@AppStorage(ActivitySettings.focusKey) private var focusEnabled = false
@State private var showFocusPermissionExplanation = false
