# Phase 42 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase adds exactly ONE new visual element — a small round secondary bubble rendered alongside (never inside) the existing primary pill/wing presentation, for the one proven pairing (Calendar Countdown primary + Now-Playing secondary). It reuses Phase 35's Liquid Glass material, Phase 41's countdown wing, and Phase 4/30's existing Now-Playing artwork pipeline verbatim. There is no new material, no new window, no new Settings surface, and no new expanded-state layout — the bubble exists only in the collapsed/ambient tier and disappears the moment either the transient tier takes over (D-10) or only one ambient activity remains live (D-04).

## Secondary Bubble Contract (DUAL-01)

**Structural constraint (locked, not a style choice):** the bubble is a SECOND, simultaneously-rendered shape, composed as a sibling to `presentationSwitch` inside `NotchPillView.body`'s outer `ZStack` — never a new `IslandPresentation` case, never composed inside `wingsShape`'s own content `ZStack`. It carries its own `matchedGeometryEffect(id: "secondaryBubble", in: ns)` on the SAME shared `@Namespace private var ns` the rest of the view already uses — a second, distinct `id:` is sufficient to keep it from colliding with the primary pill's `"island"` morph target; a second `@Namespace` is not required (see 42-RESEARCH.md Pattern 2 / Assumption A1).

**Only proven pairing today:** Calendar Countdown (primary) + Now-Playing (secondary). D-05's round-circle shape is the GENERAL contract for any future secondary activity; D-06's real-artwork content is Now-Playing-SPECIFIC. A future secondary activity that is not Now-Playing would need its own bubble-content variant, not designed in this phase (YAGNI, matches D-03's 2-entry table scope).

**Visibility (behavioral, binds the visual spec):** the bubble exists only when BOTH Countdown and Now-Playing are live and the island is in its ambient/collapsed tier — same absence-based pattern as every other conditional wing in this codebase (D-04). It never renders standalone, never renders while `isExpanded`, and is suppressed together with the primary pill by any standing transient (Charging/Device/Focus/OSD, D-10). There is no "empty bubble" state to design — per D-04, `secondary` is simply `nil` and the view tree never mounts it.

### Bubble geometry

| Property | Value | Rationale |
|----------|-------|-----------|
| Diameter | **24pt** | Bottom of CONTEXT.md's locked 24-28pt range — reads unambiguously as subordinate against the 32pt wing / 38pt collapsed-pill height (D-07). Treat as the starting point for on-device tuning per this project's established "small geometry constants get tuned on real hardware" precedent (Phase 39 OSD margin, Phase 41 wing width) — record any change in this document in the same commit as the fix. |
| Gap from primary pill's trailing edge | **8pt** | The `sm` token on this project's 8-point spacing scale (D-08: "small visible gap, not touching/overlapping") — a deliberate, named value, not an arbitrary pixel guess. |
| Vertical alignment | Centered on the primary pill's vertical midline | Matches the general "flanking element centered on the pill" convention `wingsShape` already establishes for left/right wings. |
| Positioning mechanism | Prefer expressing the primary pill + bubble as siblings inside an `HStack(spacing: 8)` (reusing this project's own Phase 39-07 lesson: "default to HStack + explicit-width-spacers... unless a strong reason exists to deviate") over `.offset(x:)`. Only fall back to `.offset(x:)`/`.alignmentGuide` if the `ZStack` composition point genuinely cannot express an `HStack` sibling relationship — verify on-device before committing to whichever mechanism is chosen, since `.offset()` has a documented history of not moving real render position reliably in this exact view hierarchy. | Directly avoids repeating the 16-round OSD-wing geometry saga (STATE.md Phase 39-07) on a brand-new shape. |
| Shape/effect/frame ordering | `.matchedGeometryEffect(...)` MUST be applied BEFORE `.frame(width:height:)` on the bubble's shape — this exact ordering bug has already been hit and fixed 3 times on other call sites in this file (`NotchPillView.swift` lines 850-858, 1803-1808, 1956-1958). Do not regress it on the new shape. | Documented, repeatedly-hit codebase-specific pitfall (42-RESEARCH.md Pattern 2). |

### Bubble content (Now-Playing, the only case this phase implements)

| Property | Value |
|----------|-------|
| Artwork present | Real album-cover artwork, `.resizable().aspectRatio(contentMode: .fill)`, `.clipShape(Circle())` at the bubble's diameter — direct reuse of the existing `artThumbnail(_:side:corner:)` pattern (`NotchPillView.swift:2519`) with `Circle()` substituted for `RoundedRectangle`, per this project's existing artwork-latency handling (art fills in asynchronously once the MediaRemote adapter delivers it, D-06) |
| Artwork absent (latency gap) | `Circle().fill(Color.white.opacity(0.12))` background + centered `Image(systemName: "music.note")` at `.font(.system(size: diameter * 0.45))`, `.foregroundStyle(.white.opacity(0.7))` — same fallback treatment already used elsewhere for missing Now-Playing artwork, adapted to a circular frame; this is a TRANSIENT rendering state (art usually arrives within one or two frames), not a designed "empty state" requiring its own copy |
| Fill/background behind artwork | `islandFill` (Phase 35 shared Liquid Glass material) — same material token every other shape in this view already uses; no new material introduced |
| Tap target | Full 24pt circle, `.onTapGesture` → expands to Home/Now-Playing exactly as tapping Now-Playing-as-primary already does today (D-12) |
| Hover behavior | None — no hover-reveal, no highlight, no scale/opacity change on hover (D-13, mirrors Phase 41 D-08's "no hover-reveal" precedent for the Countdown pill) |

### Reveal/dismiss animation (D-09, D-11)

| Transition | Mechanism | Value |
|------------|-----------|-------|
| Bubble appears when both activities become live | `matchedGeometryEffect` spring morph, same curve as every other shape transition in this view | `response: 0.6, dampingFraction: 0.62` (the project's single shared `springResponse`/`springDamping` constants — no new spring curve for this phase) |
| Staggered reveal after a transient ends (D-11) | Primary pill reappears immediately inside the existing `withAnimation(.spring(...))` block; the bubble's own state mutation is scheduled on a SEPARATE `DispatchWorkItem` + `asyncAfter`, wrapped in its own `withAnimation(.spring(...))` when it fires — mirroring the existing `graceWorkItem`/`dismissWorkItem` delayed-mutation convention, not a new sequencing primitive | Stagger delay: **150ms** (midpoint of the 100-200ms Apple-Dynamic-Island-precedent range noted in 42-RESEARCH.md Assumption A2) — starting value for on-device tuning, not a hard-locked number |
| Bubble disappears (activity drops to single-ambient or a transient starts) | Same `matchedGeometryEffect`-driven morph-out as its appear transition — no separate fade-only exit; `.transition(.scale.combined(with: .opacity))` on the `if let secondary` conditional as the SwiftUI-side complement to the geometry effect | Same spring constants as appear |

## Verification Notes for Executor

- **Do the ranking/resolver work first, the bubble view second.** D-01/D-02/D-03's ordered-pair table belongs entirely in `IslandResolver.swift`'s pure `resolve()` ambient branch — the view must be a pure consumer of a `secondary: SecondaryActivity?` field, never compute precedence itself (Anti-Pattern in 42-RESEARCH.md, "one pure arbiter" discipline).
- **Mandatory early on-device spike before building the bubble's visual layer:** confirm whether tapping the far-right edge of an EXISTING wing (e.g. the Charging percentage text) currently registers a click at all. `NotchWindowController`'s `hotZone` is fixed to the small ~179×32pt collapsed-pill rect (`NotchWindowController.swift:995`), not to the wider wing-tier rendering width — a bubble positioned further out than that rect risks silently swallowing taps, the same failure class as the Phase 40 badge-tap regression (STATE.md). This is 42-RESEARCH.md's single highest-risk item (Pitfall 2) — resolve it before investing in bubble polish, not after.
- **`secondary` must be cleared in the SAME code path that sets a transient presentation**, not left to a separate cleanup pass — a naive split (two independently-mutated `@Published` fields) can leave a stale bubble visible during Charging/Device/Focus/OSD (42-RESEARCH.md Pitfall 1). Whatever return shape `resolve()` uses (tuple/struct), the caller must set both `presentation` and `secondary` from ONE call, every time.
- **New shape call must use a distinct literal `matchedGeometryEffect` id** (`"secondaryBubble"`, not `"island"`) — grep for `"island"` after implementation to confirm no accidental copy-paste from `wingsShape`/`blobShape` left the wrong id in place (42-RESEARCH.md Pitfall 3).
- **Circular artwork crop** — extend the existing `artThumbnail(_:side:corner:)` helper or add a small sibling `artThumbnailCircular(_:diameter:)` using `.clipShape(Circle())`; do not hand-roll a custom `Shape`/`Path` mask (Don't Hand-Roll table, 42-RESEARCH.md).
- **Stagger delay (D-11) reuses the existing `DispatchWorkItem`/`asyncAfter` convention** (`graceWorkItem`/`dismissWorkItem` precedent) — do not introduce a new sequencing/state-machine abstraction for what is fundamentally one delayed state mutation.
- **Confirm on-device** that the 24pt diameter / 8pt gap actually reads as "clearly secondary" next to the real 38pt collapsed pill and 32pt wings on this project's actual notch geometry (179×32pt measured) — adjust within the 24-28pt range if needed and update this document in the same commit, per the `41-UI-SPEC.md` precedent for this exact class of tuning.
- **No hover state to build** for the bubble (D-13) — confirm the `.onTapGesture` handler has no accompanying `.onHover` modifier, matching the Countdown pill's own no-hover-reveal precedent.
