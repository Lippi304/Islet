# Milestone v1.0.1: Pre-Release Polish

**Status:** ✅ SHIPPED 2026-07-04
**Phases:** 7-9
**Total Plans:** 9 (4 executed with code changes, 4 conditional no-ops, 1 precondition-skipped)

## Overview

Closed the two remaining polish gaps before Islet's first real release: a display-only Now Playing progress bar (PBAR-01) and eliminating the fullscreen-enter island flash (FS-01). FS-01 required two phases — Phase 8 investigated and disproved one candidate signal, escalating with a documented root-cause report; Phase 9 retried with a window/Space architecture change (a dedicated max-level private CGS Space) that resolved it on the first wave of a 5-wave conditional chain.

## Phases

### Phase 7: Now Playing Progress Bar

**Goal**: Users can see exactly where playback is within the current track, at a glance, from the expanded island — display-only, no interaction.
**Depends on**: Phase 4 (Now Playing, v1.0)
**Requirements**: PBAR-01
**Success Criteria** (what must be TRUE):

  1. The expanded Now Playing view shows a horizontal progress bar reflecting the current track's playback position relative to its total duration.
  2. Elapsed and remaining/total time labels are visible next to the bar (e.g. "1:23 / 3:45").
  3. The bar and labels update smoothly while a track is playing, and hold perfectly still (no drift) while paused.
  4. The bar is strictly display-only — clicking or dragging it does not seek or change playback in any way.

**Plans**: 1 plan

- [x] 07-01-PLAN.md — Extend pure seam + monitor + state to carry playback position; render ProgressBar in expanded Now Playing view; on-device UAT

**Details:** On-device UAT caught and fixed a pause-transition backward-flash bug (stale MediaRemote sample corrected via a drift-extrapolated freeze, `resolvePublishedPosition(...)`). A post-execution code review closed a NaN/Infinity crash risk. 141/141 tests green, verified passed in `07-VERIFICATION.md` (7/7 truths).

### Phase 8: Fullscreen-Enter Flash Elimination

**Goal**: Entering true (native) fullscreen on the built-in display never produces a visible island flash, closing out the polish debt v1.0 shipped with.
**Depends on**: Phase 2 (Hover, Expand & Fullscreen Hardening, v1.0)
**Requirements**: FS-01
**Success Criteria** (what must be TRUE):

  1. On-device, entering native fullscreen (tested across multiple trigger methods — green-button, menu bar, video apps, at minimum) shows zero visible island flash during or after the transition, across repeated trials.
  2. The fix is a genuine root-cause elimination using a detection/timing signal distinct from v1.0's reactive `orderOut` approach (which was already confirmed insufficient) — not a best-effort reduction.
  3. Existing fullscreen behavior is not regressed: the island still hides for the duration of fullscreen and still restores correctly on exit.

- [x] 08-03-PLAN.md — Wave 1 (escalation path): revert exploratory code, write root-cause escalation report, surface scope decision to user

**Details:** The on-device D-05 trigger matrix found the CGS event 106/107 candidate signal never fires cross-process (option-c, Candidate A disproven). 08-03 reverted all exploratory code byte-for-byte and produced `08-ESCALATION.md`. ## Milestone Summary

**Key Decisions:**
- Two requirements (PBAR-01, FS-01) split into two phases (7, 8) rather than combined — different risk profiles: PBAR-01 straightforward SwiftUI/timer work; FS-01 required fresh on-device investigation of alternative fullscreen-transition detection signals.
- FS-01 scoped as a full root-cause elimination outcome, not a best-effort/partial mitigation — explicitly excluded partial reduction from scope (REQUIREMENTS.md Out of Scope).
- Phase 8's on-device investigation found its one new candidate signal (CGS 106/107) never fires cross-process; rather than silently drop the requirement, the phase escalated with a documented root-cause report and a user-facing scope decision — resulting in Phase 9, not a v1.0.1 ship with unresolved FS-01.
- Phase 9's Candidate C was implemented as an ADDITIVE layer (existing `.canJoinAllSpaces` behavior untouched) rather than a replacement — the only variant with real shipping precedent in the researched reference apps.
- v1.0.1 (not v1.1) — the app had not been publicly released yet, so continuing under the v1.0 line was chosen over bumping to v1.1.

**Issues Resolved:**
- Now Playing progress bar shipped with a pause-transition backward-flash bug found and fixed during on-device UAT.
- Fullscreen-enter island flash — root-caused as a per-Space auto-join race in `.canJoinAllSpaces`, eliminated via a dedicated max-level CGS Space, confirmed zero-flash on-device across all trigger methods with zero regressions.

**Technical Debt Incurred:**
- CR-01 (Phase 9): the new CGS Space leaks in WindowServer on every normal app quit because `AppDelegate.quit()` never tears down `NotchWindowController` before calling `NSApp.terminate(nil)`. Non-blocking, recommended for a `/gsd-quick` fix or early v1.1 housekeeping.
- WR-01/WR-02 (Phase 9, info): no validation of CGS private-API return values; an `Int`/`Int32` width assumption in `CGSSpaceSetAbsoluteLevel`. Low severity.
- IN-01/IN-02 (Phase 7, info): magic-number extraction and untested View-layer math in `ProgressBar`.
- Carried over from v1.0 close (unchanged, not new debt): Phase 2's 8 on-device UAT scenarios (`02-HUMAN-UAT.md`) remain unexercised; pre-existing `BluetoothMonitor` test hang in non-interactive/sandboxed environments.
