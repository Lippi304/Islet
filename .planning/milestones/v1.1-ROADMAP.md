### Phase 10: Trial & Lockout Gate

**Goal**: The app enforces a real, tamper-resistant 3-day trial with a hard functionality lockout â€” proven end-to-end using a manually-settable stub license state, with no live network dependency.
**Depends on**: Nothing new (builds on the existing v1.0.1 codebase, specifically `NotchWindowController`'s single-arbiter `shouldShow(...)`)

**Success Criteria** (what must be TRUE):

1. On first launch, the trial start timestamp is persisted to the Keychain â€” running `defaults delete` on the app or deleting/reinstalling it does not reset the trial clock.
2. On first launch (and only on first launch), the user sees an explicit one-time notice stating the 3-day trial has started â€” never a silent start.
3. With the trial active or a stub license flagged valid, the island behaves exactly as before (no regression to existing v1.0/v1.0.1 behavior).
4. With the trial expired and the stub license flagged invalid/absent, the island is fully locked â€” no pill, no activities, no expansion â€” until the stub flips to valid.
5. Flipping the stub license from invalid to valid un-locks the island at the next natural UI transition, not as an abrupt mid-interaction yank.

## Milestone Summary

**Key Decisions:**

- Phase order followed research SUMMARY.md verbatim: Phase 10 (trial/lockout on a stubbed license state) before Phase 11 (Settings UI against a stubbed LicenseService) before Phase 12 (real Polar.sh integration) â€” de-risked the single-arbiter `shouldShow(...)` and the UI state machine before live network flakiness was introduced.
- LIC-01/LIC-02 mapped to Phase 12, not Phase 11, even though Phase 11 built their UI shell â€” the real observable behavior (live checkout, real online validation) isn't true until the `PolarLicenseService` swap.
- LIC-03 (hard lockout) mapped to Phase 10, not Phase 12 â€” the lockout mechanism was fully built and testable against a manually-settable stub before real license validation touched it.
- Real Developer-ID notarization bundled with the paywall in one milestone (not split) â€” shipping a paywall without real notarization means every paying customer's first launch is a Gatekeeper warning, a broken first impression for something they just paid for.

**Issues Resolved:**

- Release-build launch crash (dyld, embedded `MediaRemoteAdapter.framework` failing Library Validation under Hardened Runtime) â€” fixed via `disable-library-validation` entitlement (quick task 260705-mzj, pre-clears a Phase 13 blocker).
- Phase 13: nested `MediaRemoteAdapter.framework` signing/entitlement mismatch â€” embedded frameworks now explicitly re-signed with the real Developer ID before the outer `.app` (`codesign` doesn't recurse; `--deep` is deprecated).
- Phase 13: `notarytool submit` requires a `.zip`/`.pkg`/`.dmg`, not a raw `.app` â€” script now zips via `ditto -c -k --keepParent` before submission.
- Phase 12: Polar.sh API error taxonomy verified against the real production API and mapped to a strict HTTPâ†’verdict split (network/5xx vs. actually-invalid key), so a transient error never hard-locks a key the user just paid for.
