# Phase 39 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** Two new collapsed-only wings (HUD-03 Volume, HUD-04 Brightness) sharing one bar-style visual language, plus one new Settings permission surface (D-05–D-08) for the optional OSD-suppression toggle. Renders inside Phase 35's Liquid Glass material and the existing split-wing `wingsShape()` structure — no new chrome, no new material, no new window.

**Resolves 39-RESEARCH.md Open Question 1 (toggle scope):** per `39-CONTEXT.md` D-06, the Settings toggle gates ONLY suppression, not the HUD itself. The Volume/Brightness HUD renders unconditionally on every key press (no permission, no toggle, no opt-in) — it is baseline behavior identical in spirit to Charging/Device/Focus's own always-on HUD rendering. Only the native-OSD-*suppression* half of the feature is opt-in and Accessibility-gated. **There is no "HUD disabled" visual state to design** — do not build a settings row that hides the Volume/Brightness HUD entirely; no such control exists per this decision.

## OSD Wing Contract (HUD-03, HUD-04)

**Structural constraint (D-01, locked — not a style choice):** reuses the existing split-wing `wingsShape(leftWidth:rightWidth:)` helper flanking the physical camera bump — same structural family as Charging/Device/Focus. This is a deliberate divergence from the Droppy reference screenshot's own layout: `reference-droppy-volume-charging-pills.png` (Phase 36) shows Volume/Brightness as a single continuous pill (icon + bar in one unbroken capsule) because Droppy has no physical notch to flank. Islet's camera bump forces the existing two-flank structure instead — the icon lives in the LEFT wing, the bar lives in the RIGHT wing, exactly as `39-CONTEXT.md` D-01 specifies. Do not attempt to build a single continuous pill; it does not fit this project's window-shell reality.

**Content divergence from Phase 36 (D-01, locked):** icon + bar, NO label text and NO numeric percentage anywhere — a deliberate departure from Phase 36's icon+label convention (Charging "Charging", Bluetooth "Connected", Focus "On"). A continuous fill level reads better as a bar than as text or a label, and this project has an exact visual reference for it (see below).

**Visibility (behavioral, binds the visual spec):** the wing exists only for ~1.5s after a volume/brightness key press (D-10's own shorter auto-dismiss, re-armed on every subsequent press within the window — D-09), and only while the island is collapsed (D-11 — same collapsed-only scoping as Focus, Phase 38 D-07; hovering to expand mid-adjustment simply removes the wing, no expanded-state variant exists). There is no "idle"/"nothing pressed" visual — the wing simply isn't in the view tree between key presses, same absence-based pattern as Charging's wing on unplug.

| Icon size | `.font(.system(size: 13, weight: .semibold))`, `.symbolRenderingMode(.hierarchical)` — identical size/weight/rendering-mode to every other wing icon in this codebase (Charging bolt, Focus moon, Bluetooth glyph) |
| Icon color (D-02, locked) | Fixed `Color.white` — NEVER `deviceAccent`/`chargingAccent`/`nowPlayingAccent` or any theme-selectable accent, for BOTH the icon and (per the Right wing section below) the bar. Same "universal system-level state ignores the user's accent theme" rule as Charging's bolt and Focus's icon+dot (D-11 in `38-UI-SPEC.md`) |

Do not start from the older `Self.wingsSize.width / 2` (145pt) default — Focus's 118pt is the current, on-device-proven icon-only-flank value in this codebase as of Phase 38 |

### Right wing (horizontal filled bar — the primary content)

The bar is the focal element of this HUD (no numeric text exists anywhere, so the bar's fill length alone communicates the level — unlike Charging/Device where a `BatteryIndicator`'s number is the primary readout and the fill is secondary confirmation).

| Property | Value |
|----------|-------|
| Structure | A `ZStack(alignment: .leading)` two-layer bar — NOT `BatteryIndicator` (that component's outline+nub+centered-% chrome is battery-specific and explicitly excluded by D-01's "no numeric text" rule). Build a new minimal two-layer bar: empty track + proportional fill, mirroring `BatteryIndicator.swift`'s existing fill-track pattern (`Islet/Notch/BatteryIndicator.swift`) but stripped of the outline stroke, terminal nub, and centered `Text` |
| Track (empty/background layer) | `Capsule().fill(Color.white.opacity(0.15))` — same opacity family as `BatteryIndicator`'s `Color.white.opacity(0.12)` empty-track convention |
| Fill (level layer) | `Capsule().fill(...)`, left-anchored, width proportional to `level / 100`, clamped `0...100` (same defensive-clamp convention as `BatteryIndicator.level`) |
| Fill color, Volume (D-02, locked) | Fixed `Color.green` — matches the reference screenshot exactly (`reference-droppy-volume-charging-pills.png`) |
| Fill color, Brightness (D-02, locked) | Fixed `Color.orange` — matches the reference screenshot's amber/yellow bar (closest system-palette color; do not introduce a custom hex, `Color.orange` is the established system-color convention this codebase already uses for low-battery amber) |
| Bar dimensions | **90pt** width (the originally requested size), height 5pt, corner radius 2.5 (full capsule) |
| Fill animation (D-04, locked) | Spring, not instant snap — `.animation(.spring(response: 0.35, dampingFraction: 0.75), value: level)`, applied directly to the fill layer via `value:`, independent of and faster than the outer wing-morph spring the controller wraps every OSD mutation in |
| Muted state (D-03, locked) | Fill width animates to 0 (fully drained) AND the left-wing icon swaps to `speaker.slash.fill` simultaneously — both changes happen together, never one without the other, so "muted" and "0%" never look ambiguous relative to each other (they're the same visual state by design) |
| Trailing padding from notch edge | **20pt** — the original UI-SPEC starting value |
| `leftWidth`/`rightWidth` | **both derived, not hardcoded** (ROUND 10) — computed in `osdWings(for:)` from an EXPLICIT excluded x-range (`excludedMinX`/`excludedMaxX`) built directly from the LIVE `interaction.collapsedNotchSize`, with the icon and bar each placed by absolute `.offset(x:)` relative to that range — not from an HStack `Spacer()`'s leftover space |

**Combined recommended wing call:** `wingsShape(leftWidth: <derived>, rightWidth: <derived>)` — both sides now vary per-machine with the real notch width.

**ROUND 10 — explicit hard exclusion zone (corrects ROUND 9's own remaining gap):** ROUND 9 correctly started reusing the live `collapsedNotchSize` measurement, but only proved the BAR's position algebraically — the ICON's position (`osdLeftWidth`, left at the old hardcoded 118) was never actually checked against the real exclusion zone, and the HStack's `Spacer()` was used as an implicit stand-in for "the camera is somewhere in there," which is backwards. The user's own diagnosis: "you must not use the spacing between the two icons as a proxy for the camera — you need to actually hard-block that middle region so nothing can ever go there." ROUND 10 computes the excluded x-range FIRST (`excludedMinX`/`excludedMaxX`, both derived directly from `collapsedNotchSize`, with the one safety margin folded into `notchHalfWidth` a single time) and places every element — icon AND bar — by absolute `.offset(x:)` relative to that range, inside a `ZStack`, not an `HStack` with a `Spacer()`. The icon's own box is now a CODE-GUARANTEED fixed size (14pt leading pad + a 20pt fixed glyph frame = 34pt), not a font-rendering estimate, so `osdLeftWidth` is DEFINED as `notchHalfWidth + iconBoxWidth` — making `excludedMinX == iconBoxWidth` exactly, by construction. Two Debug-only `assert`s check the hard invariant directly: the icon's box must end at-or-before `excludedMinX`, and the bar's leading edge must start at-or-after `excludedMaxX`.

**ROUND 11 — the actual root cause, in the SHARED `wingsShape` helper, not `osdWings(for:)` itself:** ROUND 10's exclusion-zone math was correct, but rendered wrong anyway. Root cause, found by direct code read (not another guess): `wingsShape`'s own `.overlay(content().frame(width: size.width, height: size.height))` had no `alignment:` parameter, so SwiftUI defaulted to `.center` — centering `content()` inside that box based on `content()`'s own NATURAL/intrinsic size, not pinned to the box's leading edge. Every other wing (Charging/Device/Focus) builds `content()` as `HStack(spacing: 0) { ... Spacer() ... }`; a `Spacer()` makes the HStack's natural width expand to exactly fill whatever width it's proposed, so its natural size already equalled `size.width` and centering was always a no-op for them (confirmed by reading all 3 other call sites — none behave differently under `.leading` vs the old `.center` default). ROUND 10's OSD content is a `ZStack` with no `Spacer()`, positioning icon/bar via `.offset(x:)` instead — and `.offset()` is a pure render-time transform that does not contribute to a view's reported layout size to its ancestors (documented SwiftUI behavior). That ZStack's natural width was therefore just its widest un-offset child (~90pt, the bar's own fixed frame), nowhere near the real `size.width` (250-300pt+ after the exclusion-zone math) — so the old implicit `.center` silently shifted the whole ZStack (and every offset computed relative to its origin) rightward by `(size.width - ~90) / 2`, dragging the icon behind the camera and pushing the bar's offset origin far enough right that it rendered outside the visible pill shape entirely. Fix: `alignment: .leading` added to that one `.frame()` call in the shared helper.

**Superseded history (rounds 6-9, kept for context):** ROUND 6 added a temporary on-device calibration ruler and measured an apparent boundary at local x=100 — later found to be a confounded reading, not pure camera occlusion. ROUND 7 built `rightWidth` from that (now-removed) constant; its own report additionally mis-stated the result as "122pt" when the code actually computed 102pt (a documentation typo). ROUND 8 fixed the resulting zero-slack layout bug but the underlying constant was still wrong. ROUND 9 replaced the constant with the live `collapsedNotchSize` measurement (a real improvement) but only for the bar's position, still leaving the icon's own position unverified and the gap between them implicit via `Spacer()`. ROUND 10 replaces the whole positioning model.

**On-device tuning correction (39-07, ROUND 3 — empirical re-derivation, supersedes ROUND 2's theoretical notch-half-width math):** ROUND 2 assumed a ~89.5pt notch half-width and only shrunk the bar's WIDTH (150pt -> 76pt), which moves the bar's LEFT edge but — since the RIGHT edge is anchored by a fixed trailing pad, independent of width — leaves the right edge unchanged. A third on-device round showed the fill was **only visible once it reached roughly 60% or more** — i.e., the bar's own LEFT/starting edge was still under the camera housing the whole time; a fill only pokes into visible territory once its own right edge (`trackLeft + fraction * barWidth`) crosses the hidden boundary. This observation gives the needed correction directly from the data, without depending on the (evidently unreliable) notch-half-width estimate: at ROUND 2's 76pt bar, the hidden boundary sat roughly `0.6 * 76 ≈ 46pt` inside the track's left edge, so the whole track needed to shift right by about that much (plus a margin for "~60% or so" being an estimate, not a precise reading). Shrinking the bar alone could not deliver a 46pt+ shift (trailing pad was already down to 14pt, leaving little further room), so this round combines two changes: `rightWidth` grown 190pt -> 205pt (shifts both edges of the bar right without shrinking it further) plus the bar shrunk again 76pt -> 44pt (moves the left edge further right, trailing pad unchanged at 14pt) — together moving the track's left edge right by ~47pt, matching the empirically-required shift. If still wrong, get a fresh onset-percentage reading from the user and solve the same way (`shift ≈ onsetFraction * priorBarWidth`) rather than re-deriving notch geometry from theory.

**Reference match:** `reference-droppy-volume-charging-pills.png` (Phase 36, still on disk in `.planning/phases/36-cosmetic-restyles-signature-animation/`) is the canonical visual reference for bar color/style — green volume bar, orange/yellow brightness bar, icon-left/bar-right layout — reinterpreted into this project's own two-flank structure per the Structural constraint above, not copied as a single continuous pill.

**Spacing exception:** the 14pt/20pt flank padding and the bar's 150×5pt/2.5-corner-radius dimensions are not 4pt-grid multiples — same class of exception as `36-UI-SPEC.md`'s hardware-anchored padding: these are physical-notch-geometry-constrained (padding) or primary-content-element-sized (bar), not gap/padding values subject to the grid rule.

## Settings Permission Contract (D-05–D-08)

This is the one new UI surface this phase introduces beyond the two wings — directly mirrors `38-UI-SPEC.md`'s Settings Permission Contract structure (same toggle → status hint → one-time explanation-with-deep-link pattern), adapted for Accessibility instead of Focus's permission path. Per `39-RESEARCH.md`'s Assumptions Log note, Accessibility genuinely has no completion-based re-request API the way `INFocusStatusCenter` does — so unlike Focus (which ended up NOT needing a deep link), this toggle's `x-apple.systempreferences:` deep link is real, new code, not a copy-paste of existing app behavior.

### Toggle row

| Property | Value |
|----------|-------|
| Location | Existing Settings sidebar, same "Activities" section housing Charging/Now Playing/Devices/Focus Mode HUD toggles (`SettingsView.swift` ~L202-228) — or its own adjacent row; exact placement is executor discretion, matching `38-UI-SPEC.md`'s precedent for this same open question |
| Label | `"Replace System Volume/Brightness OSD"` — deliberately does NOT say "Volume/Brightness HUD" (that would incorrectly imply the toggle controls whether the HUD shows at all, which it does not per D-06 — see the Scope note at the top of this document) |
| Control | Standard `Toggle`, **off by default** (D-05, mirrors Focus's D-01 exactly) — same `@AppStorage`-backed pattern as every other toggle in `ActivitySettings.swift`/`SettingsView.swift` |
| Status hint (mirrors D-05's locked `ActivitySettings.focusPermissionStatusHint` wording verbatim) | Reuse the SAME two strings already shipped for Focus — do not invent new copy: |

| Permission state | Status hint text |
|---|---|
| Toggle on, Accessibility not yet granted (or denied) | `"Permission needed — tap to grant"` |
| Toggle on, Accessibility granted | `"Active"` |
| Toggle off | *(no hint shown — same as Focus's pattern: nothing is requested/shown until the user opts in)* |

Tapping the `"Permission needed — tap to grant"` hint (or the toggle row itself in that state) re-opens the explanation popover below — same user-initiated-only retry pattern as Focus's D-04 (no nag, no periodic re-check).

### Explanation popover (shown once, the moment the toggle is switched on)

| Element | Copy |
|---------|------|
| Heading | `"Replace System OSD"` |
| Body | `"Islet needs Accessibility access to hide the native volume/brightness indicator. Islet only intercepts volume and brightness key presses — it never reads, modifies, or sends anything else on your Mac."` |
| Primary button | `"Open System Settings"` — triggers `NSWorkspace.shared.open(URL(string: "x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility")!)` (D-08, `39-RESEARCH.md` confirms this is genuinely new code, not a reuse of an existing deep-link call site) |
| Secondary/dismiss | `"Not Now"` — dismisses without opening System Settings; toggle stays ON (mirrors Focus's D-04), status hint remains `"Permission needed — tap to grant"`, and per D-06 the Volume/Brightness HUD keeps showing (unsuppressed, alongside the native OSD) regardless of this dismissal |

**Presentation mechanism:** `.popover`, matching Focus's exact shipped pattern (`showFocusPermissionExplanation`/`focusPermissionExplanationView` in `SettingsView.swift` ~L217-219, ~L261) — reuse the same `.onChange(of:)` + `.popover(isPresented:)` wiring shape, new `@State` boolean, new view.

### D-07: no visible UI change for auto-recovery

If Accessibility is granted later in System Settings while Islet is already running with the toggle on, suppression starts automatically (health-check-timer reinstall, per `39-RESEARCH.md` Pitfall 2) — this has **no corresponding visual change to design**: the status hint text updates from `"Permission needed — tap to grant"` to `"Active"` the next time Settings re-renders (already covered by the status-hint mapping above; no new transitional/loading state needed).

## Verification Notes for Executor

- **New wing views:** `volumeWings(for:)` / `brightnessWings(for:)` (or one shared `osdWings(for activity: OSDActivity)` if the planner adopts the shared-case model `39-CONTEXT.md`/`39-RESEARCH.md` both recommend) in `NotchPillView.swift`, mirroring `focusWings(for:)`'s structure (single icon left, `Spacer()` to clear the camera bridge, content right) — reuse `wingsShape(leftWidth:rightWidth:)` verbatim, do not invent a new wrapper shape.
- **Do not reuse `BatteryIndicator`** for the bar — its outline/nub/centered-`Text` chrome directly conflicts with D-01's "no numeric text" rule. Build a new minimal two-layer `Capsule` track+fill (see Right wing section) — small enough it may not warrant its own file; a private helper view or inline `ZStack` in `NotchPillView.swift` is fine, executor's call.
- **118/190 leftWidth/rightWidth split is a starting point, not a lock** — flag explicitly for the phase's own on-device UAT checkpoint (Success Criterion 1 territory). If it needs adjustment, update this section of `39-UI-SPEC.md` in the same commit as the fix, so this document does not go stale the way `38-UI-SPEC.md`'s Focus Wing Contract did (per `STATE.md`'s "flag for a docs-sync pass" note on that exact failure mode).
- **Muted state (D-03) is a single combined transition**, not two independent toggles — the icon swap and the bar-drain-to-0 must be driven by the SAME `OSDActivity` mapping function's `isMuted` computed property (per `39-RESEARCH.md`'s recommended `isMuted = hardwareMuted || percent == 0` pure-function shape), never two separately-triggered UI states that could desync.
- **Brightness has no muted-equivalent state** — do not build an "off" icon variant for brightness; `sun.max.fill` is the only icon this HUD ever shows.
- **Settings toggle wording is deliberate** (`"Replace System Volume/Brightness OSD"`, not `"Volume/Brightness HUD"`) — this is not a stylistic choice, it's the literal boundary of what the toggle controls (suppression only, per D-06). If a build accidentally gates the HUD's own visibility behind this toggle, that's a functional regression against a locked decision, not just a copy mismatch — flag it during review.
- **Settings permission popover** reuses Focus's exact shipped `.popover`/`.onChange` wiring shape (`SettingsView.swift` ~L211-228, ~L261-280) — new `@State` boolean + new view body, same structural pattern, not a novel presentation mechanism.
- Confirm during implementation that the bar's spring constants (`response: 0.35, dampingFraction: 0.75`, starting values only) feel consistent with the project's established fluid-island spring language when scrubbing volume/brightness rapidly — this is exactly the scenario `39-RESEARCH.md` Pitfall 3 flags as the highest-risk spot for visual stutter; profile on-device if the bar visibly skips values during rapid key repeats.
