# Phase 14 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform note:** Islet is a native macOS SwiftUI/AppKit app, not a web app. shadcn, Tailwind,
and hex-based web color tokens do not apply. This contract adapts the standard UI-SPEC
template to SwiftUI idioms: points (pt) instead of px, `Color`/`Font` values instead of hex,
SF Symbols instead of a web icon library, and no component registry — following the same
adaptation convention established in `07-UI-SPEC.md` and `11-UI-SPEC.md`. All values below
are sourced from `14-CONTEXT.md` (locked decisions D-01–D-07), `14-RESEARCH.md` (Standard
Stack, Architecture Patterns, Pitfalls), and the existing, unmodified code in
`Islet/Notch/NotchPillView.swift` (`expandedIdle`/`expandedIsland`, lines ~137–195, and the
`wings(for:)`/`deviceWings(for:)` HStack pattern, lines ~221–295, read directly this session).

This phase touches **only** the `expandedIdle` → `expandedIsland` branch. No other
presentation case (wings, media, charging, device, collapsed pill) is in scope.

## Spacing Scale (this phase)

Islet's existing UI uses SwiftUI points, not a strict 4px web multiple (existing values of 6,
10, 12, 16, 19, 32 already appear in this file). The tokens below are the phase-specific,
prescriptive values for the new 3-column layout — declared exactly so the executor doesn't
improvise:

| Token | Value | Usage |
|-------|-------|-------|
| `expandedSize` | 360×144pt — **unchanged** | No growth needed: the 3-column content (~40pt tall) fits comfortably inside the existing box; unlike Phase 7's progress bar, this phase adds no vertical rows |
| Horizontal edge padding | 16pt each side | Matches `14-RESEARCH.md` Pattern 4; usable width = 360 − 32 = 328pt |
| Weather column width | 90pt, fixed frame | LEFT column (icon + temperature), `alignment: .center` |
| Calendar column width | 100pt, fixed frame, `alignment: .trailing` | RIGHT column — bounds untrusted event-title length (V5 mitigation) before `.lineLimit(1)` even applies |
| Center column | no fixed frame — sizes to content (~60–70pt) | Naturally centered by the two `Spacer()`s flanking it, mirroring `wings(for:)`'s left/`Spacer`/right shape extended with a middle element |
| Icon ↔ temp vertical gap | 4pt | `VStack(spacing: 4)` in the weather column |
| Time ↔ date vertical gap | 2pt | `VStack(spacing: 2)` in the center column (matches the existing artist/title `spacing: 1`–`2` convention) |
| Today/Tomorrow label ↔ title ↔ time-range gaps | 2pt each | `VStack(alignment: .trailing, spacing: 2)` in the calendar column |
| Calendar color dot size | 6×6pt `Circle()` | Matches the reference image's colored-dot cue |
| Overlay alignment | **default (centered)**, not `.top` | The 3-column content (~40pt tall) is far shorter than the 144pt box; centered placement lands content at roughly y=52–92pt, clearing the 32pt camera band by a comfortable margin without needing an explicit `.padding(.top, 32)` — unlike `mediaExpanded`, which MUST top-pin because its content is 84–100pt tall. This is a deliberate, simpler choice for this phase, not an oversight. |

Exceptions: Islet's spacing is point-based SwiftUI layout, not a 4px web design-token scale —
this is a pre-existing, established project convention, not a new exception introduced by
this phase.

**Insertion point (exact):** replaces the single `Text(Date.now, format: .dateTime.hour().minute())`
overlay inside `expandedIsland` (`NotchPillView.swift`, current lines ~187–193) with the
3-column `HStack` described below.

| Role | Size | Weight | Design | Line Height | Color |
|------|------|--------|--------|-------------|-------|
| **Time (NEW, center, large)** | **20pt** | **semibold** | `.rounded` | default | `.white`, `.monospacedDigit()` |
| **Date (NEW, center, small)** | **11pt** | **regular** | `.rounded` | default | `.secondary` (grey) |
| **Weather temperature (NEW, left)** | **15pt** | **semibold** | `.rounded` | default | `.white`, `.monospacedDigit()` |
| **Today/Tomorrow label (NEW, right)** | **10pt** | **semibold** | `.rounded` | default | `.white` |
| **Calendar event title (NEW, right)** | **12pt** | **regular** | `.rounded` | default | `.white`, `.lineLimit(1)`, `.truncationMode(.tail)` |
| **Calendar event time (NEW, right)** | **10pt** | **regular** | `.rounded` | default | `.secondary` (grey), `.monospacedDigit()` |
| Weather icon glyph size | 20pt | n/a (symbol, not text) | — | — | `.multicolor` rendering (see Color section) |

**Weights introduced by this phase: exactly 2** — regular (400) and semibold (600). (The
file's other, unrelated branches use bold/medium elsewhere — e.g. `mediaExpanded`'s 15pt bold
title — those are pre-existing and untouched by this phase.)

**Time format:** `.dateTime.hour().minute()` — same locale-aware formatter already used by
the current placeholder (no fixed 12/24-hour choice; inherits system locale).
**Date format:** `.dateTime.weekday(.abbreviated).day().month(.abbreviated)` (e.g. "Wed 8 Jul").
**Calendar event time format:** `.dateTime.hour().minute()` — a **single start time**, not a
range. `14-RESEARCH.md`'s `CalendarGlance` model (Pattern 3) carries only `title`, `startDate`,
`isToday` — no end time was locked in `14-CONTEXT.md`. The reference image's "time range"
visual detail is descoped to a single start time for this phase; if the planner later extends
`CalendarGlance` with an end time, the display can upgrade to an en-dash range
(e.g. "14:00–15:00") as a follow-up, not required here.

## Color

Islet has no 60/30/10 web-style color split — it's a single black rounded "island" overlay,
not a page layout. Adapted to the app's actual visual system:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (island surface) | `Color.black` | The island shape itself — **unchanged by this phase** (D-05: background stays exactly as-is; only the weather icon animates) |
| Secondary (text) | `Color.secondary` (system grey) | Date text, calendar event time text — same grey already used for artist text (D-10 precedent) |
| Primary text | `Color.white` | Time, weather temperature, Today/Tomorrow label, calendar event title |
| Weather icon color | **`.symbolRenderingMode(.multicolor)`** — Apple's built-in per-symbol palette (yellow sun, grey/white cloud, blue raindrops/snowflakes) | **Deliberately NOT** the app's `\.activityAccent` environment accent. Weather represents literal external data (like the calendar dot below), not app-brand accent — realistic color-coding (blue=rain, yellow=sun) is more legible here than a single monochrome accent hue. This mirrors the existing precedent of NOT accent-tinting every element (D-10: accent is reserved for a short, explicit list). |
| Calendar color dot | The event's own `EKCalendar.color` (converted to SwiftUI `Color`) | Mirrors the reference image's colored dot and gives the user the same at-a-glance calendar identity they already know from Calendar.app — **not** the app's `activityAccent` either, for the same reasoning as the weather icon |
| Accent (`\.activityAccent`, reserved) | unchanged | Reserved for: equalizer bars, charging bolt, connected-device icon, progress bar fill (all pre-existing, from Phases 4/6/7). **This phase adds zero new elements to the accent-reserved list.** |
| Destructive | not applicable | This phase has zero destructive actions (display-only glance, no delete/reset/confirm flows) |

Accent reserved for: equalizer bars, charging bolt, connected-device icon, progress bar fill —
**unchanged by this phase**. Never applied to: weather icon (uses multicolor), calendar dot
(uses the calendar's own color), time/date/temperature/event text (white or secondary grey).

## Copywriting Contract

This phase is 100% display-only — there is no CTA, no destructive action, and (per D-01/D-03)
no user-facing error state is ever shown for weather/calendar failures.

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — the existing whole-blob `.onTapGesture { onClick() }` (unchanged) toggles collapse/expand; no new button/CTA is introduced |
| Weather empty state | **Column is not rendered at all** (no icon, no placeholder dash, no "--°") when location permission is denied or the WeatherKit fetch fails (D-01: silent omission). The `HStack`'s weather slot is conditionally omitted via `if let weather = outfit.weather`, mirroring the `if let calendarGlance` pattern already used for the media artwork's optional branches |
| Calendar empty state | **Column is not rendered at all** (no "No events today" text, no placeholder) when Calendar access is denied (D-03) OR when neither today nor tomorrow has an event (D-04's "nothing" case) — both converge to the same `nil` UI outcome, distinguished only in the state model for logging, never in rendering (per `14-RESEARCH.md` Anti-Patterns) |
| Today/Tomorrow label copy | Exactly **"Today"** / **"Tomorrow"** (English, verbatim per D-07's own wording) — not translated, not abbreviated |
| Error state | Not applicable — weather/calendar failures are silent by design (D-01/D-03); no error banner, no retry UI, no link to System Settings is ever shown for this phase |
| Destructive confirmation | Not applicable — no destructive actions in this phase |

## Interaction Contract (native-app addition, not in web template)

- **Whole-glance tap-to-collapse, unchanged:** the existing single `.onTapGesture { onClick() }`
  on `expandedIsland` stays scoped to the entire blob (no per-column gesture splitting needed —
  unlike `mediaExpanded`, this glance has zero interactive buttons, so there's no ambiguous
  gesture-priority concern to design around).
- **Icon-only animation (D-05):** only the weather icon's `symbolEffect` animates. Time, date,
  temperature, and calendar text are fully static — no animation, no `TimelineView` clock
  attached to any text element in this phase.
- **Weather icon animation, per category (D-06), exactly 4 mappings:**

  | Category | SF Symbol | `symbolEffect` |
  |----------|-----------|----------------|
  | Sunny | `sun.max.fill` | `.pulse, options: .repeating, isActive: true` |
  | Cloudy | `cloud.fill` | `.variableColor.iterative, options: .repeating, isActive: true` |
  | Rain | `cloud.rain.fill` | `.variableColor.iterative, options: .repeating, isActive: true` |
  | Snow | `cloud.snow.fill` | `.variableColor.iterative, options: .repeating, isActive: true` |

  `options: .repeating` is **required** and must be explicit — the default `symbolEffect`
  behavior is one-shot per value change, not continuous (`14-RESEARCH.md` Pitfall 2).
- **Idle-CPU discipline (D-04/Pitfall 5 precedent):** the weather icon's animation clock is
  gated **by construction** — it only exists while `presentation == .expandedIdle` (one case
  of the `switch presentation`, exactly like every other case in this file). No explicit
  `isPlaying`-style boolean gate is needed since SwiftUI deallocates the view (and its
  `symbolEffect` driver) the instant the presentation switches away. Verify this empirically
  on-device via `sample`/Energy sampling before/after collapsing the island, per the project's
  established `EqualizerBars`/`ProgressBar` precedent (this is inherited on-device UAT, not a
  new automated test).
- **Untrusted input bound (V5, `14-RESEARCH.md` Security Domain):** the calendar event title
  is external, untrusted data (same threat class as the existing Bluetooth device-name
  precedent) — the 100pt fixed column width + `.lineLimit(1)` + `.truncationMode(.tail)` is
  mandatory on the event-title `Text`, never removed or widened without re-checking this
  contract.
- **No cross-fade:** the 3-column content sits inside the SAME `expandedIsland` blob shape
  that already carries `matchedGeometryEffect(id: "island", in: ns)` — this phase does not
  touch the shape/morph mechanics, only the `.overlay` content.
