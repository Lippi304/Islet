## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| project.yml (committed to git) -> local Xcode signing | The real Team ID becomes a plaintext value in the repo |
| Apple Developer portal -> App ID capability record | A one-time server-side registration that any future build against this App ID relies on |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-03 | Information Disclosure | `DEVELOPMENT_TEAM` value committed in project.yml | accept | A Team ID is not a secret — it is a public identifier visible in every signed binary's code signature and already implicitly disclosed by the committed `DEVELOPER_ID` certificate string in `scripts/release.sh` (T-13-01's same accepted disposition); no credential (password, private key, API token) is written to any file |
| T-14-04 | Elevation of Privilege | `com.apple.developer.weatherkit` entitlement | accept | Grants only read-access to Apple's own first-party WeatherKit service under the existing paid Developer Program agreement — no new third-party data-sharing surface, same trust boundary as any other Apple framework already in use |

No package-manager installs in this plan (project.yml/entitlements/Info.plist config only) —
the Package Legitimacy Gate does not apply.
</threat_model>
