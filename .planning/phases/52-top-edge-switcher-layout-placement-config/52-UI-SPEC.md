## Top-Edge Layout Geometry Contract (SWITCH-03)

**Existing constants to reuse (do not invent new ones):**

| Constant | Value | Source |
|----------|-------|--------|
| `cameraClearance` | 42pt | `NotchPillView.swift:609` — the top band every full-view presentation already reserves via `.padding(.top, Self.cameraClearance)` |
| `navCircleDiameter` | 36pt | `NotchPillView.swift:1895` |
| `switcherRowHeight` | 44pt | `NotchPillView.swift:599` — the pill row's own reserved height, removed entirely in top-edge mode per D-06 |
| `expandedSize` | 420×144 | `NotchPillView.swift:273` |

**Flag for planner/executor — numeric fit risk (ties to D-04's "verify on-device" instruction):** the top-edge icons sit inside the existing 42pt `cameraClearance` band (not a new reserved row — D-06 says content area height stays the same, only the pill row's height is removed). A 36pt-diameter `navCircleButton` inside a 42pt band leaves only ~3pt of vertical margin. This is tight. Do not silently shrink `navCircleButton`'s diameter to make it fit — if it doesn't fit on-device, that is a plan-time/UAT finding per D-04, not a pre-decided fallback (do not invent a smaller variant without a discussion round).

**Horizontal placement:** 2 icons flank the camera/notch cutout on each side, positioned relative to the real cutout width — reuse `hasNotch(safeAreaTop:auxLeftWidth:auxRightWidth:)` and the `auxLeftWidth`/`auxRightWidth` values already computed in `NotchGeometry.swift`/`NSScreen+Notch.swift` (currently only consumed by `NotchWindowController` for window positioning — this phase needs a path to reach `NotchPillView`'s SwiftUI content, per 52-CONTEXT.md's Integration Points).

**Established pattern to reuse, not rediscover (Phase 39 lesson, project memory):** for any content placed with precision near the physical camera notch, use a plain `HStack(spacing: 0)` with concrete fixed-width `Color.clear.frame(width:)` spacer elements for the excluded/blocked center region (the cutout itself) — **not** `.offset(x:y:)` or `.position(x:y:)`, both of which were empirically confirmed broken inside this codebase's shape/content stack (`GeometryReader` misreports origin/size under those primitives). The top-edge row should read as: `HStack(spacing: 0) { [2 navCircleButtons, spacing 8] ; Color.clear.frame(width: auxLeftWidth + auxRightWidth-derived cutout gap) ; [2 navCircleButtons, spacing 8] }`.

Icon spacing within each 2-icon cluster: 8pt (`switcherRow`'s existing `HStack(spacing: 8)` value, reused).

## Placement Reassignment Model (D-01/D-02/D-03)

- 4 independent slots: **Left Outer, Left Inner, Right Inner, Right Outer** (left-to-right order across the flared top edge).
- Each slot independently assignable to any of the 4 icons (Home/Tray/Calendar/Weather) — full permutation, not a fixed-pair swap.
- Reassignment applies to BOTH layouts (D-03): the same slot→icon mapping reorders `switcherRow`'s pill left-to-right order AND the top-edge row's 4 positions from one shared state, not two independent configs.
- Duplicate assignment (same icon in 2+ slots) is an open implementation question — not addressed by CONTEXT.md. Recommend: Picker's `selection` per slot is independent `@AppStorage`, so duplicates are technically possible; if the planner wants a hard constraint (each icon exactly once), that's an explicit call to raise, not assumed here.

## Settings "Switcher" Section (D-07)

New `SidebarSection` case (`.switcher`), following the exact `Form`-per-section pattern Phase 51 established (`SettingsView.swift:318-331` is the closest precedent — a single `Section` with one `Picker`).

```swift
private var switcherSection: some View {
    ScrollView(.vertical) {
        Form {
            Section("Layout") {
                Picker("Layout", selection: $switcherLayout) {
                    Text("Pill").tag(SwitcherLayout.pill)
                    Text("Top Edge").tag(SwitcherLayout.topEdge)
                }
                .pickerStyle(.segmented)
                .labelsHidden()
            }

            Section("Icon Placement") {
                Picker("Left Outer", selection: $slotLeftOuter) { slotOptions }
                Picker("Left Inner", selection: $slotLeftInner) { slotOptions }
                Picker("Right Inner", selection: $slotRightInner) { slotOptions }
                Picker("Right Outer", selection: $slotRightOuter) { slotOptions }
            }
        }
        .padding(20)
    }
}
```

**Layout toggle style:** `Picker(.segmented)` with 2 options ("Pill" / "Top Edge") — mirrors the exact precedent `materialStyle`/`weatherStyle` already establish for a small enum choice in this same file (`SettingsView.swift:322-326`, `:483-489`). Do not build a plain `Toggle` — segmented Picker is this codebase's established convention for style enums.

**4 slot pickers style:** `Picker(.menu)` (native macOS dropdown, per D-02's literal "4 dropdown pickers" instruction — segmented would not read as a "dropdown"). Each offers all 4 icons as `Label(name, systemImage: icon)` rows so the icon itself is visible in the closed/open dropdown state, not just a plain text name:

```swift
Label("Home", systemImage: "house.fill").tag(SelectedView.home)
Label("Tray", systemImage: "tray.fill").tag(SelectedView.tray)
Label("Calendar", systemImage: "calendar").tag(SelectedView.calendar)
Label("Weather", systemImage: "cloud.sun.fill").tag(SelectedView.weather)
```

**Section padding/pattern:** `ScrollView(.vertical) { Form { ... }.padding(20) }` — identical to every other Phase-51-restructured section (`weatherSection`, `diagnosticsSection`), including the mandatory `ScrollView` wrapper (this codebase's own scroll-fix convention, SETTINGS-02).

## No-Notch Fallback (D-08)

- On a display without a physical camera notch, the **entire "Switcher" sidebar section is hidden** (simplest interpretation of D-08's "not reachable"; the alternative — hiding just the top-edge toggle/dropdowns within a still-visible section — is explicitly left as Claude's discretion in CONTEXT.md, but hiding the whole section is recommended here since a Placement-only section with no Layout toggle would be confusing on a non-notch display, and this phase's only Switcher-relevant control set is entirely notch-dependent).
- Gate condition: reuse `hasNotch(safeAreaTop:auxLeftWidth:auxRightWidth:)` verbatim (`NotchGeometry.swift:12`) — no new detection mechanism, no new heuristic.
- `SidebarSection.allCases` (or the `ForEach` that renders sidebar rows) needs a filter step gated on this signal — implementation detail for planning (this file currently has no notch-signal path reaching `SettingsView`; needs a new plumbing seam, flagged in 52-CONTEXT.md's Integration Points).

## Spacing Scale

This codebase's existing convention (verified across `NotchPillView.swift`/`SettingsView.swift`) does not follow a strict 4pt/8pt grid — it uses native SwiftUI/HIG defaults plus a few explicit values, all already established. This phase introduces zero new spacing tokens.

| Token | Value | Usage |
|-------|-------|-------|
| Icon-to-icon spacing (within a 2-icon cluster) | 8pt | Reused from `switcherRow`'s `HStack(spacing: 8)` |
| Section content padding | 20px | `.padding(20)` on the new `switcherSection` `Form`, matching every other Phase-51 section |
| Camera clearance band | 42pt | `cameraClearance` — the vertical band the top-edge row must fit inside |

Exceptions: the 36pt `navCircleButton` diameter inside the 42pt `cameraClearance` band is a known-tight fit — see Top-Edge Layout Geometry Contract above. Not a new exception to the scale itself, just a flagged risk.
