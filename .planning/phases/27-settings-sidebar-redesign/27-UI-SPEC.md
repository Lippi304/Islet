# Phase 27 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** Unlike Phase 25 (shell chrome only) and Phase 26 (new island-hosted carousel), this phase restructures the **Settings window** — a standard, focusable native macOS window (`Form`/`TabView` today), not the island chrome (`NotchPillView`/`blobShape()`). It uses ordinary macOS system typography, materials, and controls — the notch chrome's custom `.system(design: .rounded)` font and gradient fill do **not** apply here. Every existing control (License block, Launch-at-login, Diagnostics, Version, 4 activity toggles, Fullscreen toggle, Accent picker) is relocated verbatim into its new sidebar section (`27-CONTEXT.md` D-01/D-02/D-04) — this spec does not re-style them. This spec defines only the NEW surface: the `NavigationSplitView` sidebar shell, the Workspace placeholder, and the Theming section's 2 new controls (material-style picker, 3 per-element accent pickers).

## Layout Contract (replaces "Spacing Scale" — this phase's actual new surface)

Settings is a standard `Form`-based preferences window, not the pixel-precise island chrome — its layout follows native `NavigationSplitView`/`Form` conventions, not a custom 4px spacing scale. Values below are the concrete numbers for the one new layout shape this phase introduces (the sidebar shell); existing per-row `Form`/`Section` spacing inside each section is unchanged native default.

| Region | Value | Source |
|--------|-------|--------|
| Window frame (starting size, resolves RESEARCH.md Pitfall 4) | `.frame(width: 520, height: 380)` on the outer `NavigationSplitView` — wider/taller than today's `360×280` to fit a sidebar column | RESEARCH.md Pitfall 4 estimate; flagged for on-device tuning per this project's established sizing-iteration convention (Phases 18/20/21/23/25/26 all tuned dimensions post-implementation) — **starting point, not final** |
| Sidebar column width | `.navigationSplitViewColumnWidth(min: 160, ideal: 180, max: 220)` | Matches native macOS System Settings sidebar proportions; multiples of 4 |
| Detail-pane content padding | `.padding(20)` | Reused verbatim from today's existing `SettingsView` `.padding(20)` — unchanged |
| Sidebar row (icon + label) | Native `Label(section.title, systemImage: section.icon)` in a `List(selection:)` — no custom row view, no custom insets | Default macOS sidebar `List` styling (`.listStyle(.sidebar)`) |
| Theming section: control-to-control gap | 16px (`Section`'s default `Form` row spacing) | Native `Form` default, unchanged from today's Appearance tab |

Exceptions: none — every new number above is either a native framework default or explicitly flagged as an on-device-tuning starting point.

## Section Content Specs

### General (relocated, unchanged content)
Reuses `SettingsView.swift`'s existing `Toggle`/`Button` views and copy exactly — no new controls, no re-styling. See `27-RESEARCH.md` Recommended Project Structure for the exact line-level relocation map.

### Workspace (new placeholder, D-03)
A single centered `Text`/`Label` pair — no `Form`/`Section` chrome needed since there is nothing to configure:

| Element | Copy |
|---------|------|
| Icon | SF Symbol `tray`, 28px, `.foregroundStyle(.secondary)` |
| Heading | "Nothing to configure yet" |
| Body | "The Shelf works automatically — no settings needed right now." |

Layout: `VStack(spacing: 8) { icon; heading; body }` centered in the detail pane (`.frame(maxWidth: .infinity, maxHeight: .infinity)`), matching the plain, quiet tone of this app's other non-error empty states.

### System / Theming (new controls, D-04/D-05/D-06/D-07)

**Material style** — exactly 2 presets, segmented control (native, least code for a 2-option exclusive choice):

```swift
Section("Appearance Style") {
    Picker("Style", selection: $materialStyle) {
        Text("Gradient").tag(MaterialStyle.gradient)
        Text("Solid Black").tag(MaterialStyle.solidBlack)
    }
    .pickerStyle(.segmented)
}
```

| Property | Value |
|----------|-------|
| Control | `Picker` with `.pickerStyle(.segmented)` |
| Options | "Gradient" (default, D-05) · "Solid Black" |
| Label | "Style" (row label inside a `Section("Appearance Style")`) |

**Per-element accent** — 3 independent rows, each reusing today's existing swatch-circle picker verbatim (`SettingsView.swift` lines 107-119: `ForEach(palette.indices) { Circle().fill(...).overlay(selection ring).onTapGesture }`), called 3 times against 3 separate `@AppStorage` bindings — no new picker component (D-07, `27-RESEARCH.md` Don't-Hand-Roll table):

```swift
Section("Accent Colors") {
    LabeledContent("Now Playing") { swatchRow(selection: $nowPlayingAccentIndex) }
    LabeledContent("Charging")    { swatchRow(selection: $chargingAccentIndex) }
    LabeledContent("Device")     { swatchRow(selection: $deviceAccentIndex) }
}
```

| Row Label | Bound Key | Reused Component |
|-----------|-----------|-------------------|
| "Now Playing" | `nowPlayingAccentKey` | Existing swatch-circle row, unchanged visuals |
| "Charging" | `chargingAccentKey` | Existing swatch-circle row, unchanged visuals |
| "Device" | `deviceAccentKey` | Existing swatch-circle row, unchanged visuals |

### About / License (relocated, unchanged content)
Reuses the existing adaptive `switch` block (trial/trialExpired/licensed) and Version `LabeledContent` exactly as today — D-02.

## Typography

| Role | Value | Weight | Usage |
|------|-------|--------|-------|
| Sidebar row label | `.body` (system default, ~13pt) | regular (system handles selected-row emphasis automatically via `List` tint, no manual weight change) | The 4 sidebar section labels |

| Body / control row text | `.body` (system default, ~13pt) | regular | Toggle labels, picker row labels, License status text — unchanged from today |
| Placeholder heading/body (Workspace) | `.headline` (heading) / `.subheadline` (body) | semibold / regular | The 2 new Workspace placeholder lines only |

**Declared weights: regular + semibold only** (2, per contract) — matches this file's existing implicit convention (`.font(.headline)` used once today for the trial-expired heading; everything else is unweighted `.body`). No new font sizes or custom design variants introduced; the notch chrome's `.rounded` design is explicitly NOT used here (this is a standard system window, not island chrome).

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | System window/`Form` background (`.background(.windowBackground)` / default `Form` material — light or dark per system appearance) | Detail-pane content background — unchanged from today, no new color introduced |

| Accent (10%) | System accent color (automatic, `List(selection:)` row highlight) + `Color.primary` selection ring on swatch circles (existing convention, unchanged) | Reserved for: the selected sidebar row highlight (system-automatic) and the selected-swatch ring on the 4 color-picker rows (1 existing Appearance-tab picker relocated + 3 new Theming pickers) — nowhere else |
| Destructive | n/a | No destructive actions in this phase's scope (Settings has no delete/reset control) |

Accent reserved for: sidebar selection highlight (system-automatic) and swatch selection rings only — never applied to section headers, body text, or the Workspace placeholder icon.

## Verification Notes for Executor

- Sidebar icons (`gearshape`, `tray`, `paintbrush`, `info.circle`) are illustrative SF Symbol choices (RESEARCH.md Assumption A3) — cosmetic only, a one-line swap during UAT if a different symbol reads better.
- The Workspace placeholder needs no `Form`/`Section` wrapper — it is the only detail pane in this phase that is NOT a `Form` (all 3 other sections stay `Form`-based, matching today's convention).
- The material-style `Picker` must branch through `AnyShapeStyle` at the 4 `NotchPillView` fill sites, NOT at the Settings-UI layer — this spec only covers the picker's own appearance in Settings; the runtime type-erasure requirement is a `27-RESEARCH.md` Pattern 2 / Pitfall 2 implementation concern, not a UI design decision.
- The 3 accent-picker rows must call the exact same existing swatch-circle-drawing code 3 times (`ActivitySettings.palette` `ForEach` + selection ring) — do not build a second color-picker component; see `27-RESEARCH.md` Don't-Hand-Roll table.
- Window frame (520×380) and sidebar column width (160/180/220) are starting points — confirm on-device once all 4 sections have real content, per this project's established sizing-iteration pattern (Phases 18/20/21/23/25/26).
- Do not apply the notch chrome's `.system(design: .rounded)` font or any gradient/black fill anywhere in `SettingsView.swift` — this window is standard macOS system chrome, a deliberately distinct visual language from the island (confirmed: current `SettingsView.swift` has zero `.rounded` font usage today).
