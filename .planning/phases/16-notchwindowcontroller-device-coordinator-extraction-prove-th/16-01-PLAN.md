<objective>
Extract the 9 device-splash bookkeeping fields and 3 stateful methods (`handleDevice`,
`triggerDeviceBatteryRefreshIfPromoted`, `scheduleDeviceBatteryRefresh`) currently living
directly on `NotchWindowController` into a new, independently-testable `DeviceCoordinator`
class, behind a new narrow `ActivityCoordinator` protocol (D-02). This plan creates the new
types and their test surface in isolation — it does NOT touch `NotchWindowController.swift`
yet (that is Plan 16-02, which wires the controller to use these new types and deletes the
old inline fields/methods).

Purpose: Prove the coordinator-extraction shape on the single highest-risk, most
race-documented activity type in the app (11+ inline "gap-closure"/"Finding N" comments)
before repeating it for Charging/NowPlaying/Outfit in a future phase — with zero product
behavior change and a real, isolated unit-test surface that today's inline code has none of.
Output: `Islet/Notch/ActivityCoordinator.swift`, `Islet/Notch/DeviceCoordinator.swift`,
`IsletTests/DeviceCoordinatorTests.swift`.
</objective>

- `init(queueHead: @escaping () -> ActiveTransient?, enqueue: @escaping (ActiveTransient) -> Bool, updateHead: @escaping (ActiveTransient) -> Void, presentTransientChange: @escaping () -> Void, renderPresentation: @escaping () -> Void, batteryForAddress: @escaping (String) -> Int?)`

- `func clearPendingBatteryPolls()` — replaces the controller's old inline `pendingDeviceBatteryPolls.removeAll()` in `flushTransients(.device)`
- `nonisolated func cancelPendingWork()` — replaces the controller's old inline `deviceBatteryWork?.cancel()` in `deinit`

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-16-01 (carries T-05-01) | Tampering | `DeviceCoordinator.handle(_:)` → `deviceLabel(name:address:)` | mitigate (already mitigated, do not regress) | This extraction moves STATEFUL orchestration only — `deviceLabel`'s plain-String-only contract is untouched. Task 2's acceptance criteria require `DeviceCoordinator.swift` to call `deviceActivity(from:)`/`deviceLabel` rather than reimplement any name handling, so no new formatting/logging call is introduced that could interpolate `reading.name` unsafely. |
| T-16-02 (carries T-06-09) | Denial of Service | `DeviceCoordinator`'s debounce dictionaries + `TransientQueue.maxDepth` (reached via the `enqueue` closure) | mitigate (already mitigated, do not regress) | Pitfalls 1, 4, and 7's regression tests directly assert the debounce gate and the pending-poll cap-at-2 still hold after extraction — a flapping/malicious device cannot flood the queue. |
| T-16-03 (carries T-06-12) | — (resource exhaustion / OS-token-adjacent leak) | `DeviceCoordinator.deviceBatteryWork` teardown | mitigate | `cancelPendingWork()` is `nonisolated`, backed by `nonisolated(unsafe)` storage, callable synchronously from `NotchWindowController`'s existing `nonisolated deinit` (wired in Plan 16-02) — mirrors `BluetoothMonitor.stop()`'s already-verified pattern exactly, so the one-shot `DispatchWorkItem` cannot outlive its owner. |
