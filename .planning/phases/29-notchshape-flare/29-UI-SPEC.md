## Shape/Flare Contract (replaces "Spacing Scale" for this phase's actual scope)

Phase 29 introduces no new spacing values. The design contract for this phase is the **flare geometry** on `NotchShape`'s top edge, specified here instead.

### New shape parameter (SHAPE-01, source: D-01/D-02/D-05, code_context Integration Points)

Add one new plain `CGFloat` stored property to `NotchShape` (`Islet/Notch/NotchShape.swift`), following the exact same pattern as the existing `topCornerRadius`/`bottomCornerRadius` properties so it participates in the same interpolation behavior across the collapseŌåöexpand spring morph:

| Property | Default | Purpose |
|----------|---------|---------|
| `topFlareWidth` | `0` | Horizontal distance the top edge widens outward ŌĆö beyond the shape's own left/right walls ŌĆö before curving down into the existing `topCornerRadius` transition. `0` = today's flush/straight top edge (no flare). |

**Precedent, not replacement (Phase 25 D-09):** the existing `topCornerRadius` quad-curve already reads as a "flowing merge into the screen edge" and stays completely unchanged (still `6` everywhere it is today). `topFlareWidth` is an *additional*, independent parameter layered alongside it ŌĆö do not fold the flare into `topCornerRadius` itself or repurpose that value.

### Widen amount (D-01/D-05 REVISED 2026-07-13 during Task 3 on-device UAT ŌĆö see 29-CONTEXT.md)

The subtle-widen version below was superseded on-device: it read as imperceptible. The user provided a concrete reference (Droppy's shelf widget) confirming the flare should be a pronounced CONCAVE sweep ŌĆö narrow flat top (fixed width, matching the physical notch), then curving outward/down to the presentation's full width. This is a distinct geometry shape, not a bigger version of the old straight widen.

| Property | Value | Rationale |
|----------|-------|-----------|
| Look | Pronounced concave flare, matching Droppy reference | D-01 revised ŌĆö the original "subtle" rejection of a trumpet/bell flourish is overridden; that flourish is now the explicit target |
| Mechanism | Fixed narrow top-band width (matches the physical notch cutout), then a concave curve down to each presentation's own existing full width | D-05 revised ŌĆö the fixed constant is now the narrow top-band width, not an added margin; the flare-out distance naturally differs per presentation because it converges into each presentation's own width |
| Exact pt values / curve shape | Claude's/executor's discretion (D-02), tuned on-device against the Droppy screenshot | Try a single quad curve first (project's quad-curve-only convention, 29-PATTERNS.md); fall back to a cubic `addCurve` only if a quad curve can't produce a convincing sweep |

### Coverage ŌĆö which call sites get the flare (D-03/D-04 LOCKED)

| Site | Function | Gets flare? | Notes |
|------|----------|-------------|-------|
| Home / Tray / Calendar / Weather (expanded blob) | `blobShape()` | **Yes** | Pass `topFlareWidth: 10` (starting value) at its `NotchShape(...)` construction |
| Charging wings / Device wings | `wingsShape()` | **Yes** | Same fixed value, same construction-site edit |
| Now-Playing media wings / song-change toast | `mediaWingsOrToast(_:)` | **No (D-03, locked)** | Makes its own standalone inline `NotchShape(...)` call, not routed through `wingsShape()` ŌĆö simply omit the parameter (defaults to `0`), no special-casing needed (D-04) |
| Collapsed/idle pill | `collapsedIsland` | **No ŌĆö must stay pixel-identical** | Already calls plain `NotchShape()` with no parameters passed; leave untouched. This is Success Criterion #2 ŌĆö do not add `topFlareWidth` here even at `0`-equivalent-looking values; the call site itself must remain undiff'd |

### Animation behavior (Success Criterion #3)

`topFlareWidth` is a plain `CGFloat` stored property, added the same way `topCornerRadius`/`bottomCornerRadius` already are ŌĆö Phase 25's on-device pass confirmed plain `CGFloat` `Shape` properties morph cleanly through the existing collapseŌåöexpand spring with `matchedGeometryEffect` and needed **no** `animatableData` conformance (`25-VERIFICATION.md` item 5: zero corner-snap artifact observed). Follow the identical precedent here ŌĆö do not add `animatableData` pre-emptively.

**Watch item (carried forward from Phase 25's identical watch-item pattern):** if an on-device pass shows a visible "snap" at the flare boundary specifically (not expected, but the corner-radius delta is now larger than before), the contained fix is adding `var animatableData: AnimatablePair<CGFloat, CGFloat>` (or a 3-property `AnimatableData` triple, since there would now be three tunable geometry values) to `NotchShape` ŌĆö a small, isolated addition, not a path-mechanism rewrite. Only do this if the artifact is actually observed.

### Open geometry question ŌĆö panel-frame reservation (flag for planner/executor)

**Not resolved by this UI-SPEC ŌĆö must be checked on-device during planning/execution** (per `STATE.md` Blockers/Concerns and `29-CONTEXT.md`'s Claude's Discretion note): `NotchShape`'s path can draw points beyond its own `rect.minX`/`rect.maxX` (a `Shape`'s paint isn't clipped to its frame unless an ancestor applies `.clipped()`/`.mask()`). Confirm whether:
1. The existing `NSPanel` frame already reserves enough horizontal margin above/around the blob/wings frames for a ~10pt outward bulge to render without being clipped by the window's own bounds, **or**
2. The panel frame itself needs to grow slightly (width and/or the reservation near `screenFrame.maxY`) to accommodate the flare.

If (2), this is a `NotchWindowController`/`NotchGeometry` change, not a `NotchShape.swift` change ŌĆö scope it explicitly in the plan rather than discovering it mid-implementation. This does not change the visual contract above; it only affects where the extra pixels are legally paintable.
