## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| SwiftUI closure -> CoreAudio write (system audio-output default device, system volume) | The handlers in this plan are the ONLY place a UI-originated tap/drag turns into a real CoreAudio write in this phase. Both writes are local, single-user, no-auth, no-network operations — same trust tier as the existing OSD volume/mute self-drive path (Phase 39) already shipped in this codebase. |
| Click-through geometry (CR-01) | `positionAndShow()`/`visibleContentZone()` define which screen region is interactive vs. click-through. A drift between the two (or between either and `NotchPillView.tabHeight`) is a correctness/availability bug, not a data-exfiltration or injection risk — tracked here as a Denial-of-Service-flavored UI-availability threat per this codebase's own established framing for the same regression class. |

| T-48-07 | Tampering | handleSelectOutputDevice(_:) -> AudioOutputMonitor.setDefaultOutput | mitigate | Per D-01, `setDefaultOutput` (Phase 47) already scopes exclusively to `kAudioHardwarePropertyDefaultOutputDevice` — this plan's handler never touches `kAudioHardwarePropertyDefaultSystemOutputDevice` (system alert-sound routing), so switching the media output can never silently override a user's separately-configured alert-sound device. |
