<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| CoreAudio system HAL (local, public API) | Device UID/name/volume-control facts and the volume write path come from/go to the local machine's own audio hardware abstraction layer — not network, not user-typed input, not another process's untrusted data. Same trust tier as the existing `adjustSystemVolume()`/`readSystemVolume()` already shipped in this file. |
| System audio-output volume (write) | `setSystemVolume(_:)` mutates real system volume — a local, single-user, no-auth operation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-48-01 | Tampering | VolumeReader.setSystemVolume(_:) | mitigate | Defensively re-clamps its input to 0...1 regardless of what the caller (a future drag gesture) computed; every CoreAudio Get/Set call guarded against non-noErr, aborting the whole adjustment on any failure rather than partially applying a change — identical discipline to the already-shipped adjustSystemVolume(increase:). |
| T-48-02 | Denial of Service | startAudioOutputMonitor() called unconditionally at launch | mitigate | AudioOutputMonitor.start() is already idempotent (Phase 47, `running` guard) — the unconditional call site here cannot cause duplicate listener registration/callback storms even if invoked more than once (e.g. a future settings-change re-entry). |
| T-48-03 | Spoofing / Information Disclosure | Device UID/name strings flowing from AudioOutputMonitor into presentationState.outputDevices | accept | Values originate from the local system's own audio HAL (not network/untrusted-user input); stored as plain String/struct data only, never interpolated into a shell command, file path, or format string — same "untrusted but inert" precedent already accepted for BluetoothMonitor.device.name (Phase 6) and confirmed again for AudioOutputDevice in Phase 47's own threat model (T-47-04). |

No Repudiation / Elevation-of-Privilege entries — this plan introduces no identity/audit boundary
and requests no new system privilege beyond the app's existing (unsandboxed) entitlement set.
</threat_model>
