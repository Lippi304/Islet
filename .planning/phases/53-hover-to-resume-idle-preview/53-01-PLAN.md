<objective>
Verify the phase's one open technical question on-device, then ship the hover-to-resume idle
preview: hovering the collapsed island when nothing is playing (but something played this
session) previews the last track exactly like the live Now Playing glance, and clicking it
attempts to resume playback via the existing `togglePlayPause()` transport call, with clear
"Wiedergabe nicht möglich" feedback when the transport genuinely can't resume a given session.

Purpose: RESUME-01/RESUME-02 (v1.8) — the milestone's one open technical question (does
`togglePlayPause()` resume a fully-stopped/quit session?) must be answered empirically before
the D-03 failure-feedback path can be trusted to fire only when it should, per
53-CONTEXT.md/ROADMAP SC#4 and PROJECT.md's v1.8 Key Context.

Output: A working hover-preview + resume-tap feature (Islet/Notch/NowPlayingState.swift,
NotchPillView.swift, NotchWindowController.swift) plus a recorded 4-combination spike verdict
(Spotify paused/quit x Apple Music paused/quit) informing how routinely the failure path will
actually be exercised in real use.

### Architecture decision (Claude's Discretion, 53-CONTEXT.md)
The hover-preview is a VIEW-LOCAL branch off the existing `.idle` `IslandPresentation` case
(NotchPillView.swift), NOT a new resolver case. Rationale: `resolve(...)` (IslandResolver.swift)
has exactly one call site (NotchWindowController.swift `currentPresentation()`); threading a new
hover-flag parameter through it plus a new `IslandResolverTests.swift` coverage set is a larger
diff for a purely presentational hover affordance that 53-CONTEXT.md explicitly leaves to
discretion. The hover signal itself already exists as `interaction.isHovering`
(NotchInteractionState.swift, driven by the controller's existing global `.mouseMoved` monitor
against `collapsedInteractiveZone()`) — no new `@State` hover flag is needed either. This keeps
IslandResolver.swift and IslandResolverTests.swift untouched for this phase.
