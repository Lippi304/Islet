    `security find-identity -v -p codesigning` and copy the FULL identity string exactly as

    When prompted for a password, generate and use an app-specific password from
    https://appleid.apple.com > Sign-In & Security > App-Specific Passwords — NOT the normal Apple
    ID password. "islet-notary" is the profile NAME (`NOTARY_PROFILE`) already assumed by
    docs/RELEASE.md's example and scripts/release.sh's comment — reuse that exact name unless
    there is a specific reason to change it. Nothing from this step (Apple ID, team ID,
    app-specific password) is ever written to a file in this repo — only the certificate ends up
    in the keychain, and only the profile NAME is later referenced from the script.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Local machine keychain -> Apple notary service | The signed `.dmg` binary is uploaded over HTTPS to Apple's notary API for automated malware scanning |
| scripts/release.sh (repo, committed) -> local shell execution | The script embeds `DEVELOPER_ID`/`NOTARY_PROFILE` identifiers and runs with full local codesign/keychain privileges when invoked |
| Quarantine-tagged `.dmg` -> Gatekeeper/spctl | The exact untrusted-download boundary being simulated and tested (D-01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Information Disclosure | scripts/release.sh (committed DEVELOPER_ID/NOTARY_PROFILE strings) | accept | Per CONTEXT.md D-03: `DEVELOPER_ID` is a public certificate identifier and `NOTARY_PROFILE` is a local keychain lookup key, not a secret — the actual app-specific password never leaves the local keychain and is never written to any file in this repo (Task 1 explicitly keeps it out of scope for committed files). |
| T-13-02 | Tampering | scripts/release.sh supply chain | accept | Single-developer repo with no external contributors or unattended CI running this script; it is manually invoked and its diff is visible before each release run. |
| T-13-03 | Repudiation / Spoofing | Gatekeeper bypass via silently-skipped notarization | mitigate | Task 2's acceptance_criteria explicitly assert stdout does NOT contain "SKIPPING notarize + staple" and that `codesign` shows the real Developer ID identity (not ad-hoc) before the dmg is treated as shippable. |
| T-13-04 | Elevation of Privilege | `com.apple.security.cs.disable-library-validation` entitlement (Islet/Islet.entitlements, pre-existing) | accept | Already dispositioned in Phase 6 / quick-task 260705-mzj as required for the embedded `MediaRemoteAdapter.framework` under Hardened Runtime; unchanged by this phase — Release signing here uses the same entitlements file, no new attack surface introduced. |
| T-13-05 | Denial of Service | Apple notary service outage/rejection mid-release | accept | `set -euo pipefail` in release.sh aborts cleanly on any notarytool/stapler failure with no partial "half-notarized" artifact shipped; the script is idempotent (`rm -rf build dist` at the top), so a re-run after a transient Apple-side failure is safe. |

No package-manager installs occur in this phase (shell script + credential fill only) — the
Package Legitimacy Gate does not apply.
