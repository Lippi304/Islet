## Material Contract (replaces "Spacing Scale" for this phase's actual scope)

Phase 25 introduces no new spacing values — `blobShape`/`wingsShape`/collapsed pill frame sizes and paddings are all pre-existing and untouched. The design contract for this phase is the **gradient material** and **shape geometry**, specified here instead.

### Gradient material (VISUAL-01, source: D-01/D-02, RESEARCH.md Pattern 1)

One shared `LinearGradient`, used verbatim at all 4 fill sites (`collapsedIsland`, `blobShape`, `wingsShape`, `mediaWingsOrToast`) — replacing every `.fill(Color.black)` / `.fill(collapsedFill)` black branch:

| Property | Value |
|----------|-------|
| Stop 1 | `color: .black, location: 0.0` |
| Stop 2 | `color: .black, location: 0.65` (long solid stretch — D-02; exact stop is Claude's discretion, tune on-device, keep in the 0.6–0.7 band) |
| Stop 3 | `color: .black.opacity(0.5), location: 1.0` (D-02's ~50% floor — never `.clear`, never below ~0.4 opacity) |
| `startPoint` | `.top` |
| `endPoint` | `.bottom` |
| Base hue | Pure black only — **no grey mixed in** (D-01, locked, non-negotiable) |

Implementation notes (pre-populated from RESEARCH.md, not re-litigated here):
- Implement as `.fill(LinearGradient(...))` directly on `NotchShape`, not `.overlay().mask()` — `LinearGradient` is a first-class `ShapeStyle`.
- Use relative `UnitPoint`s (`.top`/`.bottom`), never absolute pixel offsets — this is what makes ONE shared gradient definition read correctly at all three heights (collapsed 38pt / wings 32pt / expanded 144pt) with zero per-shape special-casing.
- `collapsedFill`'s `#if DEBUG` red tint stays as the DEBUG-only override; only the `#else` branch's `Color.black` becomes the gradient.

### Corner radius (VISUAL-01/D-08/D-09)

| Element | Property | Current | New value | Source |
|---------|----------|---------|-----------|--------|
| Expanded blob (`expandedIsland`/`mediaExpanded`/`mediaUnavailable`) | `bottomCornerRadius` | 20 | **32** (starting point — "significantly rounder," Claude's discretion per D-08, tune on-device) | D-08 |
| Expanded blob | `topCornerRadius` | 6 | 6, unchanged (D-09 confirms already correct) | D-09 |
| Wings (`wingsShape`, `mediaWingsOrToast` no-toast state) | `topCornerRadius`/`bottomCornerRadius` | 6/6 | 6/6, unchanged | D-09 |
| Collapsed pill | n/a — uses default `NotchShape()` init (6/14) | 6/14 | unchanged | D-10 (position/shape already correct) |

**Watch item (RESEARCH.md Pitfall 1):** `NotchShape` has no `animatableData` conformance today — the corner radius does not tween through `withAnimation`, only `.frame()` size does (via `matchedGeometryEffect`). Raising the expanded bottom radius from 20→32 widens the radius delta vs. wings (6), which may make a pre-existing "snap" newly visible. On-device-verify during D-08 tuning; if visible, add `var animatableData: AnimatablePair<CGFloat, CGFloat>` to `NotchShape` (a contained 3-line addition, not a shape-mechanism change).

### Spring animation (VISUAL-02, source: D-05/D-06/D-07, RESEARCH.md Pattern 2)

Both constants live in `NotchWindowController.swift` lines 264–265; all 13 `withAnimation(.spring(response: springResponse, dampingFraction: springDamping))` call sites read them — retune only these two declarations, touch no call site.

| Constant | Current | New starting value | Direction / rationale |
|----------|---------|---------------------|------------------------|
| `springResponse` | 0.35 | **0.6** | Slower — D-05 "deliberately slow... even on 60Hz." Starting point for on-device iteration (D-07), not final. |
| `springDamping` | 0.65 | **0.62** | Slightly lower than today = one visible overshoot-and-settle bounce (D-06). Stay in the 0.55–0.7 band — below ~0.5 risks a second, disallowed bounce (Success Criterion #2, RESEARCH.md Pitfall 3). |

Applies identically to both the hover-widen transition and the full click-to-expand transition (D-06: no asymmetry requested).

**Watch item (RESEARCH.md Pitfall 4, Open Question 1):** `graceDelay` (0.4s hover-exit-to-collapse timer) is a separate, untouched constant. A materially slower spring could visually overlap with a fast hover-exit/re-enter cycle. Out of this phase's scope to fix — but test rapid hover-enter/exit/re-enter specifically during on-device tuning and flag as a follow-up (not a silent fix) if a double-morph artifact appears.
