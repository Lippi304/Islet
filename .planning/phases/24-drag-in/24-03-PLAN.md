<objective>
Close the drop-interception architecture gap that paused Plan 24-02's Task 3: because `NotchPanel` is deliberately click-through and never a registered `NSDraggingDestination`, a real drop currently falls through to whatever is visually underneath (Finder's Desktop), which performs its own default same-volume MOVE — relocating the user's original file. This plan implements CONTEXT.md's D-10 through D-15 fix: a new `CGEventTap` (`DropInterceptTap`) that swallows the terminating `.leftMouseUp` for drags landing in the already-armed accept region, stopping Finder's Desktop from ever seeing the drop complete, while still landing the file in the shelf via a directly-invoked callback (not a second, possibly-starved `NSEvent` consumer).

Per D-13, this is capped at 2 on-device validation rounds, mirroring D-05/D-06's discipline for the (already-proven) `DragApproachDetector`. This plan is therefore spike-first: Tasks 1-2 build and on-device-validate ONLY the core premise (does swallowing `.leftMouseUp` at `.cgSessionEventTap`/`.defaultTap` actually stop the WindowServer's drag-completion bookkeeping — RESEARCH.md's Assumption A5, explicitly unconfirmed by any documentation or real-world precedent) with the smallest possible throwaway instrumentation, mirroring Plan 24-01's own successful spike-first structure. Tasks 3-4 build the full `DropInterceptTap` type and re-run on-device UAT ONLY if the spike passes. If the spike fails after the D-13 cap, this plan stops at Task 2 — D-14's move-back mitigation is noted as the fallback trigger condition only; its own implementation is explicitly deferred (CONTEXT.md's own instruction) to a follow-up plan/discussion, not designed here.

Upon this plan's Task 4 checkpoint returning "approved," Plan 24-02's Task 3 checkpoint (paused, not approved, not skipped) is RESOLVED/superseded — Task 4's on-device checklist re-covers every item from 24-02's original Task 3 checklist plus the new interception-specific checks, so 24-02's Task 3 does not need to be separately re-run. SHELF-01/SHELF-02 are then complete and REQUIREMENTS.md should be updated to reflect this.

@MainActor
final class BluetoothMonitor: NSObject {
    private nonisolated(unsafe) var connectToken: IOBluetoothUserNotification?
    private nonisolated(unsafe) var running = false
    private let onReading: (DeviceReading) -> Void

    init(onReading: @escaping (DeviceReading) -> Void) { self.onReading = onReading; super.init() }

    func start() {
        guard !running else { return }   // idempotent
        running = true
        connectToken = IOBluetoothDevice.register(forConnectNotifications: self, selector: #selector(connected(_:device:)))
    }

    nonisolated func stop() {
        connectToken?.unregister()
        connectToken = nil
        running = false
    }
}
