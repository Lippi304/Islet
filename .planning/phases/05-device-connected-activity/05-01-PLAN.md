<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| remote BT device → app | `device.name` is an attacker-controllable string supplied by a remote/paired device; it crosses into the app at the connect/disconnect callback and is rendered in the island UI |
| macOS TCC → app | The OS gates Bluetooth access; an over-broad request produces an intrusive prompt (Success Criterion 3) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering / DoS | `deviceLabel(name:address:)` in DeviceActivity.swift | mitigate | Treat `device.name` as untrusted: `deviceLabel` returns it as a plain `String` only — NEVER interpolated into a format string or shell. Length/line bounding is applied at the SwiftUI `Text` in Plan 03 (`.lineLimit(1)` + `.truncationMode(.tail)`), mirroring T-04-09 now-playing metadata handling. SwiftUI `Text` is inert to format specifiers. |
| T-05-04 | Information disclosure / privacy (intrusive prompt) | IOBluetooth registration (spike) | mitigate | Spike registers ONLY connect/disconnect notifications — NO `pairedDevices()`, NO scanning, NO connection opening (the prompt-triggering calls per RESEARCH Pitfall 6). Adds the usage-description key ONLY if the spike proves it is required, keeping any prompt one-time + expected (Criterion 3). |
| T-05-06 | Tampering / crash | `shouldShowDeviceSplash` predicate | accept | Pure deterministic function of its arguments (no clock, no shared mutable state); a malformed/empty address is handled by the fallback label and cannot crash. Low risk, no further mitigation. |
</threat_model>
