22-02's pure seams: NotchPanel.swift permanently registers for .fileURL drag types and forwards its
four NSDraggingDestination callbacks through thin closures (never embedding hover/expand/shelf

file already uses (CR-01 discipline).

Deliberate simplification vs. Phase 21's outbound isDraggingShelfItem pin: no safety-net timer or
extra release monitor is added here. Unlike a SwiftUI .onDrag's completion (which Phase 21 could not
observe reliably from AppKit alone, hence its 20s safety net + .leftMouseUp monitor), an AppKit
NSDraggingDestination registration is GUARANTEED by AppKit to call exactly one of draggingExited or
performDragOperation -> draggingEnded to terminate every drag session -- both are wired to the same
idempotent endFileDrag(), which is sufficient on its own.

Confirmed via this session's SDK header read (AppKit.framework/Headers/NSDragging.h):
draggingExited has signature "- (void)draggingExited:(nullable id <NSDraggingInfo>)sender" -- the
ONLY one of the four NSDraggingDestination methods whose sender parameter is optional. This is why
Task 2's NotchPanelTests addition can call panel.draggingExited(nil) directly with a real nil
literal -- no mock/fake NSDraggingInfo is needed or possible for the other three (draggingEntered,
draggingUpdated, performDragOperation all require a non-optional id<NSDraggingInfo>, which has no
public Swift initializer). Also confirmed: NSWindow has NO public getter for its registered dragged
types (only the write-only registerForDraggedTypes(_:) method) -- do not attempt to assert
registration state via a "registeredDraggedTypes" property, it does not exist on NSWindow (NSView
has one, NSWindow does not).
