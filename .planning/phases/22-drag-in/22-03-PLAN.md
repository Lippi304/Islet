Confirmed via this session's SDK header read (AppKit.framework/Headers/NSDragging.h):
draggingExited has signature "- (void)draggingExited:(nullable id <NSDraggingInfo>)sender" -- the
ONLY one of the four NSDraggingDestination methods whose sender parameter is optional. This is why
Task 2's NotchPanelTests addition can call panel.draggingExited(nil) directly with a real nil
literal -- no mock/fake NSDraggingInfo is needed or possible for the other three (draggingEntered,
draggingUpdated, performDragOperation all require a non-optional id<NSDraggingInfo>, which has no
public Swift initializer). Also confirmed: NSWindow has NO public getter for its registered dragged
types (only the write-only registerForDraggedTypes(_:) method) -- do not attempt to assert
registration state via a "registeredDraggedTypes" property, it does not exist on NSWindow (NSView
has one, NSWindow does not).

Coordinate space of NSDraggingInfo.draggingLocation (developer.apple.com/documentation/appkit/
nsdragginginfo/dragginglocation): expressed in the BASE COORDINATE SYSTEM OF THE DESTINATION WINDOW
-- unlike NSEvent.mouseLocation, which handlePointer's own Pitfall-6 comment notes is ALREADY global
screen coordinates ("No coordinate conversion -- both point and hotZone are global bottom-left").
draggingLocation MUST therefore be converted to global screen coordinates before it can be compared
against hotZone with CGRect.contains(). NotchPanel is borderless (no titlebar/chrome, styleMask is
[.borderless, .nonactivatingPanel]), but the conversion should still go through the documented API
rather than assume frame == contentRect math: `panel.convertToScreen(NSRect(origin: draggingLocation,
size: .zero)).origin` is NSWindow's own supported way to convert a window-base point to a global
screen point (NSWindow.convertToScreen(_:) takes/returns an NSRect; a zero-size rect round-trips a
single point cleanly).

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Finder/other app (drag source) -> NSPasteboard -> NotchPanel's registered drag destination | Inbound: the first time an external, untrusted file/folder URL crosses INTO the app via drag (contrast Phase 21, outbound-only). |
| interaction.isExpanded state read at both draggingEntered and performDragOperation time | Internal consistency boundary between the accept decision and the actual append -- state could theoretically change between the two AppKit callbacks. |
| Drag location (NSDraggingInfo.draggingLocation) read at both draggingEntered and performDragOperation time | Because draggingUpdated is deliberately not overridden, the pointer can drift anywhere within the registered panelFrame between the two callbacks while AppKit keeps delivering the same granted operation -- the hotZone check must be re-derived at both points, not cached. |
