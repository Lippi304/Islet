pill `hotZone` (the original, now-superseded D-02). 22-01's on-device spike confirmed AppKit drag
delivery reaches the click-through panel (A1), but found the drop never completed because the drag
path crosses macOS's own top-edge Mission Control trigger before reaching that tiny hot zone

Mission-Control Fallback" revision:

- **D-02b**: the drag-accept region reuses `expandedZone` -- the SAME always-reserved
  expanded+wings footprint (`panelFrame.insetBy(dx: -hotZonePadding, dy: -hotZonePadding)`,
  `positionAndShow` ~line 647) already computed for the keep-open grace decision while expanded.
  No new geometry is invented; this plan's ONLY new geometry is the D-02c margin below.

  screen edge (`NotchGeometry.swift`'s `topPinnedFrame`, ~line 62 -- there is no screen real estate
  above the physical notch), the accept decision additionally requires the drag location to sit at
  or below `dragLandingMaxY`, a NEW property computed in `positionAndShow` alongside `hotZone`/
  `expandedZone`: `collapsedFrame.maxY - (collapsedFrame.height + hotZonePadding)` -- i.e. one full
  hot-zone-height of clearance below the physical top edge (derived directly from already-measured
  geometry, no arbitrary magic number). This gives the user room to complete the drop without
  dwelling in the thin top band where macOS's own Mission-Control-during-drag trigger lives.
- **D-05/D-06**: D-01's auto-expand-on-drag-enter and D-03's hot-feedback bounce now BOTH trigger
  off this same wider `isWithinDragAcceptRegion(_:)` gate (via `draggingEntered`), not the old tiny
  `hotZone` -- one signal drives both, exactly as before, just gated on the new, wider region.
- **D-07**: ordinary (non-drag) hover/click interactivity is COMPLETELY UNCHANGED -- `hotZone`,
  `pointerInZone`, `handlePointer`, and `syncClickThrough()` are read/written EXACTLY as they are
  today; the widening applies ONLY to the new drag-specific accept/auto-expand decision.
- **Architecture risk (CR-01 discipline, project memory `cr01-clickthrough-or-defeat-gotcha`)**:
  the new `isDraggingFileIn` pin is read ONLY inside `handleHoverExit`'s existing `graceWorkItem`
  guard (gating the collapse TIMER, mirroring `isDraggingShelfItem` exactly) -- it is NEVER read or
  written inside `syncClickThrough()`, and `syncClickThrough()`'s expanded branch stays a pure
  `visibleContentZone()?.contains(lastPointerLocation)` check, uninfluenced by any drag state. The
  widened drag-accept geometry is a completely separate decision from the `ignoresMouseEvents`
  click-through hit-test; AppKit already delivers drag callbacks regardless of `ignoresMouseEvents`
  (A1, confirmed), so no new state needs to feed into that arbiter at all.

Confirmed by 22-01's own execution (22-01-SUMMARY.md Deviations): `NSDraggingDestination` is an
`@objc optional` category protocol, NOT superclass members -- explicit `: NSDraggingDestination`
conformance on the class line (already added by 22-01, KEEP it) is required for Swift to compile the
4 methods at all; they satisfy protocol requirements (no `override` keyword), not override a
superclass implementation.

Confirmed via this session's SDK header read (AppKit.framework/Headers/NSDragging.h):
draggingExited has signature "- (void)draggingExited:(nullable id <NSDraggingInfo>)sender" -- the
ONLY one of the four NSDraggingDestination methods whose sender parameter is optional. This is why
Task 2's NotchPanelTests addition can call panel.draggingExited(nil) directly with a real nil
literal -- no mock/fake NSDraggingInfo is needed or possible for the other three (draggingEntered,
draggingUpdated, performDragOperation all require a non-optional id<NSDraggingInfo>, which has no
public Swift initializer). Also confirmed: NSWindow has NO public getter for its registered dragged
types (only the write-only registerForDraggedTypes(_:) method) -- do not attempt to assert
registration state via a "registeredDraggedTypes" property, it does not exist on NSWindow (NSView
has one, NSWindow does not).

Coordinate space of NSDraggingInfo.draggingLocation (developer.apple.com/documentation/appkit/
nsdragginginfo/dragginglocation): expressed in the BASE COORDINATE SYSTEM OF THE DESTINATION WINDOW
-- unlike NSEvent.mouseLocation, which handlePointer's own Pitfall-6 comment notes is ALREADY global
screen coordinates ("No coordinate conversion -- both point and hotZone are global bottom-left").
draggingLocation MUST therefore be converted to global screen coordinates before it can be compared
against expandedZone/dragLandingMaxY with CGRect.contains()/simple comparison. NotchPanel is
borderless (no titlebar/chrome, styleMask is [.borderless, .nonactivatingPanel]), but the conversion
should still go through the documented API rather than assume frame == contentRect math:
`panel.convertToScreen(NSRect(origin: draggingLocation, size: .zero)).origin` is NSWindow's own
supported way to convert a window-base point to a global screen point (NSWindow.convertToScreen(_:)
takes/returns an NSRect; a zero-size rect round-trips a single point cleanly).
