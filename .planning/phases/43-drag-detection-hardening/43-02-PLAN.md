  <files>Islet/Notch/NotchWindowController.swift, Islet/Notch/DragDropSupport.swift</files>
  <action>
    Build and run Islet (Debug, Cmd-R) with Plan 43-01's changes in place, then execute the 3
    on-device scenarios below in order. This task is verification-only — no further code changes
    unless a scenario fails, in which case report exactly which one and what happened instead.
  </action>
  <what-built>
    Plan 43-01 added `isGenuineFileDrag(currentChangeCount:gestureBaselineChangeCount:urls:)` and
    gated `recheckDragAcceptRegion`'s auto-expand arm branch on it, plus fixed
    `handleDragApproachEnd` to refresh the drag-pasteboard baseline on every mouse-up. This closes
    the false-trigger regression where any left-mouse-drag gesture (click wobble, window move, a
    non-file drag) inside the accept zone force-expanded the island.
  </what-built>
  <how-to-verify>
    Build and run Islet in Debug from Xcode (Cmd-R), then on-device:

    1. **Ordinary click** — with no drag in progress, click directly on the collapsed island
       several times in a row. Expected: nothing opens; the island only responds with its normal
       click-to-expand behavior (a genuine intentional click still opens the standard expanded
       view via `handleClick`, unrelated to this fix) — but a Quick Action picker must NEVER
       appear from a plain click.
    2. **Hover with no drag** — move the pointer over the collapsed island, then over the expanded
       island (after a real click), without holding any mouse button down. Expected: nothing
       related to the Quick Action picker opens; only the normal hover-affordance bounce/scale.
    3. **Real Finder file drag** — open Finder, pick up a real file, and drag it slowly toward the
       notch/island. Expected: the island still auto-expands and shows the 3-button Quick Action
       picker (Drop/AirDrop/Mail) exactly as before, with no perceptible added delay versus the
       pre-fix behavior. Drop the file on one of the 3 buttons (or drag it back out and release
       elsewhere to discard). Then, WITHOUT clicking the island again, move the pointer away and
       wait ~1 second: the island must auto-collapse on its own (D-03) — if it stays stuck
       expanded requiring a manual click to close, that is a regression to report.

    Repeat step 3 at least twice (once dropping on a button, once discarding) to confirm the
    auto-collapse recovers in both outcomes.
  </how-to-verify>
  <verify>
    <human-check>All 3 scenarios above behave as expected on-device.</human-check>
  </verify>
  <done>Ordinary click and hover never open the Quick Action picker; a real Finder file drag still opens it and the island auto-collapses on its own afterward.</done>
