## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WindowServer (private CGS API) → app process | Raw, undocumented C callback (`CGSRegisterNotifyProc`) delivered from the OS window-server connection into this un-sandboxed app, with no main-thread guarantee and no Apple-documented payload contract |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Denial of Service (crash) | `fullscreenProbeCallback` (NotchWindowController.swift) | mitigate | Callback body only hops via `DispatchQueue.main.async` before touching `self`/AppKit — never touches `@Published`/AppKit state directly on the raw callback thread, mirroring RESEARCH.md Pitfall 1 |
| T-08-02 | Tampering (use-after-free / dangling pointer) | `probeContext` (`Unmanaged<NotchWindowController>`) | mitigate | `CGSRemoveNotifyProc` called in `deinit` with the EXACT same `fullscreenProbeCallback`/`type`/`probeContext` triple used at registration, mirroring the existing `spaceObserver`/`appActivateObserver`/`powerMonitor` teardown discipline in the same `deinit` |
| T-08-03 | Information Disclosure / instability | CGS event payload (`data`/`dataLength` params) | accept (not applicable — mitigated by design) | The probe callback only reads the `type` parameter (106/107); `data`/`dataLength` are never dereferenced or parsed, so no untrusted payload parsing surface is introduced (matches RESEARCH.md Security Domain V5 minimal-input-validation finding) |
| T-08-04 | Denial of Service (this is throwaway probe code) | DEBUG-only print statements | accept | Gated entirely behind `#if DEBUG`; never compiled into Release builds, so no release-build risk from this temporary instrumentation |
