<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| System pasteboard (`NSPasteboard.general`) → `ClipboardMonitor` | Already Phase 57's boundary — untouched by this plan, this plan only becomes the first real *producer* of a marker-tagged write consumed by that existing boundary. |
| User mouse click / keyEquivalent → `AppDelegate` menu actions | New in this plan: two independent trigger paths (SwiftUI gesture, AppKit action) converge on the same `restore(_:)` function — the only place that writes to the system pasteboard. |
| On-disk encrypted store (`index.json.enc`) → in-memory `clipboardStore` | Already Phase 56's boundary (`ClipboardFileStore.load`, D-04 graceful degradation) — this plan is the first real caller of `load` at launch, on the production path. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-58-01 | Tampering / Denial of Service | `ClipboardRowView`'s image-thumbnail rendering, `restore(_:)`'s image branch | mitigate | Never force-unwrap `NSImage(data:)` — use `if let`, matching `ClipboardFileStore.load`'s existing D-04 graceful-degradation discipline; a corrupted/truncated decrypted image degrades to a missing thumbnail, never a crash |
| T-58-02 | Spoofing | A malicious/compromised app synthesizing a `keyEquivalent`-triggered menu action | accept | Same trust model as every other `NSMenuItem` action in this menu (Settings…, Quit) — no new privilege boundary crossed, no worse than the app's existing attack surface; out of scope for this phase |
| T-58-03 | Information Disclosure | `restore(_:)` writing plaintext clipboard content to the system pasteboard | accept | This is the explicit, user-initiated purpose of the feature (CLIP-02) — the system pasteboard is inherently readable by any app with pasteboard access, same as a normal Cmd+C/Cmd+V; no new disclosure beyond what a manual copy/paste would already expose |
| T-58-04 | Tampering | A future caller re-ingesting `restore(_:)`'s own write as a new capture (infinite self-capture loop) | mitigate | Every `restore(_:)` write is tagged with `ClipboardMonitor.restoreMarkerType` via `pbItem.setData(Data(), forType:)`, consumed by the already-proven `isSelfCaptureMarker` guard in `ClipboardMonitor.poll()` (Phase 57, unit-tested) |

No package-manager installs occur in this plan (Swift/AppKit-only, zero new dependencies) — the Package Legitimacy Gate is not applicable.
</threat_model>
