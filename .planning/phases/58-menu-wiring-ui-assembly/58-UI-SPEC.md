# Phase 58 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform note:** This is a native macOS `NSMenu`/`NSMenuItem` surface (AppKit), with row content hosted via `NSHostingView`-wrapped SwiftUI — not a web app. There is no shadcn/Tailwind/CSS design system in this codebase; "Tool: none" below is the correct native-platform answer, not a skipped step. Values are declared in points (pt), matching AppKit/SwiftUI convention, not px. Colors are declared as macOS system semantic colors (`NSColor`/`Color`), not hex, so they automatically adapt to light/dark menu-bar appearance — the correct native equivalent of a fixed palette.

## Spacing Scale

Native `NSMenuItem` geometry is fixed by AppKit (standard single-line row = 22pt), so this phase's spacing values are the ones locked by `58-RESEARCH.md`'s Pattern 2 code example, not a fresh 8-point-grid derivation:

| Token | Value | Usage |
|-------|-------|-------|
| Row height | 22pt | Height of every clipboard row's `NSHostingView` — matches native single-line `NSMenuItem` height (D-10) |
| Row horizontal padding | 14pt | `.padding(.horizontal, 14)` inside `ClipboardRowView` — approximates native `NSMenuItem` text inset |
| Icon-to-text gap | 6pt | `HStack(spacing: 6)` between thumbnail and text in image rows |
| Thumbnail size | 18pt × 18pt | Image entry thumbnail (D-10: "~16-20pt" range — 18pt is the exact value locked in research's code example) |
| Thumbnail corner radius | 3pt | `RoundedRectangle(cornerRadius: 3)` clipping the thumbnail |

Exceptions: none of the above values are on the project's general 4pt/8pt UI grid — they are AppKit-imposed constraints (native menu row height, native menu item text inset) rather than freely-chosen layout spacing, so the general spacing scale does not apply to this phase's menu-row geometry. This mirrors how `NotchPillView.swift`/`SettingsView.swift` also use fine-grained non-4pt values (e.g. 7.5pt, 9pt font sizes) wherever they mirror a system-imposed metric rather than a free layout choice.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Clipboard row text (text entries) | 13pt | regular | 1 line, `.lineLimit(1)` / `.truncationMode(.tail)` (single-line native menu row — no multi-line body text in this phase) |
| Clipboard row label (image entries, "Image" caption) | 13pt | regular | 1 line, same truncation rule |
| Empty-state placeholder row ("No items yet") | 13pt | regular, `isEnabled = false` (system-dimmed disabled-item color, not a custom gray) | 1 line |
| "Delete All History" menu item | 13pt | regular | 1 line |
| Pasteboard-access `NSAlert` title | System alert title style (`alert.messageText`, AppKit-default bold ~13pt) | bold (AppKit default) | 1 line |
| Pasteboard-access `NSAlert` body | System alert informative-text style (`alert.informativeText`, AppKit-default ~11-13pt) | regular (AppKit default) | wraps, 2-3 lines expected |

13pt is the macOS system default menu-item font size (`NSFont.menuFont(ofSize: 0)`) — matching this size on the SwiftUI-hosted rows is what makes them read as native rows sitting flush against the existing plain-text Settings…/Check for Updates…/Quit items, per this phase's explicit visual goal (RESEARCH.md: "as long as it reads as a single-line row matching the surrounding native NSMenuItems in height/style"). No second weight is introduced — every row uses regular weight; the only non-regular text is AppKit's own default bold alert title, which this phase does not override.

## Color

Native semantic colors only — no custom hex palette, so the menu automatically matches system light/dark appearance (already this project's convention: `NSColor.systemRed` at `AppDelegate.swift:80`, `Color.accentColor` in `SettingsView.swift:172`).

| Role | Value | Usage |
|------|-------|-------|
| Dominant | System menu background (`NSMenu`'s own drawn vibrancy — not set by app code) | The menu's background surface — entirely AppKit-owned, this phase never overrides it |
| Secondary (row hover) | Manual `.onHover` background tint approximating system selection color (e.g. `Color.primary.opacity(0.08)` or `Color(nsColor: .selectedContentBackgroundColor)`, exact token Claude's discretion at build time) | Custom-view `NSMenuItem`s lose AppKit's automatic hover highlight (Pitfall 1) — this phase's `.onHover` handler must restore a visually equivalent highlight so rows still look interactive on mouse-over, matching the native highlight the static Settings/Check-for-Updates/Quit items still get for free |
| Row text | `Color.primary` (system label color) | Text-entry and image-entry captions — no custom text color |
| Destructive | `NSColor.systemRed` (via `NSAlert.buttons.first?.hasDestructiveAction = true`, AppKit's own destructive-button styling) | Reserved exclusively for the "Delete" button inside the "Delete All History" confirmation `NSAlert` — matches the existing `NSColor.systemRed` precedent at `AppDelegate.swift:80` (the update-available dot), the only other place this codebase uses red |

Accent reserved for: nothing in this phase uses `Color.accentColor` — there is no primary/accent-colored element in the clipboard menu (rows are plain text/thumbnail, "Delete All History" is destructive-red not accent, empty-state row is system-disabled gray). This phase introduces zero new accent-colored UI.

## Interaction Contract (native-platform addendum)

Standard template sections above don't fully capture two AppKit-specific interaction requirements this phase's checker/executor must verify — recorded here since they're load-bearing visual/interaction contract, not incidental implementation detail:

1. **Dual trigger paths per row** (Pitfall 1, RESEARCH.md): every clickable clip row must independently support (a) a SwiftUI `.onTapGesture` for mouse clicks and (b) the `NSMenuItem`'s own `action`/`target`/`keyEquivalent` for ⌘0-⌘9 — both must call the identical restore function. A row that only visually responds to one of the two paths fails this contract even if it "looks" correct on hover.
2. **MRU ordering** (Pitfall 4): rows render newest-first (`ClipboardStore.items.reversed()`), top of menu = most recently copied. Oldest-at-top is a visual contract failure, not just a data-ordering bug.
3. **Section order** (D-15): clipboard rows → "Delete All History" → `.separator()` → existing Settings…/Check for Updates…/Quit block, top to bottom. Matches the user's CopyClip reference screenshot exactly.
