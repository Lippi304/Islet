<objective>
Stop discarding the real Polar.sh validation payload. `PolarLicenseService.activate` already
decodes `id`/`status`/`expiresAt` from the server's 200 response into a private
`ValidatedLicenseKey`, but the `LicenseService` protocol only returns `Result<Void, ...>`, so
`SettingsView.activate()` fabricates a fresh `LicenseRecord(licenseID: "", status: "granted", ...)`
from scratch instead of persisting what the server actually said. `LicenseManager.isLicensed` only
ever re-checks this local cached Keychain record — it never asks Polar again — so today a
refund/chargeback can never re-lock the app client-side, because the data needed for that future
enforcement was never even stored.

Purpose (CONTEXT.md item 7, explicit behavior-changing exception to this phase's otherwise
zero-behavior-change policy, D-03): widen the contract so the real payload survives end-to-end.
This phase does NOT add expiry/revocation enforcement — only the data needed for it, per
CONTEXT.md. Runtime effect for today's users is identical (still ends in `.licensed`).
Output: `ValidatedLicense` result type threaded through `LicenseService` →
`PolarLicenseService`/`StubLicenseService` → `SettingsView.activate()` →
`LicenseManager.recordValidation(key:validated:)`.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-04 | Tampering | `PolarLicenseService`'s 200-branch decode → persisted `LicenseRecord` | mitigate | Unchanged from Phase 12: the endpoint is TLS-only (`https://api.polar.sh`), and the response is only accepted when `validated.status == "granted"` (the existing guard in `PolarLicenseService.swift`, untouched by this phase) before any persistence occurs — this phase does not weaken that gate, it only stops discarding the already-validated fields after it passes. |
| T-15-05 | Information Disclosure | Persisted `licenseID`/`status`/`expiresAt` in Keychain | accept | Same Keychain item/ACL as the existing `key`/`status`/`validatedAt` fields (`kSecAttrAccessibleAfterFirstUnlock`, local-device-only, per `KeychainLicenseStore.swift`); no new sensitive data class is introduced, no network egress of this data. |
| T-15-06 | Repudiation (stale entitlement, no revocation re-check) | `LicenseManager.isLicensed` reads only the cached local record, never re-queries Polar | accept (explicitly deferred, not introduced by this phase) | CONTEXT.md item 7 explicitly scopes this phase to PERSISTING the data future revocation enforcement needs, not adding the enforcement itself — this carries forward the existing Phase-12 "no re-validation" behavior unchanged; a chargeback/refund still cannot re-lock the app client-side until a future phase adds the enforcement check. |
