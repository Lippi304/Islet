<objective>
Fix the audit's one confirmed rendering bug: `EqualizerBars`' doc comment claims its random
per-bar profile (height range, oscillation period, phase offset) is "generated ONCE at init and
held stable for the view's lifetime" — false for a plain `struct View`. `NotchPillView` reconstructs
`EqualizerBars(...)` on every parent `body` pass (which fires on every Now Playing position tick,
since `nowPlaying` is `@ObservedObject`), so `init`'s `CGFloat.random` calls re-run every time,
visibly reshuffling the bars instead of staying stable.

Purpose (CONTEXT.md item 6, explicit behavior-changing exception to this phase's otherwise
zero-behavior-change policy, D-03): make the profile ACTUALLY stable, matching what the existing
comment already promises, by moving the random generation into a `@State` initial-value expression
(evaluated once per view identity, not once per struct construction).
Output: `EqualizerBars.makeProfiles()` static factory + `@State`-seeded `profiles`;
`IsletTests/EqualizerBarsTests.swift`.

Wave 2 (depends on 15-01): both plans modify `Islet/Notch/NotchPillView.swift` — this plan starts
after 15-01's `blobShape()` extraction lands, so the file only needs one sequential set of edits
instead of a merge conflict between two same-file plans.
</objective>
