# Phase 33 — UI Design Contract

> Visual and interaction contract for the Weather widget redesign (WEATHER-01, WEATHER-02). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform note:** Islet is a native macOS SwiftUI app, not a web project — the shadcn/web-registry gate in the researcher's standard flow does not apply (no `components.json`, no Tailwind, no npm-based component library). All values below are in SwiftUI points (pt), not CSS px. The "Design System" section documents this project's own established SwiftUI conventions instead of a web preset.

## Spacing Scale

Islet does not enforce a strict 8pt web grid — it uses a native-SwiftUI point scale tuned per element via on-device UAT (see Tray's `HStack(spacing: 14)` precedent). Declared values for this phase:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Compact card's icon→temp→label vertical stack gap (existing `weatherFullContent` `VStack(spacing: 8)` uses 8, forecast chip internals use 4) |
| sm | 8pt | Forecast chip icon→weekday→H/L internal gaps; inter-chip horizontal gap in the forecast row |
| md | 16pt | Left/right content inset (matches every existing full-view: `weatherFullUnavailable`, `switcherRow`, `mediaExpanded` all use `.padding(.horizontal, 16)`); vertical gap between the compact card and the forecast row |
| lg | 24pt | Not used this phase |
| — | 42pt | `Self.cameraClearance` — fixed existing constant, top-of-card clearance for the physical camera/notch, unchanged by this phase |

Exceptions: 42pt camera clearance is a fixed pre-existing project constant, not part of the 4pt scale — reuse it verbatim, do not redefine it for Weather.

## Typography

All sizes in pt, `design: .rounded` (project-wide convention — do not switch to `.default`/`.serif`).

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Display | 32pt | semibold (600) | default (SwiftUI system) | Current temperature — reuse `weatherFullContent`'s existing style verbatim, `.monospacedDigit()`, `.foregroundStyle(.white)` |
| Body | 14pt | regular (400) | default | Reserved — unchanged `weatherFullUnavailable` text (pre-existing, out of scope this phase; currently rendered `.medium` weight — see note below) |
| Label | 13pt | regular (400) | default | Condition category label under the temperature (existing `weatherCategoryLabel`, `.foregroundStyle(.secondary)`) — also the location-name label (new, D-01/D-02), same style |
| Caption | 11pt | regular (400) | default | New: forecast-day chip weekday label (e.g. "Mon"), `.foregroundStyle(.secondary)` |

Forecast-chip H/L text: 12pt, semibold (600), `.monospacedDigit()`, `.foregroundStyle(.white)` — reuses the Display role's weight token at a smaller size rather than introducing a third weight.

**2-weight budget:** regular (400) + semibold (600) for all new copy this phase. Note: the pre-existing `weatherFullUnavailable` string already renders at `.medium` (500) — that view is unchanged by this phase's scope (WEATHER-01/02 touch the populated-data path only), so it is a pre-existing exception, not a new one.

**Forecast icon size:** 20pt SF Symbol (`weatherIcon(for:).font(.system(size: 20))`) — smaller than the compact card's 44pt icon, sized to fit inside a ~70-90pt-wide chip alongside the weekday label and H/L text.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Existing Phase 25 black→transparent vertical gradient shell material | Card background — unchanged, inherited from `blobShape()`, not touched by this phase |
| Secondary (30%) | `.secondary` (system semantic grey) | Category label, location-name label, forecast weekday labels — all non-hero text |
| Accent (10%) | none introduced | See "Accent reserved for" below |
| Destructive | not applicable | No destructive actions in this phase (display-only widget + one Settings toggle) |

**Accent reserved for:** Weather introduces **no new accent tint**. This matches the existing project convention (D-10, Phase 6/25/27 per-element accent pattern: `nowPlayingAccent`/`chargingAccent`/`deviceAccent` are reserved for elements tied to a live/in-progress activity — equalizer bars, battery indicator, progress-bar fill). Weather is a passive data-display widget, not a live activity, and its current implementation (`weatherFullContent`) already renders fully untinted (`.white` for hero text, `.secondary` for supporting text, no accent anywhere). The new forecast chips follow the exact same untinted convention — do not add a `weatherAccent` environment key or tint any forecast element. High temperature and low temperature both render in the same white/monospaced style as the compact card's current-temperature text (no separate "hot=red/cold=blue" semantic coloring — out of scope, not requested).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — Weather is a passive display widget with no button in the card itself. The one new interactive element is the Settings toggle: label **"Extended forecast"**, inside a new **"Weather"** Section in Settings (source: 33-RESEARCH.md Code Examples, mirrors `ActivitySettings.MaterialStyle`'s existing toggle pattern verbatim) |
| Location fallback | **"Local"** — shown while reverse-geocoding is pending or on any failure (no permission, no network, geocode error). Occupies the exact same layout slot the resolved place name will use — no layout shift on resolve (D-01/D-02, locked). Never blank, never an error string, never a retry affordance. |
| Empty/error state | Unchanged this phase: existing `weatherFullUnavailable` — **"Wetter nicht verfügbar"**. Flagged for awareness, not for this phase's action: every other empty/unavailable string in this codebase is English ("Nothing Playing", "No events today", "No files yet") — this German string (shared with `mediaUnavailable`'s "Now Playing nicht verfügbar") is a pre-existing outlier, most likely leftover rather than deliberate localization. WEATHER-01/02's scope is the populated-data compact/extended card, not this fallback view — do not touch it in this phase. If a future phase revisits it, align to English for consistency with the rest of the app. |
| Destructive confirmation | Not applicable — no destructive actions introduced this phase |
| Forecast day-chip copy | Weekday abbreviation (e.g. "Mon") + condition icon + high/low string (e.g. "18°/12°") — per D-06, locked. Use the existing locale-aware `.formatted(.measurement(width: .narrow, numberFormatStyle: .number.precision(.fractionLength(0))))` convention for each temperature, joined as `"\(high)/\(low)"` — no manual unit conversion, no new formatting helper. |

## Layout Contract (native-app supplement — not in the standard web template, included because this phase's core risk is geometry, not copy)

### Compact card (WEATHER-01, unchanged structurally from today, extended with H/L + location)
- Existing `weatherFullContent` structure stands: `VStack(spacing: 8)` — icon (44pt) → temperature (32pt Display) → category label (13pt Label).
- **New this phase:** add the location-name label (13pt Label, `.secondary`, D-01/D-02's "Local" fallback) and the H/L readout, matching the reference layout "Local / 16° Cloudy H:24 L:15". Recommended placement: location label above the icon (top of the stack, since it identifies *where* before *what*), H/L readout as a small line directly under the category label — do not restructure the icon/temperature hero pairing, which is already correct and shipped.
- Content height unchanged: still shorter than `switcherContentHeight` (196pt), top-aligns with empty space below exactly as today — no height-constant change for the compact-only state.

### Extended card (WEATHER-02, new)
- Appends a forecast row below the existing compact content, inside the same `blobShape()` card, gated by `@AppStorage(ActivitySettings.weatherExtendedKey)`.
- **Forecast row:** `HStack(spacing: 8)` of day-chips, no `ScrollView` (D-05, locked — a fixed-count row that always fits, not Tray's scrolling pattern).
- **Day-chip count:** 5 (today + next 4), starting from `daily[0]` (today) through `daily[4]`. Rationale: content area is `420 − 2×16 = 388pt` wide (`Self.expandedSize.width` minus the standard 16pt horizontal insets); at 5 chips with 8pt inter-chip gaps, each chip gets `(388 − 4×8) / 5 = 71.2pt` — comfortably fits a 20pt icon + an 11pt weekday label + a 12pt H/L string stacked vertically with room to spare. This is the recommended default per CONTEXT.md D-05's "pick whichever count fits cleanly" — confirm on first on-device build exactly like Tray's width required 3 UAT rounds (32-01-SUMMARY.md); drop to 4 chips (each ~91pt) only if 5 visibly crowds on the real hardware.
- **Day-chip structure:** `VStack(spacing: 4)` — weekday label (11pt Caption, `.secondary`) → icon (20pt SF Symbol, `weatherIcon(for:)`) → H/L text (12pt, semibold, `.monospacedDigit()`, `.white`).
- **`weatherExtendedContentHeight` starting value: 240pt.** Worked math (same box-math-in-a-comment convention as `trayContentHeight`/`switcherRowHeight`'s own doc comments): `cameraClearance(42) + icon(44) + spacing(8) + temp(32) + spacing(8) + location+H/L label lines(~16) + section gap(16) + forecast chip stack(~56: weekday 14 + spacing 4 + icon 20 + spacing 4 + H/L 14) + bottom inset(16) ≈ 238`, rounded to 240pt. Per 33-RESEARCH.md Open Question 1, this is explicitly a **starting point for on-device tuning**, not a locked pixel-perfect value — treat it exactly like `trayContentHeight`'s own iteration history.
- **Geometry three-site rule (Pitfall 4, Anti-Pattern 1 — this project's twice-repeated CR-01/WR-02 bug class):** `weatherExtendedContentHeight` MUST be wired into all three of: (1) `blobShape()`'s `height:` override call site in `weatherFullView`, (2) `positionAndShow`'s panel-frame union (`weatherExtendedFrame`, mirroring `trayFrame`), and (3) `visibleContentZone()`'s `.weatherExpanded` branch — in the same task/commit. Verify via the on-device hover→expand→click trace this project already uses for CR-01-class regressions.
- **Live toggle (D-04):** flipping the Settings toggle animates the compact↔extended height change via the existing spring, exactly like every other size transition in the app (Home↔Tray, collapsed↔expanded) — no new animation curve, reuse `handleSettingsChanged()`'s existing `withAnimation(...)` re-render path.
