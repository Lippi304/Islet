# Phase 33 — UI Design Contract

> Visual and interaction contract for the Weather widget redesign (WEATHER-01, WEATHER-02). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**REVISION NOTE (2026-07-15):** This supersedes the original version's forecast-row content. Plan 33-02 Tasks 1-2 shipped a 5-day weekday-chip row behind a boolean toggle (commits `f0e6bf0`, `580485d`) — this was rejected at the Task 3 on-device checkpoint. The user supplied `.planning/research/inspiration/32.png`, a real screenshot of Apple's own iOS Weather widget (Medium vs. Large), as the authoritative reference. This revision replaces the forecast-row/Settings-control/geometry sections below with a 1:1 clone of that reference: **Medium** (always-on baseline: header + hourly row) and **Large** (Settings-selectable: Medium's content + a daily forecast list with min/max range bars). The **Header** section is unchanged — CONTEXT.md confirms it was already built correctly and is not being reworked.

**Platform note:** Islet is a native macOS SwiftUI app, not a web project — the shadcn/web-registry gate in the researcher's standard flow does not apply (no `components.json`, no Tailwind, no npm-based component library). All values below are in SwiftUI points (pt), not CSS px. The "Design System" section documents this project's own established SwiftUI conventions instead of a web preset.

## Spacing Scale

Islet does not enforce a strict 8pt web grid — it uses a native-SwiftUI point scale tuned per element via on-device UAT (see Tray's `HStack(spacing: 14)` precedent). Declared values for this phase:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Chip/row internal vertical stack gaps (hourly chip's time→icon→temp gap; range-bar height) |
| sm | 8pt | Inter-chip horizontal gap in the hourly row; inter-row vertical gap in the Large daily list; internal column gap within a daily row (weekday/icon/low/bar/high) |
| md | 16pt | Left/right content inset (matches every existing full-view: `weatherFullUnavailable`, `switcherRow`, `mediaExpanded` all use `.padding(.horizontal, 16)`); vertical gap between the header and the hourly row, and between the hourly row and the daily list (Large only) |
| lg | 24pt | Not used this phase |
| — | 42pt | `Self.cameraClearance` — fixed existing constant, top-of-card clearance for the physical camera/notch, unchanged by this phase |

Exceptions: 42pt camera clearance is a fixed pre-existing project constant, not part of the 4pt scale — reuse it verbatim, do not redefine it for Weather.

## Typography

All sizes in pt, `design: .rounded` (project-wide convention — do not switch to `.default`/`.serif`). **Still a 4-size scale: 32 / 13 / 12 / 11pt** — every new element this phase reuses one of these four; no fifth size is introduced.

| Role | Size | Weight | Usage |
|------|------|--------|-------|
| Display | 32pt | semibold (600) | Current temperature — header, unchanged, `.monospacedDigit()`, `.foregroundStyle(.white)` |
| Body | 13pt | regular (400) | Header: location label, condition label, H/L readout (unchanged). Large daily list: low-temperature text, `.foregroundStyle(.secondary)` |
| Weekday/High | 13pt | semibold (600) | New: Large daily list's weekday label ("Do") and high-temperature text — same 13pt size as Body, differentiated by weight+color (`.white`, not `.secondary`) rather than a new size, per the project's 2-weight budget |
| H/L | 12pt | semibold (600) | Hourly chip's temperature text (e.g. "14°"), `.monospacedDigit()`, `.foregroundStyle(.white)` — same role/size the old daily-chip row used |
| Caption | 11pt | regular (400) | Hourly chip's time label (e.g. "18:00"), `.foregroundStyle(.secondary)`, locale-formatted via `.formatted(date: .omitted, time: .shortened)` — not a hardcoded "H:mm" string, matches the project's locale-aware-formatting convention already used for temperature |

**2-weight budget (unchanged):** regular (400) + semibold (600) for all copy this phase.

**Forecast icon sizes (SF Symbol point size, not a text role):**
- Hourly chip icon: **16pt** (`weatherIcon(for:).font(.system(size: 16))`) — smaller than the old 20pt daily-chip icon because 6 hourly chips must fit the same 388pt content width the old 5 daily chips used; tune down to 14pt only if 16pt visibly crowds on-device.
- Large daily-row icon: **18pt** — has more horizontal room (single-column list, not a multi-chip row), sized between the hourly chip's 16pt and the header's 44pt.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Existing Phase 25 black→transparent vertical gradient shell material | Card background — unchanged, inherited from `blobShape()`, not touched by this phase |
| Secondary (30%) | `.secondary` (system semantic grey) | Header's location/condition labels, hourly chip time labels, daily-row low-temperature text — all non-hero text |
| Accent (10%) | none introduced | See "Accent reserved for" below |
| Destructive | not applicable | No destructive actions in this phase (display-only widget + one Settings segmented control) |

**Accent reserved for:** Weather introduces **no new reserved accent tint**. This matches the existing project convention (Phase 6/25/27 per-element accent pattern: `nowPlayingAccent`/`chargingAccent`/`deviceAccent` are reserved for elements tied to a live/in-progress activity). Weather is a passive data-display widget, not a live activity — hero text stays `.white`, supporting text stays `.secondary`, exactly as the shipped header already does.

**Temperature-scale gradient (Large's range bar only — NOT the reserved-accent system, D-08):** The daily list's range bar is a data-visualization color encoding (matches Apple's own reference, image 32), distinct in kind from the app's live-activity accent tokens above — it does not compete for the "10% accent" budget because it renders inside Large's own dedicated forecast-list area, not on a shared/global chrome element.
- Fixed 5-stop cold→hot ramp, sampled by each day's midpoint temperature `(low + high) / 2` normalized against the **visible list's own overall min(low)/max(high) span** (D-08's exact wording — the same span every row's bar is positioned within): `0.0 → .blue`, `0.25 → .mint`, `0.5 → .yellow`, `0.75 → .orange`, `1.0 → .red`. Use SwiftUI's built-in `Gradient(colors:)` + `.interpolatingSpring`-free linear sampling — no custom color-math class needed (`Gradient(stops:)` already interpolates between named stops).
- Each row's bar itself is a short `LinearGradient` from the color sampled at that day's `low` to the color sampled at that day's `high` (not a flat fill) — reproduces image 32's visible left-to-right warm/cool fade within a single day's bar.
- Exact color stops are Claude's discretion per CONTEXT.md ("match the spirit... without needing pixel-exact color stops") — the above 5 SwiftUI system colors are the recommended default, swap for closer Apple-hue matches only if a UAT round flags them as visibly off.

## Layout Contract (native-app supplement — not in the standard web template, included because this phase's core risk is geometry, not copy)

### Header (WEATHER-01, unchanged — not reworked this revision)
Per CONTEXT.md's canonical refs, `weatherFullContent` (`NotchPillView.swift` ~790) is already correct and must be **kept as-is, not rebuilt**:
- `VStack(spacing: 8)` — location label (13pt Body, `.secondary`, "Local" fallback) → icon (44pt) → temperature (32pt Display) → category label (13pt Body, `.secondary`) → H/L readout (13pt Body, `.secondary`, omitted entirely when either bound is `nil`).
- Header content height (unchanged): ≈162pt (16 location + 8 + 44 icon + 8 + 38 temp-line + 8 + 16 category + 8 + 16 H/L).
- This structure does **not** match image 32's two-column (location/temp left, icon/condition/H-L right) layout — that visual difference is a known, accepted gap per CONTEXT.md's explicit "keep, do not rebuild" instruction; do not "fix" it to match the screenshot more closely without a separate user decision.

### Medium — always-on baseline (WEATHER-01/02, D-03/D-06/D-07 — REPLACES the old daily-chip forecast row)
There is no more header-only/"Compact" state. Medium = header + hourly forecast row, shown unconditionally.
- **Hourly row:** `HStack(spacing: 8)` of time-chips, directly below the header, `.padding(.top, 16)` (section gap), `.padding(.horizontal, 16)` — no `ScrollView` (D-07, locked — fixed-count row that always fits).
- **Chip count: 6** (matches image 32's reference exactly — content area is `420 − 2×16 = 388pt` wide; at 6 chips with 8pt inter-chip gaps, each chip gets `(388 − 5×8) / 6 = 58pt` — fits a 16pt icon + an 11pt time label + a 12pt temp stacked vertically with modest margin). Drop to 5 chips (each ~68.8pt) only if 6 visibly crowds on real hardware — same "starting recommendation, confirm on first on-device build" convention Tray's own width took 3 rounds to settle (32-01-SUMMARY.md).
- **Chip structure:** `VStack(spacing: 4)` — time label (11pt Caption, `.secondary`, locale-shortened) → icon (16pt SF Symbol, `weatherIcon(for:)`) → temperature (12pt semibold, `.monospacedDigit()`, `.white`).
- **`weatherMediumContentHeight` starting value: ≈290pt.** Worked math (same box-math-in-a-comment convention as `trayContentHeight`): `cameraClearance(42) + header(162) + rowTopPadding(16) + chipStack(time 14 + spacing 4 + icon 16 + spacing 4 + temp 15 ≈ 53) + bottomInset(16) ≈ 289`, rounded to 290pt. This is a genuinely new value (Medium was previously the toggle-off/header-only state falling through to the shared `switcherContentHeight` 196pt box — that fallback no longer applies now that the hourly row is permanent, per D-03). Treat as a starting point for on-device tuning, not a locked pixel value.

### Large — daily forecast list with range bars (WEATHER-02, D-08/D-09 — entirely NEW, highest-risk/most-novel component this phase)
Appends, below Medium's hourly row, a daily forecast list: one row per day.
- **Section gap:** `.padding(.top, 16)` above the list, matching the header→hourly-row gap.
- **Row count: 5** (matches image 32's Do–Mo reference exactly). Drop to 4 only if row height forces it — Claude's discretion per D-09, same "fit cleanly" principle as the hourly row.
- **Row structure:** `HStack(spacing: 8)` — weekday label (13pt semibold, `.white`, fixed width ~28pt, leading) → icon (18pt SF Symbol, `weatherIcon(for:)`, fixed width ~24pt) → low temperature (13pt regular, `.secondary`, fixed width ~30pt) → range bar (flexible width `Capsule()`, height 4pt, corner radius 2pt, temperature-scale `LinearGradient` fill per the Color section above, horizontally inset/sized within the row's remaining space proportional to `(day.low, day.high)`'s position inside the visible list's overall min/max span) → high temperature (13pt semibold, `.white`, fixed width ~30pt). Columns line up vertically across all 5 rows (fixed-width leading/trailing columns, flexible bar in the middle) — matches image 32's aligned-column look.
- **`weatherLargeContentHeight` starting value: ≈470pt.** Worked math: `cameraClearance(42) + header(162) + rowTopPadding(16) + hourlyChipStack(53) + dailySectionGap(16) + dailyList(5 rows × 26pt row-height + 4 gaps × 8pt spacing = 162) + bottomInset(16) ≈ 467`, rounded to 470pt. **Explicitly flagged as the least-certain number in this spec** (D-10's own framing: needs its own on-device checkpoint pass for both Medium→Large and Large→Medium transitions) — do not treat as final without on-device confirmation.
- **Geometry three-site rule — now TWO non-collapsed sizes, not one (D-10, this phase's biggest geometry-surface change):** both `weatherMediumContentHeight` and `weatherLargeContentHeight` must each be wired into all three of: (1) `blobShape()`'s `height:` override call site in `weatherFullView`, (2) `positionAndShow`'s panel-frame union (mirroring `trayFrame`), and (3) `visibleContentZone()`'s `.weatherExpanded` branch — for BOTH sizes, in the same task/commit. Verify via the on-device hover→expand→move-down click-through trace this project already uses for CR-01/WR-02-class regressions, run once at Medium and once at Large per D-10.

### Settings control (D-04, revised — segmented Picker replaces boolean Toggle)
- Two options only, no third "Compact" option (Compact no longer exists as a selectable state per the revised D-03): `Picker("Weather Style", selection: $weatherStyle) { Text("Medium").tag(WeatherStyle.medium); Text("Large").tag(WeatherStyle.large) }.pickerStyle(.segmented)`, inside the existing "Weather" `Section`.
- **Storage:** introduce a new `String`-backed `@AppStorage` key (recommend `weather.style`, values `"medium"`/`"large"` via a `WeatherStyle: String, CaseIterable` enum, mirroring `ActivitySettings.MaterialStyle`'s exact pattern) rather than mutating the existing Bool `weatherExtendedKey` in place — a Bool→String type change on the same UserDefaults key is unnecessarily fragile, and this key has not shipped in a closed v1.5 release yet (low migration risk). Absent-key default resolves to `.medium` (D-04, locked).
- **Live animation (D-05):** flipping the segmented control animates the height change live via the existing spring, exactly like every other size transition in the app — reuse `handleSettingsChanged()`'s existing `withAnimation(...)` re-render path, no new animation curve.
