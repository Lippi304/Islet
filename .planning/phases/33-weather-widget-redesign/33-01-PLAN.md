<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| App -> WeatherKit | `fetchCurrentAndForecast` sends the device's coarse coordinate to Apple's WeatherKit service and receives current+daily conditions back |
| App -> CLGeocoder | `resolvePlaceName` sends the same `CLLocation` to Apple's on-device/network reverse-geocoding service and receives a place name back |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-33-01 | Tampering | `WeatherKitService.fetchCurrentAndForecast` (external WeatherKit response) | accept | First-party Apple TLS-secured API; identical trust level to the Phase-14-shipped `fetchCurrent` call it replaces; no user-supplied input crosses this boundary |
| T-33-02 | Information Disclosure | `resolvePlaceName` reverse-geocode (device location sent to Apple's geocoding service) | accept | Reuses the SAME `CLLocation` already sent to WeatherKit for current-conditions since Phase 14; no new data collected, no new permission ask, no new network destination class |
| T-33-03 | Denial of Service | `DailyForecast` mapping via `daily.map { ... }` / `daily.first` | mitigate | Uses Swift's optional-safe `.first` (never force-unwrapped) — a WeatherKit response with zero daily entries settles `high`/`low` as `nil` rather than crashing, matching the existing D-01 silent-omission contract |
| T-33-SC | Tampering | npm/pip/cargo installs | n/a | Zero third-party packages this task — WeatherKit and CoreLocation are both already-linked first-party frameworks (RESEARCH.md Package Legitimacy Audit: not applicable) |
</threat_model>
