<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| User tap -> NSWorkspace.shared.open(deep link URL) | Opens System Settings via a URL scheme; anchor strings are compile-time constants from Plan 01's `PermissionKind`, never user/external input |
| User tap -> native permission-request APIs | `CLLocationManager`/`EKEventStore`/`FocusModeMonitor`/`NotchWindowController.requestBluetoothPermission()` — all first-party, in-process calls |
| SettingsView -> NotchWindowController (Replay Onboarding) | Cross-window in-process call via `AppDelegate.notchController`, identical trust level to the existing `focusPermissionGranted()` precedent |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-06 | Tampering | `handlePermissionTap`'s deep-link URL construction | mitigate | `kind.deepLinkAnchor` is a hardcoded, compile-time constant sourced from Plan 01's `PermissionKind` enum — never built from user/external input, matching the existing no-injection-surface precedent at `SettingsView.swift:592-593` |
| T-54-07 | Information Disclosure | Permission status display (5 rows + summary) | accept | Local-only OS-standard TCC state (granted/denied/not-determined), no PII, no network transmission |
| T-54-08 | Tampering | Replay Onboarding button | mitigate | Routes through Plan 02's `finishOnboardingReplay()`, which never writes `onboardingCompletedKey` (D-08) — this plan's UAT (Task 3, step 9) explicitly re-confirms no persisted-state change end-to-end |
| T-54-SC | Tampering (supply chain) | N/A — zero package installs | accept | No new dependency added by this plan |

</threat_model>
