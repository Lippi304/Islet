# Phase 54 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform note:** Islet is a native macOS SwiftUI/AppKit app — there is no web stack, no shadcn, no CSS. All values below are pt (points, SwiftUI's native unit) and `Color`/`Font` tokens, not hex/px. The shadcn initialization gate does not apply (not a React/Next.js/Vite project); "Tool: none" is set for that reason, not because a design system is missing — this app has its own established native design system, cataloged below from `Islet/SettingsView.swift` and `Islet/Notch/NotchPillView.swift`.

## Spacing Scale

Declared values (pt, matches this codebase's existing `SettingsView.swift` conventions — not a generic 8pt web grid):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Icon-to-status-glyph gap within a permission row (new, this phase) |
| sm | 8pt | Row-internal `HStack`/`VStack` spacing (matches `focusPermissionExplanationView`'s `spacing: 8`, `onboardingPermissionsStep`'s permission-row list `spacing: 6`→use 8 for new rows, not 6) |
| md | 16pt | Popover/card padding (matches `focusPermissionExplanationView`/`osdPermissionExplanationView`'s `.padding(16)`) |
| lg | 20pt | Section-level `ScrollView`/`Form` wrapper padding (matches every existing section: `diagnosticsSection`, `aboutSection`, `weatherSection`, all use `.padding(20)`) |
| xl | 32pt | Not used this phase |

Exceptions: pre-existing chrome elsewhere in `SettingsView.swift` uses 6pt (sidebar row `.padding(.vertical, 6)`) and 10pt (sidebar row `.padding(.horizontal, 10)`, `swatchRow` `HStack(spacing: 10)`) — these are pre-Phase-54 values, left untouched, and must NOT be copied into new Permissions-section code. All new rows/views this phase adds use only 4/8/16/20.

## Color

This is a native macOS app with adaptive (Light/Dark Mode) system colors, not a fixed hex palette — the 60/30/10 web convention maps to this app's existing semantic-color discipline instead:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | System window/`Form` background (adaptive, no hardcoded color) | Settings window background — unchanged, inherited from `NavigationSplitView`/`Form` |
| Secondary (30%) | `.secondary` (system adaptive gray) | Row hint text (the one-line "why" under each permission name), sidebar icon tint, `notYetAsked` status glyph/text |
| Accent (10%) | `Color.accentColor` | Sidebar section-selection background fill ONLY (`Color.accentColor.opacity(0.25)`, pre-existing, unchanged by this phase) — NOT used for permission status, to avoid conflating "selected in sidebar" with "permission granted" |
| Destructive / status-denied | `.red` | `denied` status glyph (`xmark.circle.fill`) and status text only — matches existing `"✗ That key wasn't recognized.".foregroundStyle(.red)` precedent. No delete/destructive actions exist in this phase's scope. |

Accent reserved for: sidebar row selection highlight only (pre-existing Phase 51 behavior, not touched by this phase).

**Status semantic-color set (this phase's actual "color contract" — a 3-state status system, not the generic 60/30/10 split):**

| Status | Color | Glyph | Precedent |
|--------|-------|-------|-----------|
| granted | `.green` | `checkmark.circle.fill` | Matches `"✓ License activated".foregroundStyle(.green)` |
| denied | `.red` | `xmark.circle.fill` | Matches `"✗ That key wasn't recognized.".foregroundStyle(.red)` |
| notYetAsked (incl. Input Monitoring's best-effort `.unknown`) | `.secondary` | `questionmark.circle` | Matches the neutral-pending convention used by `"⚠ Server not reachable.".foregroundStyle(.secondary)` |

## Layout & Component Notes (native-app supplement to the template)

Since this phase has no visual reference/mockup and extends an established native pattern, the exact `View` shape to build is specified here for the executor:

**Permissions section** (`Islet/SettingsView.swift`, new `permissionsSection` computed property, mirrors `diagnosticsSection`'s minimal `ScrollView { Form { Section(...) { ... } }.padding(20) }` shape):
- `ScrollView(.vertical) { Form { ... }.padding(20) }` — same wrapper every other section uses.
- Top: a plain `LabeledContent` or `Text` row reading "{grantedCount} of 5 granted" (15pt semibold), NOT inside its own bordered card — sits directly above the row list per D-11 ("sits ABOVE the always-visible per-row list").
- Below: 5 rows, each an `HStack`: icon (left, `.secondary` tint unless status overrides it) + `VStack(alignment: .leading, spacing: 4)` { label (13pt regular) + reason (12pt regular, `.secondary`) } + `Spacer()` + status glyph+text (right, colored per the status table above). Row spacing: 8pt vertical between rows (`VStack(spacing: 8)` wrapping all 5 rows), each row wrapped in a tappable `Button(action:) { ... }.buttonStyle(.plain)` (mirrors the sidebar's own `Button` + `.buttonStyle(.plain)` precedent, not a `List` — `List(selection:)` is documented broken in this codebase per Phase 51's own comment at `SettingsView.swift:140-145`).
- `granted` rows: button action is a no-op (inert tap, Claude's discretion exercised — matches the "no confirmation, nothing happens" simplest option since there's nothing to do).

**About section addition** (D-09): append a new `Section("Onboarding")` (or a `LabeledContent`-style row inside the existing Form, exact grouping at implementation discretion) below the existing "Credits" section, containing exactly one `Button("Replay Onboarding") { ... }` — no icon, no subtitle, matches the plain-button style of `buyNowButton`.

**Replay-mode close button** (D-12): added only to the onboarding carousel's chrome when `isOnboardingActive` was entered via `replayOnboarding()` (not real first-launch) — a small icon-only button, top-trailing corner of the carousel card, using the same 44×44pt-minimum tap-target convention Apple/this app already applies to icon-only controls elsewhere (e.g. transport buttons in Now Playing).
