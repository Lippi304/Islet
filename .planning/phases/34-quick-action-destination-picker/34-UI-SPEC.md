# Phase 34 — UI Design Contract

> Visual and interaction contract for the Quick Action Destination Picker (TRAY-02, TRAY-03, TRAY-04). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform note:** Islet is a native macOS SwiftUI app, not a web project — the shadcn/web-registry gate does not apply (no `components.json`, no Tailwind, no npm-based component library). All values below are SwiftUI points (pt), following this project's established hand-tuned-named-constant convention (`expandedSize`, `traySize`, `weatherMediumContentHeight`, etc.), not a strict CSS token system. No visual reference screenshot of the picker exists (34-CONTEXT.md `canonical_refs`: Droppy's only "Quick Action" capture is a *Settings config* screenshot, not the runtime picker) — every dimension below is this spec's own design, built from the project's own existing component vocabulary (`ShelfItemView`, `chipButton`, `navCircleButton`, `trayEmptyState`), not traced from an image.

## Spacing Scale

Reuses this project's existing point-value convention (named constants, not a strict 4pt grid). Every value this phase *introduces* stays on the standard 4pt scale (4, 8, 16, 24, 32, 48, 64) — no new odd values, matching Phase 33's precedent of grandfathering only the one pre-existing non-4pt constant (`cameraClearance`).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Preview block's icon→filename gap (mirrors `ShelfItemView`'s `VStack(spacing: 2)` — this phase uses 4 not 2 since the preview sits alone, not in a dense scrolling row) |
| sm | 8pt | Button chip's internal icon→label gap (`VStack(spacing: 8)`); button chip's internal vertical padding (`.padding(.vertical, 8)`) — **reused verbatim from `chipButton`'s own existing `.padding(.vertical, 8)`** (`NotchPillView.swift:1310`), not a new value |
| md | 16pt | Outer content horizontal inset (`.padding(.horizontal, 16)`, matches every existing full-view); section gap between the file preview and the 3-button row; gap between the 3 button chips (`HStack(spacing: 16)`) |
| — | 42pt | `Self.cameraClearance` — existing fixed constant, top-of-card clearance, unchanged (grandfathered, pre-existing, same precedent as Phase 33) |

Exceptions: none. Every value newly introduced by this phase (4pt, 8pt, 16pt) is a standard 4pt-scale multiple; the only non-4pt number in this spec is the pre-existing `cameraClearance` (42pt), grandfathered exactly as Phase 33's approved UI-SPEC grandfathered it. Treat the 8pt gap/padding values and the 188pt content height (see Layout Contract) as starting values, on-device-tunable like every prior phase's own geometry (Tray took 3 rounds, Weather took 6) — tuning the *number* is expected; introducing a *non-4pt* number is not.

## Typography

Reuses two of the project's already-established sizes (9pt, 11pt) — **no new size is introduced**, staying inside the project's existing type scale rather than adding a fifth value. Both roles below are single-line and truncate rather than wrap — no custom line height is needed for either (default single-line metrics apply, same as `ShelfItemView`'s existing caption).

| Role | Size | Weight | Usage |
|------|------|--------|-------|
| Filename caption | 9pt | regular (400) | Single-file preview filename — **reused verbatim from `ShelfItemView`** (D-02: "reuses the existing `ShelfItemView` icon+filename convention"), same `.foregroundStyle(.secondary)`, `.lineLimit(1)`, `.truncationMode(.middle)` |
| Multi-file count | 11pt | regular (400) | Multi-file preview's "N files" label, `.foregroundStyle(.secondary)`, single line, no truncation needed (short fixed-format string) |
| Button label | 11pt | semibold (600) | "Drop" / "AirDrop" / "Mail" chip labels, `.foregroundStyle(.white)` — dims to `.white.opacity(0.3)` when disabled (D-09 fallback, see Layout Contract); single line, fixed short strings, no truncation needed |

**2-weight budget (unchanged):** regular (400) + semibold (600).

**Icon point sizes (SF Symbol, not a text role):**
- Single-file preview icon: 40×40pt frame, `NSWorkspace.shared.icon(forFile:)` — identical sizing to `ShelfItemView`'s existing icon (Phase 32 / TRAY-05 convention), not a new size.
- Multi-file preview icon (generic, D-02 "file count + generic icon for multiple files"): same 40×40pt frame, SF Symbol `doc.on.doc.fill` at `.font(.system(size: 32))` (centered in the frame — smaller than the frame so the multi-file glyph doesn't visually outweigh the single-file `NSWorkspace` icons it sits alongside across drop sessions).
- Button icon: 22pt SF Symbol, one size larger than `navCircleButton`'s 14pt (these are primary destination choices, not secondary nav chevrons) but smaller than `trayEmptyState`'s 28pt empty-icon (this isn't an empty/degraded state).

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Existing Phase 25 black→transparent vertical gradient shell material (`islandFill`) | Card background — unchanged, inherited from `blobShape()`, not touched by this phase |
| Secondary (30%) | `.secondary` (system semantic grey) | Multi-file count label, filename caption (reused `ShelfItemView` styling) |
| Accent (10%) | none introduced | See "Accent reserved for" below |
| Destructive | not applicable | D-07 (locked): dismissing without a choice silently discards the file(s) — no confirmation dialog, no red/warning-colored UI element, matching this project's existing no-confirmation immediate-delete convention (Tray's per-item/clear-all trash) |

**Accent reserved for:** This phase introduces **no new reserved accent tint**. Matches the established per-element pattern (Phase 6/25/27/33: `nowPlayingAccent`/`chargingAccent`/`deviceAccent` are reserved for elements tied to a *live, in-progress* activity). The picker's 3 destination buttons are equal-weight, non-live choices — all render in the same neutral `Color.white.opacity(0.12)` chip fill as `chipButton`'s existing style, no button is visually "primary" over the others.

**Disabled-button state (D-09 fallback only — AirDrop and/or Mail, if the phase's spike finds no working approach):** chip fill drops from `Color.white.opacity(0.12)` to `Color.white.opacity(0.06)`; icon + label opacity drops to `0.3` (dimmer than every other "muted but still legible" opacity already in the app — `deviceWings`' disconnected-icon `0.5`, `wings`' non-charging bolt `0.6` — so a disabled destination unambiguously reads as inert, not just quieter); `Button`'s own `.disabled(true)` (not just a visual dim) so it doesn't fire `canPerform`/`perform` at all. Drop is never disabled — TRAY-03 carries no risk per D-09.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — 3 equal-weight destination buttons, no single primary action: **"Drop"**, **"AirDrop"**, **"Mail"** (button labels, exact strings — Title Case, one word each, matching the switcher tab / `chipButton` label convention already used project-wide) |
| Empty state | Not applicable — the picker only ever renders in response to a real drop event, so it always has ≥1 pending file by construction; there is no "picker with nothing to show" state to design for |
| Error state | No dedicated error text/toast (this view has no existing toast infrastructure to reuse and Phase 37's drop-session chip is future scope). Per RESEARCH.md Pitfall 2 ("never a silent no-op"): if `canPerform(withItems:)` returns `false`, or the share attempt fails, the picker falls back to **the exact same discard-and-dismiss path as D-07** — the tap still produces a visible state change (picker closes) rather than doing nothing, satisfying the "not a silent no-op" requirement without inventing new error copy |
| Destructive confirmation | None (D-07, locked) — dismissing the picker (hover-away, D-06) or hitting the D-09 error-fallback path discards the pending file(s) immediately, no confirmation dialog, no "Are you sure?" step — matches Tray's existing immediate-delete convention |
| Single-file preview copy | The real filename via `item.filename`, truncated middle (`ShelfItemView` convention, reused verbatim, D-02) |
| Multi-file preview copy | **"{n} files"** (e.g. "12 files") — generic `doc.on.doc.fill` icon, one ONE picker/ONE decision for the whole batch (D-03, locked) |

## Layout & Interaction Contract

*(Supplementary section — this phase's real design surface is a new full-takeover presentation + its geometry wiring, not a token table alone. Executor: read this before touching code.)*

**Focal point:** the single-file preview (icon + filename) is the visual anchor at the top — it answers "what am I about to send" before the user looks at the 3 buttons below. The button row is secondary/equal-weight (no button reads as "primary"), matching the "Accent reserved for" note above: nothing on this screen competes with the preview for attention.

### 1. Presentation shape and switcher-row visibility (open decision, resolved here)

34-CONTEXT.md's `code_context` flagged this explicitly as unresolved: *"the picker case needs an explicit decision here too (does the switcher stay visible during the picker, or is it hidden like Charging/Device wings are?) — left to planning since it wasn't asked directly."*

**Decision: the switcher row is HIDDEN while the picker is showing** (`showsSwitcherRow(for:)` returns `false` for the new case). Rationale: D-01 explicitly frames the picker as behaviorally analogous to the Charging/Device wings splash — "a full-takeover presentation... replacing whatever tab was showing... the same shape as the existing Charging/Device wings splash but interactive instead of auto-dismissing" — and the project's own established precedent is that transient full-takeover splashes (Charging, Device) never show the switcher row (`showsSwitcherRow(for:)`'s `default: return false` branch already covers them). Showing tab navigation during a modal-feeling destination decision would also undercut the "you must choose before anything else happens" framing D-07 establishes. This is the researcher's recommendation, not a locked product decision from CONTEXT.md — flag to the user if on-device UAT suggests otherwise.

Note: "same shape as... wings splash" is a **behavioral** description (full-takeover, not an overlay/sheet), not a literal pixel-dimension match — the 290×32pt `wingsSize` strip has no room for a preview + 3 labeled buttons. The picker uses the same downward-`blobShape` family as `trayFullView`/`homeEmptyState` (topCornerRadius 24, bottomCornerRadius 32), sized via its own new height override — see below.

### 2. New geometry constant

```swift
// New — no existing constant fits (expandedSize.height=144 is too short for preview+3 buttons;
// switcherContentHeight=196 is for switcher-row presentations, which this is NOT — see decision above).
// Worked math (all spacing values on the standard 4pt scale — see Spacing Scale above):
//   cameraClearance(42) + preview(icon 40 + gap 4 + caption ~11 ≈ 55) + sectionGap(16)
//   + buttonChip(icon 22 + gap 8 + label ~13 + vPadding 2×8 ≈ 59) + bottomInset(16) = 188
static let quickActionPickerContentHeight: CGFloat = 188
```
Reuses `Self.expandedSize.width` (420pt) unchanged — no new width constant needed. Content width available for the 3-button row: `420 − 2×16 = 388pt`; at `HStack(spacing: 16)` for 3 flexible-width chips, each gets `(388 − 2×16) / 3 ≈ 118.7pt` — ample room for a 22pt icon + 11pt label stacked vertically. Treat 188pt as a starting value, confirm/tune on first on-device build (same convention as every prior phase's own geometry number).

### 3. New view structure (`quickActionPickerView`, mirrors `trayFullView`'s call shape)

```swift
private func quickActionPickerView(_ pending: PendingDrop) -> some View {
    blobShape(topCornerRadius: 24, bottomCornerRadius: 32, alignment: .top,
              height: Self.quickActionPickerContentHeight, shelfItems: [],
              shelfVisible: false, showSwitcher: false) {
        VStack(spacing: 16) {
            quickActionPreview(pending)      // ShelfItemView-style icon+filename, or generic+count
            quickActionButtonRow(pending)    // Drop / AirDrop / Mail chips
        }
        .padding(.top, Self.cameraClearance)   // camera/notch clearance — matches every other full-view
    }
}
```

### 4. Preview block (D-02)

Single file: reuse `ShelfItemView`'s exact icon+filename visual convention (40×40 `NSWorkspace` icon, 9pt truncated-middle caption) — either render a lightweight non-interactive twin (no `onTap`/`onDelete`/`onDrag`, since the file isn't in the shelf yet) or a purpose-built `VStack(spacing: 4)` matching its look; do not reuse `ShelfItemView` itself with no-op closures, since its `.onDrag` would offer a drag source for a file not yet staged anywhere. Multi-file: same `VStack(spacing: 4)` shape, generic icon (see Typography) + "{n} files" caption at 11pt.

### 5. Button row (D-01 "match the spirit of Droppy's icon+label buttons")

```swift
private func quickActionButtonRow(_ pending: PendingDrop) -> some View {
    HStack(spacing: 16) {
        quickActionButton(icon: "tray.and.arrow.down.fill", label: "Drop", enabled: true, action: onDropChosen)
        quickActionButton(icon: "personalhotspot", label: "AirDrop", enabled: airDropAvailable, action: onAirDropChosen)
        quickActionButton(icon: "envelope.fill", label: "Mail", enabled: mailAvailable, action: onMailChosen)
    }
}

private func quickActionButton(icon: String, label: String, enabled: Bool, action: @escaping () -> Void) -> some View {
    Button(action: action) {
        VStack(spacing: 8) {
            Image(systemName: icon)
                .font(.system(size: 22))
            Text(label)
                .font(.system(size: 11, weight: .semibold, design: .rounded))
        }
        .foregroundStyle(.white.opacity(enabled ? 1.0 : 0.3))   // D-09 disabled dim
        .frame(maxWidth: .infinity)
        .padding(.vertical, 8)   // reused verbatim from chipButton's own .padding(.vertical, 8)
        .background(
            RoundedRectangle(cornerRadius: 8, style: .continuous)
                .fill(Color.white.opacity(enabled ? 0.12 : 0.06))   // mirrors chipButton's existing fill convention
        )
    }
    .buttonStyle(.plain)
    .disabled(!enabled)
}
```
Exact SF Symbols are Claude's discretion per CONTEXT.md — `tray.and.arrow.down.fill` (Drop, echoes the switcher's `tray.fill`), `personalhotspot` (AirDrop — SF Symbols has no literal "airdrop" glyph; this concentric-rings symbol is the closest visual match to the system AirDrop icon), `envelope.fill` (Mail) are the recommended defaults — swap only if an on-device UAT round flags one as unclear, same discretion Weather's icon choices were given.

`airDropAvailable`/`mailAvailable` are `true` unless the phase's own required spike (RESEARCH.md Open Question 1) finds no working invocation path — see D-09.

### 6. Geometry three-site rule (CR-01/CR-02 discipline — mandatory, not optional)

Per this project's own repeated failure mode (Phase 32/33's own explicit warnings, RESEARCH.md Pitfall 4): `quickActionPickerContentHeight` MUST be wired into all three of, in the SAME commit:
1. `blobShape()`'s `height:` override at the `quickActionPickerView` call site (above).
2. `positionAndShow`'s panel-frame union in `NotchWindowController.swift` (new `quickActionPickerFrame` member, mirroring `trayFrame`/`weatherExpandedFrame`, sized `CGSize(width: expandedSize.width, height: quickActionPickerContentHeight)` — no switcher-row addend since showSwitcher is false here).
3. `visibleContentZone()`'s matching new branch for the picker's `IslandPresentation` case.

Then verify via the mandatory on-device hover→expand→move-down click-through trace (ROADMAP success criterion 5 already names this explicitly for this phase).

### 7. Pending-drop state ownership (D-05, RESEARCH.md Pitfall 5)

Not a view-layer concern, but a hard constraint on where state lives that affects what the view can assume: the pending-drop payload must live on the CONTROLLER (`NotchWindowController`), not inside `IslandResolver`'s pure return value — `IslandPresentation` is a fresh `Equatable` value on every `resolve()` call with no memory across calls. The view (`quickActionPickerView`) simply renders whatever `pending: PendingDrop` it's handed; it does not need to know how that payload survived a Charging/Device interruption.

### 8. What this phase does NOT touch

No color, material, or shell-chrome change (reuses `islandFill` verbatim). No new font, no new type size beyond the 2 already-reused (9pt/11pt). `ShelfItemView` itself is unchanged — the preview block is a new, separate lightweight view (see §4), not a modification to the shared shelf-row component.
