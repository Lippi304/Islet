# Phase 34 — UI Design Contract

> Visual and interaction contract for the Quick Action Destination Picker (TRAY-02, TRAY-03, TRAY-04). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**REVISION 2 (2026-07-15, post-UAT):** The original click-based picker (preview block + tap-to-select, 188pt height) was rejected on real hardware — see `34-02-SUMMARY.md` §"Outcome: CHANGES REQUESTED" and `34-CONTEXT.md` D-10–D-15. This revision replaces the interaction model wholesale: the picker now appears DURING the drag (not after release), shows ONLY the 3 destination buttons (no file preview, D-14), and selection happens by releasing the mouse directly on top of a button (D-12), with the button under the pointer live-highlighting as a real drop target (D-11). Everything NOT called out below (shell chrome, `islandFill` material, corner radii, 420pt width, button-label typography, color palette, CR-01 three-site geometry discipline, copywriting for CTA/error/destructive states) is unchanged from Revision 1 and restated here for completeness, not re-derived.

**Platform note:** Islet is a native macOS SwiftUI app, not a web project — the shadcn/web-registry gate does not apply (no `components.json`, no Tailwind, no npm-based component library). All values below are SwiftUI points (pt), following this project's established hand-tuned-named-constant convention (`expandedSize`, `traySize`, `weatherMediumContentHeight`, etc.), not a strict CSS token system. No visual reference screenshot of the picker exists (34-CONTEXT.md `canonical_refs`: Droppy's only "Quick Action" capture is a *Settings config* screenshot, not the runtime picker) — every dimension below is this spec's own design, built from the project's own existing component vocabulary (`chipButton`, `navCircleButton`), not traced from an image.

## Spacing Scale

Reuses this project's existing point-value convention (named constants, not a strict 4pt grid). Every value this phase *introduces* stays on the standard 4pt scale (4, 8, 16, 24, 32, 48, 64) — no new odd values, matching Phase 33's precedent of grandfathering only the one pre-existing non-4pt constant (`cameraClearance`).

| Token | Value | Usage |
|-------|-------|-------|
| sm | 8pt | Button chip's internal icon→label gap (`VStack(spacing: 8)`); button chip's internal vertical padding (`.padding(.vertical, 8)`) — reused verbatim from `chipButton`'s own existing `.padding(.vertical, 8)` (`NotchPillView.swift:1310`), not a new value |
| md | 16pt | Outer content horizontal inset (`.padding(.horizontal, 16)`, matches every existing full-view); gap between the 3 button chips (`HStack(spacing: 16)`); bottom inset below the button row to the card's bottom edge |
| — | 42pt | `Self.cameraClearance` — existing fixed constant, top-of-card clearance, unchanged (grandfathered, pre-existing, same precedent as Phase 33) |

**Revision 2 change:** the `xs` (4pt, preview icon→filename gap) token and the "section gap between preview and button row" usage are REMOVED — both existed only to serve the now-deleted preview block (D-14). No replacement value needed; the card is now a single section (button row only).

Exceptions: none. Every value newly introduced by this phase (8pt, 16pt) is a standard 4pt-scale multiple; the only non-4pt number in this spec is the pre-existing `cameraClearance` (42pt), grandfathered exactly as Phase 33's approved UI-SPEC grandfathered it. Treat the button gap/padding values and the new 117pt content height (see Layout Contract) as starting values, on-device-tunable like every prior phase's own geometry (Tray took 3 rounds, Weather took 6) — tuning the *number* is expected; introducing a *non-4pt* number is not.

## Typography

Reuses one of the project's already-established sizes (11pt) — **no new size is introduced**.

| Role | Size | Weight | Usage |
|------|------|--------|-------|
| Button label | 11pt | semibold (600) | "Drop" / "AirDrop" / "Mail" chip labels, `.foregroundStyle(.white)` — dims to `.white.opacity(0.3)` when disabled (D-09 fallback, see Layout Contract); single line, fixed short strings, no truncation needed |

**Revision 2 change:** the "Filename caption" (9pt) and "Multi-file count" (11pt) roles are REMOVED — both existed only for the now-deleted preview block (D-14). Typography for this phase is now a single role.

**2-weight budget (unchanged):** regular (400) + semibold (600) — the picker itself now only uses semibold; regular remains reserved for other views' body/secondary text project-wide, not introduced here.

**Icon point sizes (SF Symbol, not a text role):**
- Button icon: 22pt SF Symbol, one size larger than `navCircleButton`'s 14pt (these are primary destination choices, not secondary nav chevrons) but smaller than `trayEmptyState`'s 28pt empty-icon (this isn't an empty/degraded state). **Revision 2 fix (Pitfall 9, UAT-flagged):** the icon MUST render inside a fixed `.frame(width: 22, height: 22)` box (centered), not left to its own intrinsic SF Symbol bounds — `tray.and.arrow.down.fill` (Drop) and `personalhotspot`/`envelope.fill` (AirDrop/Mail) have visibly different natural glyph heights at the identical `.font(.system(size: 22))`, which is exactly the "Drop button renders at a different height than AirDrop/Mail" bug UAT reported. The fixed frame normalizes all 3 buttons to identical layout height regardless of each glyph's own bounds.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Existing Phase 25 black→transparent vertical gradient shell material (`islandFill`) | Card background — unchanged, inherited from `blobShape()`, not touched by this phase |
| Secondary (30%) | `.secondary` (system semantic grey) | Not used by this phase's own content anymore (Revision 2: the preview block that used it is removed) — token stays documented for consistency with the rest of the app, no active usage inside the picker |
| Accent (10%) | none introduced | See "Accent reserved for" below |
| Destructive | not applicable | D-07 (locked): dismissing without a choice silently discards the file(s) — no confirmation dialog, no red/warning-colored UI element, matching this project's existing no-confirmation immediate-delete convention (Tray's per-item/clear-all trash) |

**Accent reserved for:** This phase introduces **no new reserved accent tint** — matches the established per-element pattern (Phase 6/25/27/33: `nowPlayingAccent`/`chargingAccent`/`deviceAccent` are reserved for elements tied to a *live, in-progress* activity). The 3 destination buttons stay equal-weight — no button reads as visually "primary" over the others, even the live drag-hover highlight (below) uses the SAME neutral white channel at a higher opacity, not a new hue.

**Idle vs. drag-hover vs. disabled button fill (Revision 2 — D-11, new):**

| State | Chip fill (`Color.white.opacity(...)`) | Scale | Icon+label opacity |
|-------|------------------------------------------|-------|----------------------|
| Idle (not hovered, enabled) | 0.12 (unchanged from Revision 1's `chipButton`-derived fill) | 1.0 | 1.0 |
| **Drag-hovered (D-11, new)** | **0.22** — one clear, visible step brighter, still the same neutral white, no hue shift | **1.04** — subtle scale-up, matches this project's existing hover-scale convention (`NotchPillView.swift:564`) | 1.0 |
| Disabled (D-09 fallback only) | 0.06 | 1.0 (no hover response — `.disabled` buttons don't hit-test) | 0.3 |

The 0.12 → 0.22 fill step and the 1.04 scale are BOTH required together (per D-11's "brighter fill and/or slight scale" — this spec locks it to "and", both signals reinforcing the same drop-target affordance, not an either/or). Treat these as starting values, tunable on first on-device drag trial like every other geometry/opacity number in this project.

**Disabled-button state (D-09 fallback only — AirDrop and/or Mail, if the phase's spike finds no working approach):** chip fill drops to `Color.white.opacity(0.06)`; icon + label opacity drops to `0.3` (dimmer than every other "muted but still legible" opacity already in the app — `deviceWings`' disconnected-icon `0.5`, `wings`' non-charging bolt `0.6` — so a disabled destination unambiguously reads as inert, not just quieter); the button's own disabled state does not hit-test as a valid drop target (a release over a disabled button behaves the same as D-13's "released inside the card but not on a button" — discard, not selection). Drop is never disabled — TRAY-03 carries no risk per D-09.

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — 3 equal-weight destination buttons, no single primary action: **"Drop"**, **"AirDrop"**, **"Mail"** (button labels, exact strings — Title Case, one word each, matching the switcher tab / `chipButton` label convention already used project-wide) |

## Layout & Interaction Contract

*(Supplementary section — this phase's real design surface is a new full-takeover presentation + its geometry wiring + live drag-target hit-testing, not a token table alone. Executor: read this before touching code.)*

**Focal point:** with the preview removed, the 3-button row IS the entire content — there is no longer a separate "what am I about to send" anchor above it. The buttons stay equal-weight in their IDLE state (no button reads as primary); the only visual differentiation is the transient, pointer-driven drag-hover highlight (Color table above), which exists purely as real-time drop-target feedback, not a persistent "primary action" styling.

### 0. Trigger timing (D-10, new — changes WHEN this view appears)

**Revision 1** showed the picker only after `.leftMouseUp` (release). **Revision 2 (D-10, locked):** the picker must appear at the exact same edge that already auto-expands the island today — the `dragEntered` transition inside `recheckDragAcceptRegion()` — so the 3-button picker is visible for the ENTIRE hover-before-release window, not just after the drop lands. This is a controller/state-timing change (see 34-RESEARCH.md Pattern 4), but it directly determines what the user sees during a drag: from the moment the dragged file enters the island's accept geometry, the card described below is what renders — never Now Playing, never any other tab's content, for as long as `pendingDrop != nil`.

### 1. Presentation shape and switcher-row visibility (unchanged from Revision 1)

**Decision: the switcher row is HIDDEN while the picker is showing** (`showsSwitcherRow(for:)` returns `false` for the new case). Rationale: D-01 frames the picker as behaviorally analogous to the Charging/Device wings splash — a full-takeover presentation, not an overlay/sheet — and the project's own established precedent is that transient full-takeover splashes never show the switcher row. This still applies unchanged in Revision 2; only the trigger timing and content changed, not the takeover/switcher-visibility behavior.

The picker uses the same downward-`blobShape` family as `trayFullView`/`homeEmptyState` (topCornerRadius 24, bottomCornerRadius 32), sized via its own new (now shorter) height override — see below.

### 2. New geometry constant (Revision 2 — recomputed, shrinks from 188pt to 117pt)

```swift
// Revision 2 (D-15) — the preview block (D-14) and its 16pt section gap are REMOVED entirely.
// Only cameraClearance + the button row + a bottom inset remain:
//   cameraClearance(42) + buttonChip(icon 22 + gap 8 + label ~13 + vPadding 2×8 ≈ 59) + bottomInset(16) = 117
static let quickActionPickerContentHeight: CGFloat = 117
```
Reuses `Self.expandedSize.width` (420pt) unchanged — no new width constant needed. Content width available for the 3-button row: `420 − 2×16 = 388pt`; at `HStack(spacing: 16)` for 3 flexible-width chips, each gets `(388 − 2×16) / 3 ≈ 118.7pt` — ample room for a fixed 22×22pt icon + 11pt label stacked vertically. Treat 117pt as a starting value, confirm/tune on the first on-device drag trial (same convention as every prior phase's own geometry number — this project's own research doc (34-RESEARCH.md Assumption A4) flags the exact same starting-value caveat).

### 3. New view structure (`quickActionPickerView`, mirrors `trayFullView`'s call shape)

```swift
private func quickActionPickerView(_ pending: PendingDrop) -> some View {
    blobShape(topCornerRadius: 24, bottomCornerRadius: 32, alignment: .top,
              height: Self.quickActionPickerContentHeight, shelfItems: [],
              shelfVisible: false, showSwitcher: false) {
        quickActionButtonRow(pending)    // Drop / AirDrop / Mail chips — ONLY content (D-14: no preview)
            .padding(.top, Self.cameraClearance)   // camera/notch clearance — matches every other full-view
    }
}
```
**Revision 2 change:** the outer `VStack(spacing: 16) { quickActionPreview(pending); quickActionButtonRow(pending) }` wrapper is gone — `quickActionButtonRow` is now the direct (and only) child under the camera-clearance padding. `quickActionPreview` itself (and its helper, `ShelfItemView`-style icon+filename twin) should be deleted from the codebase, not left as dead code.

### 4. Preview block — REMOVED (D-14)

The file preview (icon + filename for a single file, or generic-icon + "{n} files" for multiple) is deleted entirely, uniformly for both single-file and multi-file drops — no exception, no count badge anywhere in the picker. This was the #1 UAT-flagged visual-polish request. `quickActionPreview(_:)` and any purpose-built preview subview from Revision 1 should be removed from `NotchPillView.swift`, not merely unused.

### 5. Button row + live drag-target interaction (Revision 2 — D-11/D-12/D-13, materially new)

```swift
private func quickActionButtonRow(_ pending: PendingDrop) -> some View {
    HStack(spacing: 16) {
        quickActionButton(icon: "tray.and.arrow.down.fill", label: "Drop", enabled: true,
                           isHovered: hoveredQuickActionButtonIndex == 0)
        quickActionButton(icon: "personalhotspot", label: "AirDrop", enabled: airDropAvailable,
                           isHovered: hoveredQuickActionButtonIndex == 1)
        quickActionButton(icon: "envelope.fill", label: "Mail", enabled: mailAvailable,
                           isHovered: hoveredQuickActionButtonIndex == 2)
    }
}

private func quickActionButton(icon: String, label: String, enabled: Bool, isHovered: Bool) -> some View {
    VStack(spacing: 8) {
        Image(systemName: icon)
            .font(.system(size: 22))
            .frame(width: 22, height: 22)   // Pitfall 9 fix — fixed icon box, all 3 buttons render identical height
        Text(label)
            .font(.system(size: 11, weight: .semibold, design: .rounded))
    }
    .foregroundStyle(.white.opacity(enabled ? 1.0 : 0.3))   // D-09 disabled dim
    .frame(maxWidth: .infinity)
    .padding(.vertical, 8)   // reused verbatim from chipButton's own .padding(.vertical, 8)
    .background(
        RoundedRectangle(cornerRadius: 8, style: .continuous)
            .fill(Color.white.opacity(enabled ? (isHovered ? 0.22 : 0.12) : 0.06)))   // D-11 hover step
    .scaleEffect(isHovered ? 1.04 : 1.0)   // D-11 slight scale
}
```

**What changed from Revision 1, and why (each maps to a locked CONTEXT.md decision):**

- **No `Button(action:)` wrapper, no tap handler as the primary selection path (D-12).** Selection now happens by the CONTROLLER hit-testing the release point (`NSEvent.mouseLocation` at `.leftMouseUp`) against 3 analytically-computed `CGRect`s and calling `handleQuickActionDrop/AirDrop/Mail()` directly — the view only RENDERS the hover state, it does not decide selection. See 34-RESEARCH.md Pattern 3/Pattern 5 for the exact geometry math (`computeQuickActionButtonFrames`) — this is controller-owned, pure-arithmetic geometry, deliberately NOT a `GeometryReader`/`PreferenceKey` pipeline (research's explicit recommendation: this codebase has zero existing `PreferenceKey` usage, and the 3-equal-column `HStack` layout is fully deterministic from constants the controller already owns, so no SwiftUI round-trip is needed). Executor note (research's own open item, not locked): keeping an ordinary `Button(action:)` tap as a SECONDARY fallback for a user who prefers click-after-hover is a reasonable UX safety net and does not conflict with D-12 — left as an executor/planner discretion call, not required by this spec.
- **`isHovered` is a read-only view input, sourced from a controller-published `hoveredQuickActionButtonIndex: Int?`** — updated ONLY when the hit-tested index actually changes (34-RESEARCH.md Pitfall 8: publishing on every `.leftMouseDragged` tick unconditionally would re-render the view dozens of times/second for no visual change).
- **D-13 (release not on any button):** discards the pending file(s) — same rule as D-07, no safety net, no default destination. Also applies to a release over a `.disabled` button (AirDrop/Mail under the D-09 fallback) — a disabled chip is never a valid drop target.
- **D-13b + Pitfall 6 (drag exits the island's geometry before release):** collapses the picker with no destination chosen. **Executor must implement the full fix, not just the visual behavior** — 34-RESEARCH.md Pitfall 6 found that the pre-existing exit branch in `recheckDragAcceptRegion()` only clears `isDragApproaching`, not `pendingDrop`; once D-10 moves `pendingDrop`'s population earlier (to `dragEntered`), that exit branch MUST also call `discardPendingDrop()` (+ re-render), or the picker visually never disappears and a session-copied temp file leaks on disk. This is a functional requirement this UI-SPEC surfaces because it directly determines what's on screen after a drag-out — not a pure backend concern.

### 6. Geometry three-site rule (CR-01/CR-02 discipline — mandatory, not optional)

Per this project's own repeated failure mode (Phase 32/33's own explicit warnings, RESEARCH.md Pitfall 4): `quickActionPickerContentHeight` (now 117pt, Revision 2) MUST be wired into all three of, in the SAME commit:
1. `blobShape()`'s `height:` override at the `quickActionPickerView` call site (above).
2. `positionAndShow`'s panel-frame union in `NotchWindowController.swift` (the `quickActionPickerFrame` member, mirroring `trayFrame`/`weatherExpandedFrame`, sized `CGSize(width: expandedSize.width, height: quickActionPickerContentHeight)` — no switcher-row addend since `showSwitcher` is false here). This is also where `computeQuickActionButtonFrames` (34-RESEARCH.md Pattern 3) should be computed and stored, right alongside `quickActionPickerFrame`.
3. `visibleContentZone()`'s matching branch for the picker's `IslandPresentation` case.

Then verify via the mandatory on-device hover→expand→move-down click-through trace (ROADMAP success criterion 5 already names this explicitly for this phase) — Revision 2 adds a SECOND required trace: drag a file in (picker shows), move the pointer between the 3 buttons WITHOUT releasing (confirm each highlights individually per §5), drag the pointer back out without releasing (confirm the picker actually disappears per D-13b/Pitfall 6), then re-enter and confirm the picker resumes cleanly.

### 7. Pending-drop state ownership (D-05, unchanged)

Not a view-layer concern, but a hard constraint on where state lives that affects what the view can assume: the pending-drop payload must live on the CONTROLLER (`NotchWindowController`), not inside `IslandResolver`'s pure return value. The view (`quickActionPickerView`) simply renders whatever `pending: PendingDrop` it's handed, plus the controller-published `hoveredQuickActionButtonIndex`; it does not need to know how either survived a Charging/Device interruption (D-04/D-05, unchanged).

### 8. What this phase does NOT touch

No color, material, or shell-chrome change (reuses `islandFill` verbatim). No new font, no new type size. `ShelfItemView`/`chipButton` themselves are unchanged. Icon choice for AirDrop stays `personalhotspot` (Claude's discretion, re-checked this revision per CONTEXT.md's "may need a closer match" note — SF Symbols has no literal "airdrop" glyph as of this SDK; `personalhotspot`'s concentric-arcs-over-a-dot shape remains the closest available visual match to the system AirDrop icon, no better candidate found — revisit only if a future on-device UAT round flags it again).
