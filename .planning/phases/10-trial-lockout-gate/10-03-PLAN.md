<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DEBUG build ↔ Release build | The DEBUG-only secondary status item and its 3 actions must never exist in a shipped Release binary |
| Menu-bar click ↔ AppKit event routing | `NSStatusItem.menu` and `button.action` are mutually exclusive at the AppKit level — a naive implementation of D-05 silently no-ops instead of erroring, so this is a correctness risk, not just a security one |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03 | Elevation of Privilege | `AppDelegate` DEBUG-only status item + actions (writer side) | mitigate | `debugStatusItem`, `setupDebugMenu()`, and all 3 debug `@objc` actions are compiled only under `#if DEBUG`; verified via a Release-configuration build in this plan's wave-merge `<verification>` step (pairs with the reader-side gate already verified in Plan 01/`LicenseState.status`) (D-08) |
| T-10-05 | (UX-adjacent, not classic STRIDE) | Locked-state menu-bar click routing (D-05) | mitigate | `applyMenuBarClickRouting` deterministically sets ONLY one of `statusItem.menu` / `button.action` at a time (never both), re-applied via a UserDefaults observer whenever license state changes — no dual-firing / stale-menu risk (Pitfall 3). This routing swap is a discrete assignment affecting only the *next* click, never an ongoing interaction, so it needs no D-13 idle-state guard — that guard (mid-hover/mid-expansion deferral for the island itself) is engineered in Plan 10-02's `updateVisibility()`/`pendingLockoutHide`, not duplicated here |
</threat_model>
