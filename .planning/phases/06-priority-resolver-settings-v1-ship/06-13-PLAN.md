<objective>
Fix two real, still-unfixed logic defects in `Islet/Notch/NotchWindowController.swift` that were
independently confirmed by both `06-VERIFICATION.md` (the phase gap-closure verification pass) and
`06-REVIEW.md` (a fresh code review), both introduced by the 06-07 gap-closure fix itself:

1. **WR-1 — battery-poll identity desync.** `pendingDeviceAddresses: [String]` is a FIFO that tracks
   which connected device still owes a deferred post-connect battery poll. It can desync from
   `transientQueue`'s own `pending` list once a disconnect transient evicts a *different* pending entry
   via the queue's `maxDepth` bound (disconnects are never appended to the FIFO, only connects are, so
   the two lists silently diverge). `triggerDeviceBatteryRefreshIfPromoted()` then pops
   `pendingDeviceAddresses.first` and polls that address for whatever device is NOW the promoted head —
   which can be a completely different device. The live splash can end up showing device B's name/glyph
   with device A's battery percentage.
2. **WR-2 — over-eager dismiss-timer reset.** `flushTransients(_:)` unconditionally cancels and
   restarts the shared ~3s dismiss timer whenever ANY head remains after a removal — even when the
   surviving head was never touched by the removal at all. Toggling one activity category off (e.g.
   disabling Charging in Settings) while an unrelated category's splash (e.g. Device) is the standing
   head silently extends that unrelated splash's on-screen time by resetting its dismiss window to a
   fresh 3s.

Purpose: Close the one open COORD-01 gap from `06-VERIFICATION.md` — "activities coexist without
overlapping or glitching" is not yet fully true while these two defects stand, both of which are real
data/timing bugs (not cosmetic).
Output: `IslandResolver.swift` gains a small, pure, unit-tested identity-matching helper that WR-1's
fix is built on; `IslandResolverTests.swift` gains regression coverage proving both the WR-1 matching
logic and the WR-2 head-unchanged invariant that `flushTransients` now relies on;
`NotchWindowController.swift` is wired to use both fixes.
</objective>
