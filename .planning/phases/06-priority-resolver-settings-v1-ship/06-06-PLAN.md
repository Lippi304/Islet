<objective>
Close the two real, actionable gaps surfaced by 06-UAT.md (Tests 1 and 4), diagnosed in
`.planning/debug/charging-yield-width-jump.md` and `.planning/debug/battery-indicator-accent-not-tinted.md`.
Both root causes are confirmed with no further investigation needed — this plan applies the
two targeted fixes directly.

1. **Charging-yield width jump (minor, Test 1).** `scheduleActivityDismiss()` in
   `NotchWindowController.swift` mutates `chargingState.activity = nil` (via `syncActivityModels()`)
   BEFORE opening the `withAnimation(.spring)` block that animates `renderPresentation()`. This
   produces an un-animated SwiftUI commit sandwiched before the animated one, breaking
   `matchedGeometryEffect`'s frame interpolation on the charging-wings → media-wings yield-back —
   exactly the "widens, then symbols pop in" the user reported. Fix: move the mutation inside the
   animated block. A secondary/compounding factor — `positionAndShow(on:)` forcing an unconditional
   `panel.setFrame(_, display: true)` redisplay even when the frame hasn't changed — is also
   addressed defensively.

2. **Battery indicator accent (major, Test 4).** `BatteryIndicator` already supports an `accent:`
   parameter and the environment plumbing is correct end-to-end, but the charging wings' call site
   in `NotchPillView.swift` never forwards it — a plain oversight from when `BatteryIndicator` was
   wired up in 06-04. Fix: pass `accent: accent` at that one call site.

**Explicitly NOT touched by this plan:**
- The **device wings' battery indicator** (`NotchPillView.swift` `deviceTrailing`) — its
  green/amber/red-regardless-of-accent behavior is a documented, user-confirmed design decision
  (see the debug session and the code comment above the call site). Out of scope.
- The **fullscreen-enter flash** (cosmetic, Test 5). Confirmed via
  `.planning/debug/fullscreen-enter-flash.md` to be the SAME pre-existing Phase-2 (02-04)
  window-server compositing limitation, re-surfacing during Phase-6 UAT — not a new Phase-6
  regression, and there is no known application-layer fix (the compositor renders the panel onto
  the activating fullscreen Space before any reactive app-level signal can fire; a debounce was
  already tried in Phase 2 and reverted since there is nothing on our side to debounce). This gap
  is carried forward as a known, accepted limitation — the same status it held after Phase 2 — and
  must NOT be "closed" by a code change here. Record this explicitly in the plan's SUMMARY.

Purpose: Close UAT gaps 1 and 2 from `06-UAT.md` without touching the intentionally-scoped-out
device battery indicator or attempting an unfixable compositor-level flash.
Output: Two targeted diffs in `NotchWindowController.swift` and `NotchPillView.swift`; an on-device
human-verify checkpoint confirming both fixes and reconfirming the fullscreen-flash carry-over.
</objective>

  <name>Task 1: Fix charging-yield width jump — animate the model clear with the presentation switch</name>
  <files>Islet/Notch/NotchWindowController.swift</files>
  <action>
Per 06-UAT.md Gap 1 and `.planning/debug/charging-yield-width-jump.md` (root cause confirmed, no further
investigation needed):

(a) In `scheduleActivityDismiss()`'s `DispatchWorkItem` body (currently lines 613-629), move the
`self.syncActivityModels()` call OUT of its current position (immediately after
`_ = self.transientQueue.advance()`, before the `withAnimation` block opens) and INTO the
`withAnimation(.spring(response: self.springResponse, dampingFraction: self.springDamping))` closure,
as the first statement inside it — immediately before `self.renderPresentation()`. Both calls must now
execute inside the same animated transaction so the `@Published` model clear and the
`presentationState` switch commit together as one SwiftUI update, giving `matchedGeometryEffect` a
single valid animated frame to interpolate from (instead of an un-animated commit immediately followed
by an animated one). `_ = self.transientQueue.advance()` stays OUTSIDE the `withAnimation` block —
it is a pure value mutation with no SwiftUI publish, it does not need to be animated. Leave
`self.updateVisibility()` and the re-arm `if self.transientQueue.head != nil { self.scheduleActivityDismiss() }`
after the `withAnimation` block, unchanged.

(b) Address the secondary/compounding factor the debug session flagged: `positionAndShow(on:)`
(currently ~lines 403-441) calls `panel.setFrame(panelFrame, display: true)` unconditionally on
every invocation, even when `panelFrame` is unchanged — which it is on every same-screen activity
switch (per the debug session's ruled-out section, `panelFrame` is a fixed per-screen union of
`expandedFrame` and `wingsFrame` that never changes size across charging/media/device/idle
activities). `display: true` forces an immediate AppKit redisplay outside the SwiftUI/Core Animation
transaction on every transition, which can compound the interpolation issue from (a). Wrap the
existing `panel.setFrame(panelFrame, display: true)` line in `if panel.frame != panelFrame { ... }`
so the forced redisplay only fires when the panel's frame is actually changing (CGRect is Equatable,
so `!=` works directly). Use the existing local `panel` binding already in scope at that point in
`positionAndShow(on:)` — do not introduce a new panel lookup or change any other line in the function.

Run `xcodegen generate`; build the app target.
