<objective>
Fix three confirmed Now-Playing reliability bugs surfaced by a fresh multi-agent code review, all
living in the controller/glue layer that coordinates `NowPlayingMonitor`'s persistent stream with the
UI's hover and auto-dismiss timing:

1. **Launch-time health-check race** — `NowPlayingMonitor.runHealthCheck` spawns a SECOND, independent
   one-shot perl child purely to probe health, racing the persistent stream child. If the stream sets
   `isHealthy = true` from a real snapshot, the unrelated probe's own timeout can still fire afterward
   and silently overwrite it back to `false`, showing "unavailable" while media is actively playing.
2. **Hover does not pause the paused-media 15s auto-dismiss** — `handleHoverEnter`/`handleHoverExit`
   only cancel/reschedule the CHARGING `dismissWorkItem`; `mediaDismissWorkItem` (the D-06 ~15s paused-
   media linger) is never touched by hover, so a user hovering the expanded transport controls can have
   them vanish mid-interaction — unlike the charging splash, whose dismiss correctly IS hover-paused.
3. **No debounce on duplicate `.paused` emissions** — `handleNowPlaying`'s `.paused` case restarts the
   15s countdown on EVERY emission, unlike `handlePower`'s `shouldTriggerSplash` category-change gate.
   A documented artwork-latency re-emission of the same paused track can keep resetting the timer,
   sticking the paused glance on-screen indefinitely.

Purpose: Restore the D-06/D-12/D-13 timing and health-flag guarantees the Now-Playing quartet was
designed to provide, without touching the pure `nowPlayingPresentation(from:)` seam or
`NowPlayingMonitor`'s public surface — per `PowerSourceMonitor`'s sibling discipline (ONE source of
truth per property), the fix lives entirely at the controller call site that currently lets two
independent writers race.
Output: Three targeted fixes in `NotchWindowController.swift`.
</objective>
