<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Islet process <-> Music.app/Spotify.app via private `MRMediaRemoteSendCommand` | Islet sends an undocumented private command; the receiving app's response/state is outside Islet's control |
| Islet process <-> TCC/Automation subsystem <-> Music.app via Apple Events | First-ever Apple Events send in this codebase; crosses a new, previously-unused OS permission boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-49-01 | Elevation of Privilege | `com.apple.security.automation.apple-events` entitlement (Islet.entitlements) | accept | The entitlement is a genuine, unavoidable prerequisite for FAV-01's shipped write-back feature (Phase 50), not spike overreach — Phase 50 will scope the actual AppleScript target explicitly to Music.app/Spotify only, matching this project's own precedent of accepting the minimum entitlement a feature genuinely requires (e.g. `com.apple.security.cs.disable-library-validation` for MediaRemoteAdapter) |
| T-49-02 | Tampering | `spikeLikeCurrentTrack()` DEBUG hook mutates the user's real Music.app/Spotify.app "liked" state as a side effect of testing | accept | This is the explicit, intended test per Success Criterion #1 ("verify effect, not existence") — DEBUG-only, user-triggered via an explicit menu click (never automatic), and trivially reversible by un-liking the track manually afterward |
| T-49-03 | Information Disclosure | `NSLog("SPIKE ...")` lines may include track title/artist text in the system log | accept | DEBUG-only builds never ship (verified by Task 2's Release-build grep check); track metadata is not secret and is already displayed in Islet's own UI |
| T-49-04 | Information Disclosure | AppleScript error dict / TCC dialog text logged verbatim to console | accept | Local Debug-console-only output, no network transmission, no persistence beyond the current session's log — matches this project's existing `NSLog` diagnostic conventions elsewhere (e.g. `PowerSourceMonitor`, `BluetoothMonitor`) |

This plan ships zero production code (both hooks are `#if DEBUG`-gated and will be
removed/replaced once Phase 50 builds the real star-button feature) — the threat surface is
limited to the entitlement's genuine, unavoidable scope broadening and the intentional,
reversible test-mutation of the user's own liked-track state, both accepted per the table above.
</threat_model>
