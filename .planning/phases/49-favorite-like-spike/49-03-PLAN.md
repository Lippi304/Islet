<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Local shell session <-> accounts.spotify.com / api.spotify.com | Real OAuth PKCE authorization-code flow and a real bearer-token-authorized write call over HTTPS to Spotify's public Web API |
| Spike-harness terminal session <-> local filesystem | Access token and Client ID handling — must never persist beyond the terminal session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-49-07 | Information Disclosure | Spotify Client Secret baked into a distributed, unsandboxed binary | mitigate | PKCE flow only (no client secret at all, `code_challenge_method=S256`) — the only flow this script or Phase 50's production code uses, per STACK.md's recommendation for unsandboxed direct-distributed apps |
| T-49-08 | Spoofing | OAuth `code` interception via a non-loopback/non-HTTPS redirect URI | mitigate | `REDIRECT_URI` hardcoded to the literal loopback value `http://127.0.0.1:8888/callback` in the committed script — Task 1's acceptance criteria greps for this exact string |
| T-49-09 | Information Disclosure | Spike-harness access token or real Client ID persisted somewhere durable (shell history, a committed script with a hardcoded token) | mitigate | Task 1's committed script keeps `CLIENT_ID` as a literal placeholder string (grepped in acceptance criteria); Task 2's instructions explicitly direct the tester to substitute the real Client ID locally/uncommitted only, never to edit-and-commit the real value; the access token exists only in local shell variables for the terminal session's duration, never written to a file or `UserDefaults` |
| T-49-10 | Tampering | The real `PUT /me/library` call mutates the tester's actual Spotify library (adds a liked track) | accept | Explicitly the intended test per Success Criterion #3, using a track the tester chooses, trivially reversible by unliking it afterward |

No production Islet.app code or entitlement is touched by this plan — the entire threat surface
is the throwaway shell script's own OAuth mechanics, mitigated per the table above.
</threat_model>
