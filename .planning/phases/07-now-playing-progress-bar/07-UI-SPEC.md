# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Platform note:** Islet is a native macOS SwiftUI/AppKit app, not a web app. shadcn, Tailwind,
and hex-based web color tokens do not apply. This contract adapts the standard UI-SPEC
template to SwiftUI idioms: points (pt) instead of px, `Color`/`Font` values instead of hex,
SF Symbols instead of a web icon library, and no component registry. All values below are
sourced from `07-CONTEXT.md` (locked decisions D-01–D-07), `07-RESEARCH.md` (source-verified
integration points), and the existing, unmodified code in `Islet/Notch/NotchPillView.swift`
(read directly this session, lines 372-429).

## Spacing Scale (this phase)

Islet's existing UI uses SwiftUI points, not a strict 4px web multiple (e.g. existing values
of 6, 10, 19 already appear in `mediaExpanded`). The tokens below are the phase-specific,
prescriptive values for the new progress row — declared exactly so the executor doesn't
improvise:

| Token | Value | Usage |
|-------|-------|-------|
| Bar thickness | 3pt | Progress bar height (D-04: "thin minimalist line, 2-3pt") |
| Bar corner radius | 1.5pt (fully rounded / capsule) | Rounded caps on both bar layers (D-04) |
| Progress row height | 20pt | Total height of the new bar+labels row, replacing the D-09 4pt spacer |
| Row internal spacing | 6pt | Horizontal gap: elapsed label — bar — total label (matches the file's existing `VStack(spacing: 6)` convention, reused, not invented) |
| Time label reserved width | 28pt min, trailing/leading aligned | Elapsed label right-aligned, total label left-aligned, both fixed-min-width — prevents the bar's rendered length from jittering as digit glyph widths change (e.g. "0:59" → "1:00") |
| `expandedSize` height delta | +16pt (128pt → **144pt**) | 84pt old content − 4pt (removed D-09 spacer) + 20pt (new row) = 100pt; 32pt top clearance + 100pt content + 12pt bottom inset = 144pt total. Satisfies D-06's "~16-20pt growth" at the lower bound. |

Exceptions: Islet's spacing is point-based SwiftUI layout, not a 4px web design-token scale —
this is a pre-existing, established project convention (see Spacing Scale note above), not a
new exception introduced by this phase.

**Insertion point (exact, from `07-RESEARCH.md`):** replaces
`Spacer(minLength: 0).frame(height: 4)` at `NotchPillView.swift` line 411, inside the existing
`VStack(spacing: 6)` between the art/title/artist/bars row and the transport-button row.

## Typography

| Role | Size | Weight | Design | Line Height | Color |
|------|------|--------|--------|-------------|-------|
| Title (existing, unchanged) | 15pt | bold | `.rounded` | default | `.white` |
| Artist (existing, unchanged) | 12pt | regular | `.rounded` | default | `.secondary` (D-10 grey) |
| **Time label (NEW, this phase)** | **11pt** | **regular** | **`.rounded`** | **default** | **`.secondary` (D-05: matches artist grey, NOT accent)** |
| Transport glyph (existing, unchanged) | 16pt | semibold | default | default | `.white` |

**New element spec (D-05, D-01, D-02):**
- Two `Text` labels: elapsed (left of bar) and total duration (right of bar) — format
  `m:ss` (e.g. `"1:23"`), joined visually as `"1:23 / 3:45"` by their position flanking the
  bar, not as one concatenated string.
- Apply `.monospacedDigit()` to both labels — prevents the row's layout from micro-jittering
  every second as digit glyph widths change. (Not explicitly requested in CONTEXT.md; added
  here as a display-quality requirement consistent with D-06's "no cramping" intent.)
- `m:ss` formatting: hand-rolled helper (no `DateComponentsFormatter`) per
  `07-RESEARCH.md`'s Standard Stack recommendation — always `minutes:seconds`, zero-padded
  seconds, no hour component (tracks are always well under an hour).

## Color

Islet has no 60/30/10 web-style color split — it's a single black rounded "island" overlay,
not a page layout. Adapted to the app's actual visual system:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (island surface) | `Color.black` | The island shape itself (`NotchShape().fill(Color.black)`) — unchanged by this phase |
| Secondary (text/track) | `Color.white.opacity(0.25)` | **NEW:** the progress bar's unfilled track portion (D-03: "dim grey/white") — chosen to read as a subtle background line, dimmer than any existing icon-opacity value in the file (0.5–0.7) since it's a passive background element, not an icon |
| Accent (10% — reserved) | `\.activityAccent` environment value (existing, `@AppStorage`-backed) | Reserved for: **equalizer bars** (existing), **charging bolt** (existing), **connected-device icon** (existing), **progress bar filled portion (NEW, this phase, D-03)** — no other element in this phase uses accent |
| Destructive | not applicable | This phase has zero destructive actions (display-only bar, no delete/reset/confirm flows) |

Accent reserved for: equalizer bars, charging bolt, connected-device icon, progress bar fill.
Never applied to: time label text (D-05 explicitly rejects this), track/background portion of
the bar, title/artist text.

## Copywriting Contract

This phase is 100% display-only (PBAR-01: "no tap-to-seek") — there is no CTA, no user-facing
empty state, and no destructive action. The contract below documents the one genuine copy
decision this phase introduces (time-label formatting) and the fallback-state decision the
research flagged as an open gap.

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — display-only bar per PBAR-01; clicking/dragging the bar performs no action |
| Time label format | `m:ss` per side, e.g. elapsed `"1:23"`, total `"3:45"` — flanking the bar reads as `"1:23 / 3:45"` (D-01, D-02) |
| Missing-data fallback (resolves Research Open Question 2) | When `duration` or `elapsed` is `nil` (adapter hasn't reported timing yet, or a player doesn't expose it): **reserve the row's 20pt height, but render the bar and both labels at `opacity(0)`** — no "--:--" placeholder text, no layout collapse/jump. The row fades in once real data arrives, mirroring the existing artwork-latency precedent ("design the UI to fill art in asynchronously" — CLAUDE.md). This avoids both a broken 0-width/NaN bar (Pitfall 5) and a jarring height change once data resolves. |
| Error state | Not applicable — this phase doesn't touch the existing `.unavailable`/"Now Playing nicht verfügbar" branch (D-12, out of scope per CONTEXT.md); that state's copy is unchanged |
| Destructive confirmation | Not applicable — no destructive actions in this phase |

## Interaction Contract (native-app addition, not in web template)

Explicitly required by PBAR-01 and reconfirmed in CONTEXT.md Deferred Ideas — stated here
because it's the single most important behavioral constraint for the checker/executor to
verify:

- The bar and its labels are **inert**: no `.onTapGesture`, no `Button`, no drag gesture of
  any kind attached to the `ProgressBar` view or its children.
- The existing `.onTapGesture { onClick() }` on the art/title/artist/bars row (line 409) must
  **not** be extended to cover the new progress row — the new row sits in its own scope in the
  `VStack`, outside that tap gesture's reach (same discipline already documented at line
  404-409 for why the transport-button row is excluded).
- Continuous animation (D-07): bar fill driven by `TimelineView(.animation(paused:
  !isPlaying))`, mirroring `EqualizerBars` exactly (`NotchPillView.swift` lines 519-537) —
  ticks every frame while playing, zero clock while paused (idle-CPU discipline).
- Paused-freeze (success criterion 3): the displayed elapsed value while paused must come
  from the ported `isPlaying != true { return elapsedSeconds }` guard, never from
  `context.date` — see `07-RESEARCH.md` Pitfall 1 and Pattern 2 for the exact formula to
  port verbatim.
