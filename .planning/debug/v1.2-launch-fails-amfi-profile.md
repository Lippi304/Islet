## Symptoms

- **Expected behavior:** v1.2 launches normally after a Sparkle update or a fresh DMG install, same as v1.1 did.
- **Actual behavior:** `open /Applications/Islet.app` (and a direct exec of the binary) fails immediately, both via Sparkle's in-place update AND via a completely clean install from the same notarized `dist/Islet.dmg` — this is not an update-mechanism artifact, the v1.2 binary itself cannot launch.
- **Error messages:**
  - User-facing: "Islet kann nicht geöffnet werden."
  - `open`: `Error Domain=RBSRequestErrorDomain Code=5 "Launch failed." ... Error Domain=NSPOSIXErrorDomain Code=163 "Launchd job spawn failed"`
  - Direct exec: process is SIGKILLed immediately (exit 137).
  - Unified log (`amfid`): `Error Domain=AppleMobileFileIntegrityError Code=-413 "No matching profile found" ... unsatisfiedEntitlements=(com.apple.developer.usernotifications.communication, com.apple.developer.weatherkit)`
  - `spctl --assess` and `codesign --verify --deep --strict` both report the app as valid/accepted — this is NOT a Gatekeeper/notarization rejection, it is AMFI's separate, stricter entitlement-vs-profile check at process-spawn time.
- **Timeline:** First observed 2026-07-22, immediately after installing the freshly-built, freshly-notarized v1.2 (build 3). v1.1 (built yesterday via the same `scripts/release.sh` pipeline) does not have this problem — confirmed both by the user's own experience (used v1.1 extensively earlier this session, including for the separate bluetooth-prompt-on-relaunch debug session) and by directly launching an archived v1.1 DMG copy from `~/Downloads` during this investigation.
- **Reproduction:** 100% reproducible. `open /Applications/Islet.app` after any v1.2 install (Sparkle update OR clean DMG copy) always fails with the AMFI error above.

## Resolution

root_cause: |
  CONFIRMED. v1.2's Developer-ID-signed binary correctly carries the
  com.apple.developer.weatherkit and com.apple.developer.usernotifications.communication
  entitlements (a previously-known entitlements-loss bug was already fixed in release.sh
  before this session). However, the app's embedded provisioning profile
  ("Mac Team Provisioning Profile: com.lippi304.islet") is Xcode automatic-signing's
  development-flavored profile type, which does not authorize a Developer-ID Application
  signing identity to use these two restricted ("managed capability") entitlements. AMFI
  rejects the process at spawn time with Code=-413 "No matching profile found". This
  problem has existed since day one of Developer-ID distribution for this project — it was
  only ever masked because v1.1 (and presumably earlier versions) had accidentally lost all
  entitlements during the Developer-ID re-signing step, which sidestepped AMFI's profile
  check entirely (at the cost of WeatherKit/Calendar/Location/Automation silently not
  working — the previously-reported "Wetter nicht verfügbar" symptom).
